+# PR 6245 Design Doc
+
+## Overview
+This PR addresses regressions related to how character levels and rewards updates handle integer and string types.
+
+## Motivation
+Recent changes introduced false-negative validations when Firestore state flags (`level_up_in_progress`, etc.) came in as loosely-typed int (`1`) or string (`"1"`) instead of booleans, breaking the internal state transitions.
+Additionally, when `xp_awarded` fields default to `0`, previous chained `or` fallbacks were overriding valid zeroes with stale pending state.
+
+## Implementation Details
+1. **Unify `extract_character_xp` extraction**:
+   Both `world_logic.py` and `game_state.py` parse legacy combinations of `experience.current` dicts and flat `xp` scalars. We unified these to only use the safe `extract_character_xp` from `game_state`.
+2. **Boolean casting width**:
+   The `_is_state_flag_true` function was deliberately widened to safely accept integers and stringly-typed numeric equivalents `1`, `"1"` that LLMs routinely output in mock schemas, explicitly denying unparseable terms like `"yes"` to remain compatible.
+3. **Explicit zero checks**:
+   In `world_logic.py`, `xp_from_combat` parsing changed `... or ...` short-circuiting to explicit `if target is not None:` loops, preserving active 0 awards from erroneously pulling default pool numbers.
