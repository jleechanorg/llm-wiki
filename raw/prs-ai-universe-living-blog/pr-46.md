## Summary
Phase 2 Cloud Run deployment for ai_universe_living_blog — adds containerized production deployment.

## Background
The blog MCP server currently requires `npm run dev:blog` running locally, making it impossible to receive real GitHub webhooks. Phase 2 packages the server as a Docker image and deploys it to Cloud Run for always-on HTTPS availability.

Design spec: [docs/superpowers/specs/2026-03-30-cloud-run-deployment-design.md](docs/superpowers/specs/2026-03-30-cloud-run-deployment-design.md)

## Goals
- One-command deploy via `./scripts/deploy-cloudrun.sh`
- Server reachable at stable HTTPS URL (Cloud Run auto-provisions TLS)
- GitHub webhook events reach the server and create posts in Firestore
- Zero downtime deploys (Cloud Run default)
- Cost < $5/month at AO worker traffic levels

## Tenets
- Min-instances=1 to prevent cold-start webhook failures
- Workload Identity Federation — no stored service account keys
- Path-filtered CI trigger — only deploys when deploy files change

## High-level description of changes

### Dockerfile
- `FROM node:20-alpine` base
- Production-only deps via `npm ci --only=production`
- Copies `dist/` (TypeScript compiled) and `public/` (web UI)
- `STORAGE_TYPE=firestore` and `NODE_ENV=production` env vars
- `EXPOSE 8081`, `CMD ["node", "dist/blog/server.js"]`

### scripts/deploy-cloudrun.sh
- Infers GCP project from `GOOGLE_CLOUD_PROJECT` env or `gcloud config`
- Tags image with short commit SHA for traceability
- Deploys to Cloud Run with `--min-instances 1 --max-instances 3 --memory 256Mi`
- Sets `STORAGE_TYPE=firestore,NODE_ENV=production` env vars
- `--allow-unauthenticated` for open MCP client access

### .github/workflows/deploy-cloudrun.yml
- Triggers on push to `main` when `Dockerfile`, `deploy-cloudrun.sh`, or this workflow change
- `workflow_dispatch` for manual deploys
- Uses `google-github-actions/auth@v2` with Workload Identity Federation
- Runs `npm ci && npm run build` then `scripts/deploy-cloudrun.sh`
