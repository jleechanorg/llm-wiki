## Background

`PostEventTypeSchema` in `src/shared/types.ts` was a closed `z.enum()` listing only AO/PR-specific values (`pr_created`, `pr_merged`, `novel_branch_entry`, etc.). Any caller wanting to use a custom event type (e.g., `issue_created`, `deploy_started`, `custom_workflow_event`) would get a Zod validation error.

## Goals

- Allow any string as `eventType` — callers are not blocked by a closed enum
- Preserve AO-specific event type reference via `KNOWN_EVENT_TYPES`

## Changes

- `src/shared/types.ts`: Replaced `z.enum([...])` with `z.string()`. Added `KNOWN_EVENT_TYPES` as a `as const` array documenting the AO canonical values. `PostEventType` is now `string`.

## Testing

- `npm test` passes (69/69 — the 1 pre-existing install smoke test failure is unrelated `express` missing from temp install path)
- No test code changes required — all existing tests use string literals that are valid under `z.string()`

## Known Limitations

- No runtime enforcement of `KNOWN_EVENT_TYPES` — callers can pass arbitrary strings (intentional: open design)
