## Background

`FileBlogStorage.load()` directly passed persisted JSON objects into `MemoryBlogStorage.createPost()` without validating them. If a `blog-data.json` file contained a post with an invalid `status` field (any string not matching `'draft' | 'published'`), Zod validation in `MemoryBlogStorage.createPost()` would throw — causing the server to crash on startup with no recovery path.

## Goals

- Server starts cleanly even when the data file contains corrupted/malformed posts
- Invalid posts are skipped with a warning log so valid data is preserved
- No behavior change for valid persisted data

## Tenets

- Minimal change — only the `load()` method is modified
- Load stays synchronous — no async/await changes
- Tests verify skip-on-invalid behavior without mocking internal state

## High-level description of changes

**`src/blog/storage-file.ts`**
- Import `PostSchema` from `../shared/types.js`
- In `load()`, wrap each post with `PostSchema.safeParse()` before calling `mem.createPost()`
- On parse failure: log a warning with post ID and error issues, then `continue` (skip)
- On success: pass `parsed.data` to `createPost()` as before

**`tests/blog/storage-file.test.ts`** (new file)
- Three test cases:
  1. Valid data → all posts loaded correctly
  2. One invalid post (bad `status` enum) + one valid → valid post loaded, invalid skipped, warning logged
  3. Valid post + malformed post (missing required fields) → valid post loaded, malformed skipped, warning logged
