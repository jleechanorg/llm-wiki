## Raw Body
PROBLEM: Assistant message timestamps were set to userTimestamp + 1ms when the request arrived, not when the LLM actually generated the response. This resulted in inaccurate timestamps that didn't reflect actual generation time.

SOLUTION:
1. Added failing test "should set assistant timestamp AFTER LLM response is generated, not when request arrives" that simulates 100ms LLM delay
2. Modified SingleModelChatService.generateAssistantMessage() to capture timestamp AFTER LLM call completes (line 227)
3. Ensured timestamp is still after user timestamp for proper ordering
4. Updated existing tests to reflect new behavior (actual generation time vs artificial +1ms)

TEST RESULTS:
- New test passes ✓ (verifies timestamp reflects actual generation time)
- All 10 MessageTimestampOrdering tests pass ✓
- Full test suite: 302 tests passed, no regressions

CHANGES:
- backend/src/services/SingleModelChatService.ts:224-239
  * Set assistantTimestamp = new Date().toISOString() AFTER LLM call
  * Fallback to userTimestamp + 1ms only if generation time is before user time
- backend/src/test/MessageTimestampOrdering.test.ts
  * Added TDD test for timestamp generation timing
  * Updated 2 tests to reflect new behavior
