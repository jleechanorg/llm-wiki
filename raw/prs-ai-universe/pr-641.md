## Summary

Fixes VIP allowlist fallback support and critical bug where `RATE_LIMIT_RENDER_ASSUME_SINGLE` was not being honored, which caused production deployments on Render/Cloud Run to incorrectly apply ultra-strict 1 req/min limits.

## Changes

### 1. **VIP Allowlist Fallback Support** ✅
- Added `VIP_ALLOWLIST` environment variable support for VIP allowlist when Secret Manager is unavailable
- Supports comma-separated format: `email1@example.com,email2@example.com`
- Supports JSON array format: `["email1@example.com","email2@example.com"]`
- VIP rate limits (50/hour) are preserved even when Secret Manager is unavailable

### 2. **Critical Fix: RATE_LIMIT_RENDER_ASSUME_SINGLE** 🚨
- **Bug**: `RATE_LIMIT_RENDER_ASSUME_SINGLE=true` was set in `render.yaml` but not honored by code
- **Impact**: Production on Render/Cloud Run was incorrectly detected as distributed, applying 1 req/min limits instead of normal production limits
- **Fix**: Code now checks `RATE_LIMIT_RENDER_ASSUME_SINGLE` before detecting Render/Cloud Run as distributed
- **Result**: Single-instance deployments now correctly use normal production limits (10/day anon, 30/day auth, 50/hour VIP, 1000/hour admin)

### 3. **Development Defaults Consistency** ✅
- Added missing `vip` tier to development defaults
- Now consistent with production/test defaults

### 4. **Secret Manager Configuration** ✅
- Secret Manager remains enabled (relies on PR #642 fix for correct production values)
- VIP allowlist fallback ensures VIP functionality works even if Secret Manager is unavailable
