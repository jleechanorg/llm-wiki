## Summary

Fixes critical production bug where admin user `jleechan@gmail.com` is incorrectly rate limited at **5 requests/hour** (anonymous limit) instead of **1000 requests/hour** (admin limit).

**Impact:** Production blocker - admin cannot use the system due to aggressive rate limiting.

## Root Cause Analysis

### The Bug

Admin users hitting rate limit after only 5 requests despite being correctly configured in `FIREBASE_ADMIN_EMAILS`.

**Production Evidence:**
```json
{
  "isAdmin": false,
  "uid": "UlET1VDO7ORksFiAPQ2b9jI9K9t1",
  "error": "Rate limit exceeded. Limit: 5, Reset: 2025-11-14T22:13:13.845Z"
}
```

### Deep Dive: Why This Happens

#### 1. Firebase Authentication Token Structure

Firebase ID tokens are JWT tokens that contain **claims** - key-value pairs of user information. The `email` claim is **optional** and may not be present in all tokens, depending on:

- Authentication method used (Google, email/password, custom, etc.)
- Token generation settings
- User account configuration
- Token refresh behavior

**Example Token Claims:**
```typescript
// Token WITH email claim (expected)
{
  "uid": "UlET1VDO7ORksFiAPQ2b9jI9K9t1",
  "email": "jleechan@gmail.com",  // ✅ Present
  "email_verified": true
}

// Token WITHOUT email claim (causing bug)
{
  "uid": "UlET1VDO7ORksFiAPQ2b9jI9K9t1",
  // email claim missing! ❌
}
```

#### 2. Code Path That Triggers the Bug

**Step 1: Token Verification**
```typescript
// FirebaseAuthTool.ts:98
const decodedToken = await this.auth.verifyIdToken(idToken);

const user: User = {
  id: decodedToken.uid,
  uid: decodedToken.uid,
  email: decodedToken.email || '',  // ⚠️ DEFAULTS TO EMPTY STRING
  name: decodedToken.name || decodedToken.email || 'Anonymous',
  isAuthenticated: true
};
```

When `decodedToken.email` is `undefined`, JavaScript's `||` operator returns `''` (empty string).
