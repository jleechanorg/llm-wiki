**Problem:**
- Google Cloud Monitoring was rejecting metric writes with INVALID_ARGUMENT errors
- Error: "One or more points were written more frequently than the maximum sampling period"
- Timestamps were only 0.355 seconds apart (23:09:31.584 vs 23:09:31.939)
- Caused noisy error logs and potential data loss

**Solution:**
1. **Time-bucketing**: Round timestamps to nearest second (configurable via minSamplingPeriodMs)
2. **Aggregation**: Sum metric values that fall within the same time bucket + label combination
3. **Graceful error handling**: Catch INVALID_ARGUMENT for duplicate timestamps with WARN log (not ERROR)
4. **Reduced logging noise**: Summary logs only for duplicate timestamp errors, removed stack traces

**Key Changes:**
- Added `minSamplingPeriodMs` config parameter (default: 1000ms)
- Implemented `roundTimestampToBucket()` helper for time-bucketing
- Enhanced `flushMetrics()` to aggregate metrics in same time bucket
- Specific error handling for code 3 (INVALID_ARGUMENT) duplicate timestamp errors
- Comprehensive regression tests covering all scenarios

**Testing:**
- All 57 MonitoringService tests pass
- Tests verify time-bucketing, aggregation, and error handling
- Matches actual production error case (0.355s interval)

**Impact:**
- ✅ No more INVALID_ARGUMENT errors from Cloud Monitoring
- ✅ Graceful handling prevents app crashes
- ✅ Reduced logging noise (WARN instead of ERROR with stack traces)
- ✅ Accurate metric aggregation for sub-second writes
- ✅ Configurable sampling period for different use cases

Fixes #issue-metrics-flush-error
