## Problem

The `getConversationContext` method used a simplified sort that didn't handle edge cases properly. It called `new Date(a.createdAt).getTime()` directly without checking for invalid timestamps. When a timestamp cannot be parsed, `getTime()` returns `NaN`, causing unstable sort behavior (NaN comparisons). This was inconsistent with the sophisticated sorting logic in `toConversationHistoryResponse` which properly handles invalid timestamps, tie-breaking by sequence number, and Firestore Timestamp objects.

## Solution

- Added `sortMessagesChronologically()` helper method that:
  - Normalizes timestamps using existing `normalizeTimestampValue()` (handles Firestore Timestamp objects, strings, numbers)
  - Properly handles invalid timestamps (NaN checks)
  - Implements tie-breaking by sequence number, then message ID
  - Ensures deterministic ordering

- Fixed `getConversationContext()` to use the robust sorting helper instead of naive `Date.getTime()` sort
- Refactored `toConversationHistoryResponse()` to use the shared helper (consolidates duplicate sorting logic)
