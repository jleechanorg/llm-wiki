## Summary

Add new GitHub Actions workflow to simulate deployment build scenarios and catch module resolution issues before they reach production.

## Problem

Recently encountered a deployment failure when:
1. New `mcp-server-utils` dependency was added to `package.json`
2. Code imported functions from the new module
3. Local development didn't run `npm install` after pulling changes
4. Deployment script tried to build TypeScript without the symlink
5. **Build failed**: "Cannot find module '@ai-universe/mcp-server-utils'"

The existing CI didn't catch this because it always runs `npm ci` before building, which creates all symlinks. However, the deployment script (`deploy.sh`) has a conditional install that can skip creating symlinks if `tsc` already exists.

## Solution

New `deployment-simulation.yml` workflow that:

### ✅ Simulates Deployment Build
- Builds shared libraries (like `deploy.sh` does)
- Installs backend dependencies with `npm ci`
- Tests TypeScript compilation

### ✅ Verifies Symlinks
Dynamically extracts all `@ai-universe/*` dependencies from `package.json` and verifies each symlink exists:

```bash
Expected @ai-universe dependencies: auth-utils config-utils firestore-utils mcp-server-utils

✅ Found: node_modules/@ai-universe/auth-utils
✅ Found: node_modules/@ai-universe/config-utils  
✅ Found: node_modules/@ai-universe/firestore-utils
✅ Found: node_modules/@ai-universe/mcp-server-utils
```

If any symlink is missing:
```bash
❌ Missing: node_modules/@ai-universe/mcp-server-utils
🚨 ERROR: Missing symlinks for: mcp-server-utils
This will cause deployment build failures!
```

### ✅ Tests Runtime Resolution
Not just TypeScript compilation - also tests that modules can actually be imported at runtime, catching packaging issues.
