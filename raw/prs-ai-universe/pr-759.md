## Problem

PR #648 introduced P1 metrics for inbound HTTP request/response sizes, but they were silently failing to write to Google Cloud Monitoring due to a **label schema conflict**.

### Root Cause

**HttpClient** and **Express middleware** both used the same metric names with incompatible label schemas:

1. **HttpClient creates metrics FIRST:**
   - Names: `http_request_size_bytes`, `http_response_size_bytes`
   - Labels: `{domain, operation, status}`

2. **Middleware tries to write LATER:**
   - Same names: `http_request_size_bytes`, `http_response_size_bytes`
   - Different labels: `{endpoint, method, status_code, status}`
   - **Google Cloud REJECTS** - schema mismatch

3. **Silent failure:** Errors suppressed by `.catch()` handlers

**Credit:** monitoringv agent identified the validation gap leading to this discovery.

---

## Solution

### Renamed Metrics for Clear Separation

**Inbound (Express middleware):**
- `http_inbound_request_size_bytes`
- `http_inbound_response_size_bytes`
- Labels: `{endpoint, method, status_code, status}`

**Outbound (HttpClient):**
- `http_outbound_request_size_bytes`
- `http_outbound_response_size_bytes`
- Labels: `{domain, operation, status}`

### Additional Fixes
- Middleware always records metrics (even size=0) for consistent time series
- Old names marked DEPRECATED for backward compatibility
- Comprehensive test suite (7 tests) validates P1 requirements
