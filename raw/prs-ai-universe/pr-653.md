## Raw Body
Root cause analysis identified two issues preventing admin users (jleechan@gmail.com) from bypassing rate limits:

Issue 1: Environment variable mismatch (PRIMARY)
- RateLimitTool reads `ADMIN_EMAILS` env var
- Deploy script sets `FIREBASE_ADMIN_EMAILS` env var
- Fix: RateLimitTool now checks both ADMIN_EMAILS and FIREBASE_ADMIN_EMAILS

Issue 2: Missing email in User object (SECONDARY)
- Agents created User objects with empty email: { id, email: '', ... }
- RateLimitTool requires user.email to check admin status
- Fix: Add optional userEmail parameter to agent schemas
  - SecondOpinionInputSchema: Added userEmail field
  - sendMessageSchema: Added userEmail field
  - rate_limit.status tool: Added userEmail field
- Agents now populate email from userEmail parameter when provided

Changes:
- shared-libs/packages/mcp-server-utils/src/RateLimitTool.ts:
  * getAdminEmailSet() now checks both ADMIN_EMAILS and FIREBASE_ADMIN_EMAILS
- backend/src/agents/SecondOpinionAgent.ts:
  * Added userEmail to SecondOpinionInputSchema
  * Updated User object construction to use userEmail
  * Updated rate_limit.status tool schema to accept userEmail
  * Updated handleRateLimitStatus to extract and use userEmail
- backend/src/agents/ConversationAgent.ts:
  * Added userEmail to sendMessageSchema
  * Updated User object construction to use userEmail

Fixes intermittent admin rate limiting where jleechan@gmail.com was treated as regular authenticated user (50 req/hour) instead of admin (1000 req/hour).
