## Summary
Fixes HIGH severity issue where the PR preview build polling loop could fail on transient 404/NOT_FOUND responses from Cloud Build API immediately after async submit.

## Problem
The build status polling loop at `.github/workflows/pr-preview.yml:62` was running:
```bash
STATUS=$(gcloud builds describe "$BUILD_ID" --format="value(status)")
```

With `set -e` active, any transient 404 error from Cloud Build would immediately terminate the step before the retry loop could continue. Cloud Build API often returns 404 when queried too quickly after an async submit.

## Solution
- Added `2>/dev/null || echo "UNKNOWN"` to handle command failures gracefully
- Added logic to treat `UNKNOWN` or empty status as "still building"
- Added explicit `continue` statement to avoid duplicate sleep

## Changes
```yaml
# Handle transient 404/NOT_FOUND responses from Cloud Build API
STATUS=$(gcloud builds describe "$BUILD_ID" --format="value(status)" 2>/dev/null || echo "UNKNOWN")
echo "Build status: $STATUS ($i/90)"

# Treat UNKNOWN/empty as still building (transient 404 after async submit)
if [ -z "$STATUS" ] || [ "$STATUS" = "UNKNOWN" ]; then
  echo "⏳ Build status unknown (transient), continuing to poll..."
  sleep 10
  continue
fi
```
