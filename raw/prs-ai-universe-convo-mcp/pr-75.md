Root Cause:
- Metadata was persisted correctly in Firestore but NOT returned in the AddMessageResult/CreateConversationResult responses
- This forced clients to call get-history to retrieve metadata, which experienced 20-25s lag due to Firestore index propagation delays

Changes:
1. Add metadata field to CreateConversationResult and AddMessageResult interfaces
2. Update post-transaction read-back in createConversationWithMessage to retrieve and return metadata
3. Update post-transaction read-back in addMessageToConversation to retrieve and return metadata
4. Update sendMessage in unified-mcp-tools to include metadata in response
5. Add Matrix 5 tests demonstrating immediate metadata availability

Impact:
- Clients now receive metadata immediately (<100ms) in send-message response
- Eliminates need to call get-history after sending messages
- Resolves #556: conversation history lag and missing metadata after agent.second_opinion

Testing:
- All existing tests pass
- New Matrix 5 tests verify metadata is returned immediately for both new conversations and adding to existing conversations
- Type-check, lint, and build all pass

- backend/src/tools/unified-mcp-tools.ts
