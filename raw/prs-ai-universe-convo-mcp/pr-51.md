## Summary

## Problem
The auto-deploy workflow failed after PR #48 was merged with this error:
```
ERROR: (gcloud.builds.submit) 
The build is running, and logs are being written to the default logs bucket.
This tool can only stream logs if you are Viewer/Owner of the project and, 
if applicable, allowed by your VPC-SC security policy.
```

The `deploy.sh` script at line 189 was using synchronous `gcloud builds submit` which tries to stream build logs. This fails in VPC-SC protected environments where the GitHub Actions service account doesn't have log streaming permissions.

## Solution
Applied the same async build polling approach that successfully fixed the PR preview workflow in PR #48.

### Changes to deploy.sh:189
**Before:**
```bash
(cd backend && gcloud builds submit . --tag "$IMAGE_TAG")
```

**After:**
```bash
# Submit build asynchronously to avoid VPC-SC log streaming issues
BUILD_ID=$(cd backend && gcloud builds submit . --tag "$IMAGE_TAG" --async --format="value(id)")

# Wait for build to complete with polling loop
for i in {1..90}; do
    STATUS=$(gcloud builds describe "$BUILD_ID" --format="value(status)" 2>/dev/null || echo "UNKNOWN")
    
    # Handle transient 404s, check for success/failure
    if [ "$STATUS" = "SUCCESS" ]; then
        break
    elif [ "$STATUS" = "FAILURE" ]; then
        exit 1
    fi
    sleep 10
done
```

## Key Improvements
- ✅ Async build submission with --async flag
- ✅ Build status polling with 15-minute timeout (90 × 10s)
- ✅ Transient 404 handling for Cloud Build API race conditions
- ✅ Clear build logs URL on failure
- ✅ Colored output for better visibility
