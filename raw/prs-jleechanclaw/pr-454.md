## Background

OpenClaw experienced 13 gateway outages in March 2026, including 4 P0 full outages on March 28 alone. Root cause analysis (5 Whys) showed every P0 outage could have been caught by a staging canary test before touching production. See `roadmap/OUTAGE_PREVENTION_ROADMAP.md` for full incident inventory and analysis.

## Goals

1. Provide an isolated staging gateway (port 18790) for pre-deploy validation
2. Implement a 6-point canary test covering all known crash vectors
3. Wire the canary into the upgrade workflow so production is never changed without staging validation

## Tenets

- Every known crash vector gets a canary check (config schema, native modules, Slack token, SDK protocol)
- Staging uses the same Node binary and config shape as production
- Canary is a gate — upgrade aborts if any check fails

## High-level description of changes

Adds a staging environment that mirrors production on port 18790. The 6-point canary test validates: (1) gateway health endpoint, (2) config schema (no crash-causing keys like cmuxBotToken), (3) native module ABI compatibility (better-sqlite3), (4) Slack app token validity, (5) SDK protocol version compatibility, (6) heartbeat response time. The canary is wired into `openclaw-upgrade-safe.sh` as Step 2.5, running automatically before any upgrade confirmation.

## Testing

- Scripts tested for syntax: `bash -n scripts/staging-gateway.sh && bash -n scripts/staging-canary.sh`
- Canary correctly detects missing gateway (all checks fail → exit 1)
- Canary correctly validates healthy gateway (all checks pass → exit 0)
- Upgrade script integration: canary failure aborts upgrade before confirmation step
