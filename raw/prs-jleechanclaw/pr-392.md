## Background

AO workers currently post to `#mcp-mail` threads but there's no human-audience channel forJeffrey to observe what's happening. The requested Slack channel `C0ANK6HFW66` needs human-readable spawn/exit messages, a startup health check for the MCP mail listener, and a periodic worker heartbeat.

## Goals

- Mirror AO worker lifecycle (spawn/exit) into Slack channel `C0ANK6HFW66` with human-readable messages
- Startup health check that fails loud if MCP mail listener isn't active
- Worker mailbox heartbeat every N minutes so silence is detectable
- Avoid spam loops and respect existing MCP mail ack policy
- Config/env flags to enable/disable each feature

## Tenets

- Read-only from AO's perspective — no changes to AO event emission or worker behavior
- State deduplication prevents double-posts across polling cycles
- MCP mail ack policy unchanged — no changes to mcp-mail plugin
- Bridge runs as a launchd LaunchAgent, not inside AO

## High-level description of changes

### Bridge Python module (`src/orchestration/human_channel_bridge.py`)

Polls `~/.agent-orchestrator/{hash}-{project}/sessions/` every run, detects `spawning` → active and active → terminal transitions, and posts human-readable Slack messages. Maintains a JSON state file (`~/.openclaw/state/human_channel_bridge.json`) for deduplication. Reads channel history and replies contextually to human questions ("how's it going", "status worker") with rate-limiting (max 3 replies/poll).

**Safety guardrails:**
- State file tracks `reported_spawns`/`reported_exits` — each session posted at most once
- Thread reply: exit messages use `thread_ts` to reply in the same thread as spawn
- Bot filter: only replies to non-bot users
- Max 3 contextual replies per poll cycle
- `BRIDGE_ENABLED=0` skips all Slack operations
