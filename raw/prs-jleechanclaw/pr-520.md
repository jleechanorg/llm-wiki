## Background
The jleechanclaw harness was previously using Groq for fact extraction in mem0 hooks, which introduced external API dependencies and costs. To improve local-first capabilities and reduce reliance on third-party providers for extraction tasks, we are transitioning to using local OSS LLMs (specifically Ollama with llama3.2:3b) as the default fact-extraction provider.

## Goals
- Transition the default mem0 fact-extraction provider from Groq to local Ollama.
- Centralize mem0 configuration in `.claude/hooks/mem0_config.py` to ensure consistency across recall/save hooks and CLI tools.
- Implement robust, fail-open enablement gates (`mem0_hooks_enabled()`) to prevent misconfigured memory hooks from blocking the core developer loop.
- Align documentation and test fixtures with the new Ollama-default architecture.

## Tenets
- **Local-First:** Prefer local OSS models (Ollama) for fact-extraction and non-embedding tasks.
- **Fail-Open:** Non-critical orchestration hooks should never block the user prompt or session end on configuration errors.
- **Config-Centric:** All plugins and hooks should honor a single source of truth (`openclaw.json`) via `OPENCLAW_CONFIG_PATH`.

## High-Level Description
- Updates the repo’s mem0 “harness default” to use **Ollama for the OSS LLM** (default `llama3.2:3b`) while keeping **OpenAI `text-embedding-3-small` @ 768** as the default embedder.
- Claude Code mem0 hooks now centralize config in `.claude/hooks/mem0_config.py`, merge overrides from `openclaw.json`, and use a new `mem0_hooks_enabled()` gate.
- `scripts/auto_fact_capture.py` and `scripts/mem0_shared_client.py` are refactored to honor the centralized config and provider selection.
