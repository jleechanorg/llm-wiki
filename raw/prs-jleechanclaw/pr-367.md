## Raw Body
## Background
The monitor-e2e-canary Slack post fired on every monitor run, including healthy ones. This created noise in `#ai-slack-test` on every successful run, making it impossible to distinguish a deliberate failure-triggered canary from routine monitoring.

## Goals
- Canary (monitor-e2e-canary post to C0AKYEY48GM) fires only when there is a pre-existing probe failure, not on every healthy run.

## Tenets
- Minimal change — only the gating condition changes, all other logic unchanged.

## High-level description of changes
Added `PRE_CANARY_FAILURE` detection before the canary block: set to 1 if any of `PROBE_REQUEST_RC`, `GATEWAY_PROBE_RC`, or `HTTP_GATEWAY_RC` is non-zero. The canary now only executes when `RUN_CANARY=1` AND `PRE_CANARY_FAILURE=1`. On healthy runs, `SLACK_CANARY_RC` stays 0 and `SLACK_CANARY_SUMMARY` remains "canary skipped".

## Testing
- `bash -n monitor-agent.sh` — syntax valid
- Pre-existing test suite (`python -m pytest src/tests/ tests/`) — 196 passed, 1 unrelated pre-existing failure in `test_regression_detector.py`
- Manual: verified `PRE_CANARY_FAILURE=0` on a healthy run would skip the curl post entirely

## Low-level details

### monitor-agent.sh
**Why**: The unconditional canary post caused noise on every healthy run.  
**What**: Added `PRE_CANARY_FAILURE` guard that checks the three initial probe return codes before executing the canary block.  
**How**: Only changes the outer `if` condition; all canary post, polling, and RC logic unchanged.
