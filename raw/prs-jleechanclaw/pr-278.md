## Background
PR #269 added Phase 2 reviewer loop — reviewer_agent posts GitHub reviews and MCP mail. But GitHub blocks APPROVED reviews on your own PRs (HTTP 422). This PR switches from APPROVED reviews to machine-readable verdict comment markers that merge_gate reads from issue comments. Also fixes several fail-open bugs in merge_gate.

## Goals
- Replace APPROVED GitHub reviews with `<!-- reviewer-verdict: APPROVE sha:<HEAD_SHA> -->` comment markers
- Fix fail-open bugs: missing SHAs in CR approval, JSON parse errors in verdict comments, GraphQL failures in blocking comments
- Skip verdict marker when review is downgraded to COMMENT (self-review fallback)
- Authenticate verdict markers via authorized reviewer allowlist
- Config cleanup: remove PR size limit rule, keep team-mini for MiniMax only

## Tenets
- Fail closed: missing SHAs, parse errors, and API failures block merge — never pass through
- Verdict markers must be authenticated (only `reviewer-agent[bot]`, `jleechan2015`, `jleechan`)
- Self-review fallback (COMMENT) must not emit APPROVE verdict marker
- No silent discards of gate signals

## High-level description of changes
The reviewer_agent now posts a verdict comment marker after its GitHub review. merge_gate reads these markers from issue comments (not PR reviews) to satisfy evidence condition. This bypasses GitHub's self-review restriction.

Key safety properties:
- Marker SHA must match current PR HEAD (stale markers ignored)
- Only authorized identities can satisfy evidence via markers
- When GitHub downgrades APPROVE to COMMENT (self-review), no verdict marker is posted
