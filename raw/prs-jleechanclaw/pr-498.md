## Background
Silent Slack failure (2026-04-04): OpenClaw bot stopped responding to Jeffrey's messages. Root cause: `auth-profiles.json` was missing from `~/.openclaw_prod/agents/main/agent/` (the prod state dir). HTTP `/health` returned `{"ok":true,"status":"live"}` throughout — every diagnostic check looked green while every LLM call silently failed with "No API key found for provider anthropic".

The plist had also been using `~/.openclaw-production/` (wrong dir) instead of `~/.openclaw_prod/` (the deploy.sh target), which was fixed live via `install-launchagents.sh`. This PR hardens the system so neither failure can recur silently.

Beads: orch-zmw (deploy seeding), orch-3fz (doctor STATE_DIR), orch-65m (canary auth probe), orch-rp8 (CLAUDE.md rule)

## Goals
1. Canary catches missing auth-profiles before and after deploy
2. deploy.sh seeds auth-profiles.json into prod (fail-close if missing everywhere)
3. doctor.sh validates plist OPENCLAW_STATE_DIR matches deploy target AND auth-profiles present
4. CLAUDE.md documents the liveness-not-functional trap permanently

## Tenets
- Fail-closed: missing auth is a hard failure, not a warning
- Liveness probes must not lie: check 8 makes the contract explicit
- Smallest safe change per file

## High-level description of changes
- **staging-canary.sh**: Added check 8 (7→8 total) — verifies `auth-profiles.json` exists in the state dir corresponding to the probed port (18789→prod, else staging). HTTP liveness alone is insufficient proof.
- **deploy.sh Stage 3**: After syncing config, checks if `$PROD_DIR/agents/main/agent/auth-profiles.json` exists. Seeds it from staging if missing; hard-fails deploy if missing from both.
- **doctor.sh**: Added two checks after the port-match check: (1) plist `OPENCLAW_STATE_DIR` must resolve to `~/.openclaw_prod`; (2) `auth-profiles.json` must exist in that state dir.
- **CLAUDE.md**: Added liveness≠functional callout under Gateway section, referencing check 8 and doctor.sh as the verific
