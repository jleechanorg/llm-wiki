## Background

Two shell scripts referenced throughout `agent-orchestrator.yaml` agentRules did not exist on disk, causing the CR-loop-guard protocol to silently no-op:
- `scripts/extract-unresolved-comments.sh` тАФ referenced in `changes-requested` reaction
- `scripts/cr-loop-guard.sh` тАФ referenced in `agentRules`, `agent-stuck`, and `changes-requested` reactions

## Goals

- Scripts exist at the paths the agent expects
- Both scripts produce correct output when run against real PRs
- Loop-count guard prevents `@coderabbitai all good?` spam when SHA hasn't changed

## Changes

### `scripts/extract-unresolved-comments.sh`
Fetches unresolved CR inline comments via `gh api .../comments`, filters to `coderabbitai[bot]`, sorts by severity (Critical > Major > Normal > Nit), outputs one per line.

### `scripts/cr-loop-guard.sh`
Compares PR head SHA vs CR formal review `commit_sha`. Three outputs:
- `cr-trigger` тАФ new commits since last CR review; agent may ping CR
- `copilot-expanded` тАФ no SHA change, loop count < 3; agent should run `/copilot-expanded`
- `skip` тАФ loop limit (3) reached without SHA progress; escalate

Loop counts stored in `$HOME/.agent-orchestrator/guard-data/loop-counts.json`.

## Testing

- Smoke-tested `cr-loop-guard.sh` against PR #439 (CR `commit_sha=null`, returned `cr-trigger` with incremental-stall detection)
- Both scripts: executable bit set, `set -euo pipefail`, proper error messages
