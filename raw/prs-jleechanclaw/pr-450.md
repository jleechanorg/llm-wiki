## Summary
A dismissed CodeRabbit `CHANGES_REQUESTED` review does not mean the issues were resolved — it means the review was hidden. The previous CR gate logic only checked the latest review state, so a `DISMISSED` or `COMMENTED` review after a dismissed `CHANGES_REQUESTED` could incorrectly pass the gate without a fresh `APPROVED`.

## Background
A dismissed CodeRabbit `CHANGES_REQUESTED` review does not mean the issues were resolved — it means the review was hidden. The previous CR gate logic only checked the latest review state, so a `DISMISSED` or `COMMENTED` review after a dismissed `CHANGES_REQUESTED` could incorrectly pass the gate without a fresh `APPROVED`.

Fixes [orch-n0l](https://github.com/jleechanorg/jleechanclaw/issues?q=is%3Aissue+orch-n0l).

## Changes
**`.github/workflows/skeptic-cron.yml`**: Replace the simple "latest CR review state" gate with a timestamp comparison:

- Find the timestamp of the most recent `CHANGES_REQUESTED` (or `DISMISSED`, which was a dismissed `CHANGES_REQUESTED`)
- Find the timestamp of the most recent `APPROVED`
- Only pass gate 3 if `APPROVED` is newer than `CHANGES_REQUESTED`
- Fallback to verdict-comment detection unchanged (for CR incremental-stall cases)

- Verified on PR #446 (CR: CHANGES_REQUESTED → DISMISSED → CHANGES_REQUESTED → APPROVED): jq correctly returns `true` (APPROVED after last CHANGES_REQUESTED)
- All 395 existing tests pass
