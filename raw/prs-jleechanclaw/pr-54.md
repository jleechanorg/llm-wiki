## Summary

- Adds `src/orchestration/reconciliation.py` implementing the stuck-task detection loop
- Adds `src/tests/test_reconciliation.py` with 11 tests covering all required scenarios
- No auto-transition: reconcile_once returns stuck IDs only, human review required

## Changes

- **reconciliation.py**: `run_tmux_sessions()` uses `tmux ls -F #{session_name}` with `timeout=10` and nonzero-returncode guard; `get_stuck_tasks()` calls `list_tasks(board_id, status="in_progress")` and matches by task_id prefix in session names; `reconcile_once()` detects and logs stuck tasks but never mutates state
- **test_reconciliation.py**: 11 tests — 4 for `get_stuck_tasks`, 4 for `run_tmux_sessions`, 3 for `reconcile_once` (including explicit mock assertion that `update_task` is never called)

## Testing

```
PYTHONPATH=src python -m pytest src/tests/test_reconciliation.py -v
11 passed in 0.02s
```

All required scenarios covered:
- `get_stuck_tasks` returns tasks with no matching session
- `get_stuck_tasks` returns empty list when all tasks have sessions
- `run_tmux_sessions` returns empty set when tmux not running (FileNotFoundError, TimeoutExpired, nonzero exit)
- `reconcile_once` returns stuck task IDs
- `reconcile_once` does NOT call `update_task` (verified with Mock assertion)

## Known Limitations

- `MissionControlClient` is imported under `TYPE_CHECKING` only; the concrete client must expose `list_tasks(board_id, status=...)` matching this signature at runtime
