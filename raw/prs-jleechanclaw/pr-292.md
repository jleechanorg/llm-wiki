## Background

After merging PR #278 (https://github.com/jleechanorg/jleechanclaw/pull/278) which added empty-body CR APPROVED detection to the merge gate, we ran an E2E test dispatching 9 open PRs to agento via /claw. All 9 PRs reached CI-clean + mergeable, but **0/9 achieved genuine CR approval**. The agents saw `state: APPROVED` and stopped — not realizing `body=0` means it was auto-triggered by thread resolution.

## Goals

- Agents should detect empty-body APPROVED as fake and take corrective action (`@coderabbitai all good?`)
- Close the gap between merge_gate detection (PR #278) and agent awareness

## Tenets

- Config-first: policy changes in SOUL.md and agent-orchestrator.yaml, no Python changes
- Harness over PR: fix the systemic pattern, not individual PRs
- Fail-closed: agents must verify, not assume

## High-level description of changes

Two policy files updated to teach agents the empty-body APPROVED trap:

1. **SOUL.md** — New "Empty-body APPROVED is NOT genuine" section in PR Work Protocol with verification command and remediation steps
2. **agent-orchestrator.yaml** — Updated green criterion #3 to require body>0, updated post-push loop to verify genuine approval, updated approved-and-green trigger message to warn about the trap

## Testing

- E2E test: dispatched 9 PRs via /claw, observed agents plateau at empty-body APPROVED (0/9 genuine) — this PR would have prevented the plateau
- Unit tests: 1137 passed (16 pre-existing failures unrelated to this change)
- No Python code changed — config-only PR

## Low-level details

- **workspace/SOUL.md**: Added verification section with gh api command to check body length, explains genuine vs fake APPROVED, instructs agents to post @coderabbitai all good? when body=0
- **agent-orchestrator.yaml**: Updated criterion #3 definition, added VERIFY CR APPROVAL IS GENUINE section to post-push loop, updated approved-and-green trigger to check body length before declaring green
