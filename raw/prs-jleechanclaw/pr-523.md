## Raw Body
## Background
Ralph sessions were being flagged as stuck or killed by the orchestrator because they weren't responding to heartbeat polls. This happened because ralph runs agents (like Claude Code) by piping input, which prevents the agent from seeing interactive heartbeat polls typed into the tmux pane.

## Goals
Ensure ralph always responds to orchestrator heartbeat polls, even when the agent is busy or between iterations.

## Tenets
- Robustness: Use background loops to monitor for polls.
- Non-interference: Write directly to TTY to avoid stdin buffer pollution.
- Compatibility: Support dynamic AO session names (ra-XXXX).

## High-level description
Added a heartbeat response mechanism to the terminal recorder loop in `ralph/lib/terminal_recorder.sh` and a dedicated background loop to the root `ralph.sh`. These loops detect the poll string on screen and respond with `HEARTBEAT_OK` via TTY write.

## Testing
- Reproduced the bug with a simulation script (`repro_ralph_heartbeat.sh`).
- Verified the fix with a validation script (`test_fix_ralph_heartbeat.sh`) confirming `HEARTBEAT_OK` response without stdin buffer pollution.
- Ran existing ralph tests (`bash ralph/tests/test_terminal_recorder.sh`).

## Low-level details
- `ralph/lib/terminal_recorder.sh`: Updated `terminal_record_start` to resolve current tmux session if needed and added heartbeat detection/response to the background loop.
- `ralph.sh`: Added background heartbeat loop and trap for cleanup.
