## Raw Body
### Background
Canary sender was always SLACK_USER_TOKEN (xoxp), conflating real user identity with bot canary checks. A dedicated bot token allows cleaner bot-to-bot canary verification.

### Goals
- Add OPENCLAW_MONITOR_CANARY_BOT_TOKEN env var for dedicated canary sender
- Implement token precedence: OPENCLAW_MONITOR_CANARY_BOT_TOKEN -> ~/.mcp_mail/credentials.json:SLACK_BOT_TOKEN -> SLACK_USER_TOKEN (legacy fallback)
- Auth-error retry: if preferred bot token fails with not_authed, missing_scope, etc., fall back to SLACK_USER_TOKEN before reporting false outage
- Update README docs

### Tenets
- Backward compatible — existing SLACK_USER_TOKEN behavior unchanged when new var not set

### High-level description of changes
Add resolve_canary_slack_token() function in monitor-agent.sh with 3-level precedence chain (env var -> credentials.json -> SLACK_USER_TOKEN); is_placeholder_token() guard at all levels; hydrate OPENCLAW_MONITOR_CANARY_BOT_TOKEN from bashrc; auth-error retry on both chat.postMessage and conversations.replies with break-fast on exhausted fallback; update README env var table.

### Testing
Manual review of token precedence logic. Backward compatible: unset var falls through to legacy path.

### Low-level details

- **README.md**: Updated OPENCLAW_MONITOR_RUN_CANARY row; added new OPENCLAW_MONITOR_CANARY_BOT_TOKEN row
