## Background

The gateway architecture previously lacked clean separation between staging and production. Changes applied directly to ~/.openclaw/ affected the live gateway immediately with no validation gate. Multiple P0 outages in March 2026 (config drift, native module ABI mismatches, Slack token issues) resulted from untested changes hitting production.

## Goals

1. Separate staging (~/.openclaw/, port 18810) and production (~/.openclaw_prod/, port 18789) into distinct directories
2. Create a deploy pipeline that validates staging before updating production
3. Install staging gateway as a managed LaunchAgent alongside production

## Tenets

- Staging validates before production touches anything
- Production dir uses symlinks to shared resources (extensions, agents, credentials) -- only openclaw.json is copied
- Both gateways are managed by launchd (KeepAlive, ThrottleInterval=30)

## High-level description of changes

Introduces the ~/.openclaw_prod/ directory as the production gateway state dir. The production gateway plist now reads config from this dir instead of ~/.openclaw/. A new deploy.sh script gates production deployments behind staging canary validation (6-point check). The installer creates the prod directory structure and installs both gateway plists.

## Testing

- Staging canary: 6/6 PASS on port 18810
- Production canary: 6/6 PASS on port 18789
- Both gateways verified healthy via /health endpoint
- launchctl print confirms prod gateway reads from ~/.openclaw_prod/openclaw.json
- Installer tested: creates prod dir, symlinks, installs both plists
