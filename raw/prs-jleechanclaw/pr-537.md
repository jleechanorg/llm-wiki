## Raw Body
### Background
Deploy and monitor runs were intermittently failing on prod heartbeat checks while the gateway process was otherwise reachable. At the same time, AO dashboard launchd wiring remained default-on, which kept re-enabling dashboard behavior even when the operator wanted HTTP-first validation only.

### Goals
1. Stop auto-enabling dashboard behavior unless explicitly requested.
2. Reduce false/indefinite monitor hangs by bounding direct HTTP health probe timings.
3. Lower gateway event-loop pressure during validation runs.
4. Persist operator preference to avoid regression.

### Tenets
- Prefer harness fixes over one-off operator workarounds.
- Keep defaults safe and non-intrusive (opt-in GUI).
- Fail fast with bounded probe timeouts, not indefinite waits.
- Preserve explicit override paths for advanced/debug use.

### High-level description of changes
This PR makes AO dashboard installation opt-in via `OPENCLAW_INSTALL_AO_DASHBOARD=1`, and actively `bootout/disable`s the dashboard label when not enabled. It also sets launchd raw stream logging defaults to off and adds connect/max timeouts to monitor-agent HTTP gateway probes (initial, post-phase1, post-phase2), preventing unbounded curl waits. Finally, it records this operator preference in durable memory/learnings docs.

### Testing
- Prod gateway health (live): `curl -m 5 http://127.0.0.1:18789/health` returned `{"ok":true,"status":"live"}`.
- Prod canary: `bash scripts/staging-canary.sh --port 18789` passed `9/9` (`RC=0`), including heartbeat response.
- Health stability loop: 20 repeated `/health` calls (`-m 3`) all returned `200` with low latency.
- launchd dashboard state: `launchctl print-disabled gui/$(id -u)` shows `ai.agento.dashboard => disabled`.
