## Background

The AO PR poller (`ao-pr-poller.sh`) was silently failing to clean up stray worktrees at `/private/tmp/` paths every cycle. This caused `ao spawn` to fail with `fatal: already checked out` for PRs #311 and #310, leaving those PRs stuck indefinitely with no agent working on them.

Root cause analysis: `cleanup_zombie_worktree` had two bugs, identified via 5 Whys harness analysis in the `jleechanclaw` main session.

Beads: orch-tzc (cleanup bug), orch-gws (ao-* zombies), orch-i61 (test), orch-9yi (dead modules partial)

## Goals

1. Fix `cleanup_zombie_worktree` so stray worktrees at any path are actually removed
2. Add `ao-*` zombie session cleanup to stop 55+ dead sessions accumulating
3. Add regression test proving cleanup works on `/tmp`-style paths
4. Delete 3 truly dead orchestration modules (no live callers)

## Tenets

- No behavioral change to live code paths
- Verified via `python -m pytest tests/ -q` — same 6 pre-existing failures, no new ones
- Test proves the specific bug was fixed (real git worktree at /tmp path, verified removed)

## High-level description of changes

**Bug 1 — wrong grep key:** `cleanup_zombie_worktree` iterated `.git/worktrees/` metadata dirs and grepped for `^path $wt_path`. But git porcelain outputs `worktree /path` (not `path`). The grep never matched so the removal block never ran.

**Bug 2 — wrong path to `git worktree remove`:** Even if the grep had matched, the code would pass the `.git/worktrees/HASH` metadata dir to `git worktree remove` instead of the actual checkout path (e.g., `/private/tmp/jleechanclaw-orch-tqr`).

**Fix:** Replaced the broken loop with a single `awk` invocation that parses porcelain and extracts the actual checkout path for the target branch, then passes that to `git worktree remove --force`.

**ao-* zombies:** `cleanup_zombie_sessions` only matched `jc-*` sessions. Dead `ao-*` sessions from agent-orchestrator rate-limit failures were never reaped. Added separate cleanup loop.
