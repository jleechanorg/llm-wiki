## Raw Body
### Background
`agent-orchestrator.yaml` is now intentionally safe to track with shell placeholders like `${SLACK_WEBHOOK_URL:-...}`. In practice, AO launch paths in this repo were inconsistent: `bootstrap.sh` symlinked `~/agent-orchestrator.yaml` to the tracked repo file, `install.sh` copied that tracked file to `~/.agent-orchestrator.yaml`, and launchd/AO manager paths pointed at a mix of those locations. AO's Slack notifier rejects an uninterpolated placeholder in `webhookUrl`, so the live deploy ended up depending on a hand-edited local `~/.agent-orchestrator.yaml` instead of a reproducible install step.

### Goals
1. Make AO startup/install flows produce a concrete local runtime config automatically from the tracked repo YAML.
2. Keep the tracked repo config secret-free while preserving AO compatibility.
3. Make launchd and AO manager paths consistently target the rendered local runtime config.
4. Add regression tests so future config-path drift fails loudly.

### Tenets
- Keep secrets out of git-tracked files.
- Prefer one reproducible render/install path over manual local edits.
- Preserve compatibility for tools that still expect `~/agent-orchestrator.yaml`.
- Add harness checks around the installer contract, not just a one-off operational note.

### High-level description of changes
This PR adds a dedicated renderer script that turns the tracked `agent-orchestrator.yaml` into a concrete local runtime file at `~/.agent-orchestrator.yaml`, resolving shell placeholders from the current environment first and then from an interactive bash login shell so `~/.bashrc`-backed secrets are available. `bootstrap.sh` now renders that runtime file and repoints `~/agent-orchestrator.yaml` to it as a compatibility symlink. AO launch paths that should consume the concrete runtime file now default to `~/.agent-orchestrator.yaml`, and the docs now describe the tracked-source vs rendered-runtime split explicitly.
