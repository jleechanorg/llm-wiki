## Background

Bug orch-jp8u: workers performing their own 6-point green check via `/agento_report` read condition 4 which previously said `state == APPROVED AND body length > 0`. This single-condition check always fails because CodeRabbit consistently posts APPROVED reviews with `body=0` (body_len is always 0 on the APPROVED review itself — the walkthrough appears in a preceding COMMENTED review). As a result, `/agento_report` would always flag `NO_CR` even for genuine approvals.

The correct two-path detection logic already existed in `agent-orchestrator.yaml` (lines 48-51) and `CLAUDE.md` condition 3, but `agento_report.md` was not aligned with it.

## Goals

- Align `agento_report.md` condition 4 with the two-path CR approval criteria in `CLAUDE.md` and `agent-orchestrator.yaml`
- Eliminate false `NO_CR` status labels for genuine CR approvals
- Ensure workers using `/agento_report` use the same detection logic as the merge gate

## Tenets

- Criteria must be explicit and machine-checkable, not rely on implied knowledge
- All policy surfaces (CLAUDE.md, agent-orchestrator.yaml, agento_report.md) must be consistent

## High-level description of changes

Single changed file: `.claude/skills/agento_report.md`

- Condition 4 "CodeRabbit APPROVED" pass criteria updated from `body_len > 0` (single path, always false since CR never posts APPROVED with body>0) to the correct two-path criteria matching CLAUDE.md:
  - `body_len > 0` on the APPROVED review itself, OR
  - `body_len=0` APPROVED with confirming CR issue comment containing "all good" or "✅" posted after `@coderabbitai all good?`
  - Empty-body APPROVED with no confirming CR comment is explicitly labeled fake
- `NO_CR` status label description updated to match, clarifying the exact failure condition

No changes to `CLAUDE.md` or `agent-orchestrator.yaml` — those already had the correct two-path description.
