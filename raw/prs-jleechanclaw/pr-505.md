## Raw Body
### Background
2026-04-05 incident: immediately after a successful deploy (8/8 canary PASS), OpenClaw went completely dark. 3 \`openclaw-gateway\` processes competed for \`sessions.json.lock\` → WS pong starvation → total HTTP unresponsiveness (curl exit 28).

Root cause: \`deploy.sh\` Stage 4 used \`launchctl stop\` + \`launchctl start\` without killing orphaned processes not tracked by launchd. Multiple instances can coexist on the same state dir silently past the old 8-check canary.

### Goals
- Prevent multiple concurrent \`openclaw-gateway\` instances after any deploy/restart
- Make the canary detect this class of failure before it manifests

### Tenets
- Fail loudly and early (hard failure in deploy.sh, not just a warning)
- Verify at the right abstraction level (process count, not just HTTP 200)

### High-level description of changes
1. **deploy.sh Stage 4**: before \`launchctl start\`, kill ALL orphaned \`openclaw-gateway\` processes and clear stale session locks in both state dirs. After start, assert exactly 1 process is running (hard die if count ≠ 1).
2. **staging-canary.sh**: add check 9 — single-instance assertion. Bumps all step numbers from 8→9 total.
3. **CLAUDE.md**: document the single-instance mandate + 2026-04-05 root cause for future agents.

### Testing
- Ran 9/9 canary against prod after recovery: PASS (including check 9 confirming 1 instance)
- Check 9 correctly caught the 3-orphan state before recovery (1 FAIL output)

### Low-level details
- **scripts/deploy.sh**: \`_orphan_count\` check + \`pkill -x openclaw-gateway\` before restart; \`_clear_stale_locks()\` helper for both state dirs; post-start \`_running_gw\` count assertion with hard die
- **scripts/staging-canary.sh**: check 9 added before summary block; step numbers bumped 8→9 throughout
- **CLAUDE.md**: new "Gateway restart — single-instance mandatory" section with fix commands and 2026-04-05 incident note
