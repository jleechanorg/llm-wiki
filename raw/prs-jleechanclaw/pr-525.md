## Background
PR #383 in `agent-orchestrator` experienced a race condition where the `approved-and-green` reaction fired before all CI checks had completed. This was caused by loose verification logic that only looked for the absence of `FAILURE` conclusions, missing `IN_PROGRESS` or `PENDING` states.

## Goals
- Ensure `approved-and-green` reaction only fires (posts green signal) when all CI checks are truly finished and successful.
- Prevent firing when `mergeStateStatus` is `UNSTABLE`, which recently blocked a merge despite loose CI checks passing.

## Tenets
- Safety over speed: Wait for all signals to be positive before declaring a PR green.
- Robustness: Use `statusCheckRollup` and `mergeStateStatus` for more granular state awareness than simple commit status.

## High-level description
Updated the AO reaction message instructions and the `ao7green-pr-monitor.sh` script to use a strict `jq` filter on `statusCheckRollup`. The filter now catches anything that is not `COMPLETED` or not one of `SUCCESS`/`SKIPPED`/`NEUTRAL`. It also enforces a `CLEAN` merge state.

- Ran a dry run of `ao7green-pr-monitor.sh` on the `jleechanclaw` repo to confirm it correctly identifies non-green PRs as "concerning".
