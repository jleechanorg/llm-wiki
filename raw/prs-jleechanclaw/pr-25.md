## Raw Body
## Motivation
OpenClaw reminder/scheduling operations were documented in a few places with system `crontab` usage. That creates drift from the intended gateway-managed scheduling model and risks accidental host-level automation changes.

## What Changed
- Added explicit guardrails in canonical operator docs:
  - `AGENTS.md`
  - `TOOLS.md`
  - `openclaw-config/TOOLS.md`
- Updated runbooks/docs to remove OpenClaw crontab guidance and point to gateway cron + launchd where appropriate:
  - `README.md`
  - `AUTO_START_GUIDE.md`
  - `SETUP.md`
  - `openclaw-config/AUTO_START_GUIDE.md`
  - `openclaw-config/BACKUP_AND_RESTORE.md`
  - `docs/openclaw-backup-jobs.md`
- Added guardrail checker script:
  - `scripts/check-openclaw-cron-guardrail.sh`
- Wired guardrail checker into lint/check workflows:
  - `package.json` (`check:cron-guardrail`, plus included in `check`)
  - `scripts/run_lint.sh`
- Updated backup installers to stop adding system crontab entries and remove legacy OpenClaw cron lines:
  - `scripts/install-openclaw-backup-jobs.sh`
  - `openclaw-config/enable-auto-backup.sh`
  - `scripts/setup-openclaw-full.sh`

## How This Prevents Accidental System Crontab Usage
- Operator docs now clearly state:
  - forbidden: system `crontab` edits for OpenClaw reminder/scheduling jobs
  - required: OpenClaw gateway cron workflow (`openclaw cron ...`)
- New check script fails when OpenClaw-related `crontab` usage is introduced in tracked docs/scripts.
- Existing installer scripts were changed so they no longer add OpenClaw crontab entries, and they proactively clean up legacy entries.

## Testing
- `bash -n scripts/check-openclaw-cron-guardrail.sh scripts/install-openclaw-backup-jobs.sh scripts/setup-openclaw-full.sh openclaw-config/enable-auto-backup.sh scripts/run_lint.sh`
- `bash scripts/check-openclaw-cron-guardrail.sh`
- `pnpm check:cron-guardrail`
