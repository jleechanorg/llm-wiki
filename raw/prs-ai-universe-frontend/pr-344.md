## Summary
Fixes a race condition that prevented conversations from loading when navigating directly to a URL with `?conversationId=...` parameter.

## Problem
When navigating directly to a URL like `/?conversationId=HzHLNr2HnR8jZDT8lsTL`, the conversation messages wouldn't load. This was caused by a race condition in `useConversations`:

1. Hook initializes with empty `userId` (auth still loading)
2. URL effect reads conversationId and calls `setActiveConversation`
3. `setActiveConversation` captures the current (empty) userId in `activeConversationOwnerRef`
4. Auth completes, `userId` becomes valid
5. Messages query has `enabled` condition: `activeConversationOwnerRef.current === userId`
6. This fails because ref holds `''` but userId is now `'test-user-123'`
7. Query never runs, messages never load

## Solution
Remove the stale ref check from the query's `enabled` condition. Security is maintained by:
- Server-side validation of user access to conversations
- The user-switching effect that clears state when users actually change

## Changes
- `src/hooks/useConversations.ts`: Simplified `enabled` condition to `!!activeConversationId && !!userId`
- `tests/useConversations.urlRaceCondition.test.tsx`: Added unit tests for the race condition fix
