# Fix: Rate Limit Reset Time Uses Retry-After Header

## Problem

**Bug reported by Cursor bot**: The `buildRateLimitResponse` function in `mcpShared.ts` ignores the `retryAfter` parameter for reset time calculation.

- **Line 316**: `throw new RateLimitError(buildRateLimitResponse())` called with NO arguments
- **Impact**: All rate limit errors default to 1-hour reset time regardless of server value
- **User experience**: Incorrect countdown timers (e.g., "60 minutes" when should be "5 minutes")

This differs from `mcpClient.ts` which correctly uses `ClientRateLimitError.retryAfter` to calculate accurate reset times.

## Solution (TDD Approach)

### RED Phase: Write Failing Tests

Added 3 new tests in `src/services/__tests__/mcpShared.test.ts`:

1. **Valid Retry-After header** (300 seconds) → should calculate 5-minute reset time
2. **Missing Retry-After header** → should default to 1 hour
3. **Invalid Retry-After value** ("invalid") → should gracefully default to 1 hour

**Initial test result**: ❌ FAILED - Expected ~300s, got ~3600s (defaulting to 1 hour)

### GREEN Phase: Implement Fix

**Code change** in `src/services/mcpShared.ts:317-321`:

```typescript
// Extract retry-after from HTTP header (in seconds)
const retryAfterHeader = response.headers.get('Retry-After')
const retryAfter = retryAfterHeader ? parseInt(retryAfterHeader, 10) : undefined

throw new RateLimitError(buildRateLimitResponse(retryAfter))
```

**Test result**: ✅ PASSED - All 3 tests now pass with accurate reset times
