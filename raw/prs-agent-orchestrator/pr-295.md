## Background

The lifecycle-worker PID-file lock mechanism was already implemented (orch-886k) to prevent duplicate workers per project. However, there was no recovery path when the lock file is stale and an operator needs to force-restart the worker.

## Goals

- Add `--force` flag to `ao lifecycle-worker <project>` to bypass the startup lock and overwrite the PID file for recovery scenarios.

## Tenets

- Minimal change — only the override path, no restructuring of the existing lock logic.
- Idempotent: `--force` is safe to use even when no stale lock exists.

## High-level description

- Added `--force` boolean option to the `lifecycle-worker` command.
- When `--force` is set: skips `tryAcquireLifecycleLock`, directly calls `writeLifecycleWorkerPid` to overwrite any stale PID, and logs a yellow warning.
- Normal path (no `--force`): unchanged — full lock + verification.
- Added unit tests for `writeLifecycleWorkerPid`, `getLifecyclePidFile`, and `clearLifecycleWorkerPid` covering the recovery path.
