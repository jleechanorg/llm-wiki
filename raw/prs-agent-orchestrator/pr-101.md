# PR #101: [agento] feat(scm-github): add TTL cache + in-flight dedupe for gh API calls

**Repo:** jleechanorg/agent-orchestrator
**Merged:** 2026-03-22
**Author:** jleechan2015
**Stats:** +469/-20 in 6 files

## Summary
- Adds `gh-cache.ts`: a shared 15-second TTL cache with in-flight request deduplication for all `gh` CLI read calls in the scm-github plugin
- Caches `gh api`, `gh pr view`, and `gh pr checks` by full arg array + cwd key
- Concurrent sessions targeting the same PR now share one network round-trip per poll cycle
- Exports `getGhCache()` and `GhCache.metrics` for observability
- Zero impact on write operations (merge, close, create — never cached)

## Background
The lifecycle manager polls every 30s per session. With 10 sessions targeting the same PR, that was 40+ `gh` CLI invocations per cycle, exhausting rate limits quickly. This cache ensures identical calls within the TTL window are served from memory.

> [!NOTE]
> **Medium Risk**
> Introduces a shared in-memory cache and in-flight deduplication around GitHub CLI reads, which could surface short-lived stale results or subtle keying/invalidations issues under concurrency (though write operations are explicitly excluded).
> 
> **Overview**
> Adds a shared `GhCache` (15s TTL + in-flight request dedupe) to the `scm-github` plugin so repeated `gh` *read* invocations (`gh api`, `gh pr view`, `gh pr checks`) reuse results and collapse concurrent identical calls into a single execution.
> 
> Updates the `ghWithRetry` wrapper to bypass caching for any detected write-style args, and exports `getGhCache`/`_resetGhCache` (plus a new package export `./gh-cache`) for observability and test control; tests were added/updated to validate cache semantics and ensure writes are never deduped, with integration/unit suites resetting the singleton between cases.
