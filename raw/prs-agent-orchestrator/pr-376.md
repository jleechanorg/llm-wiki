## Summary

Design doc for embedding proactive evolve loops into existing manager tmux sessions (ao-orchestrator, jc-orchestrator, etc.).

## What it covers

- Gap analysis: lifecycle-manager is reactive-only, orchestrator sessions are inert between events, no per-project evolve loop exists
- Architecture: lightweight OBSERVE every poll cycle; full MEASURE→DIAGNOSE→PLAN→FIX→RECORD on anomaly
- Shared knowledge base (~/.ao-evolve-knowledge/*.jsonl) for cross-manager coordination — no master coordinator needed
- Safety: autonomousFixScopes cap, escalation triggers, kill switch, anti-stall rules
- Composio contrast: reactive/event-driven vs. proactive/continuous
- 4-file implementation scope (types.ts, config.ts, orchestrator-prompt.ts, this doc)
