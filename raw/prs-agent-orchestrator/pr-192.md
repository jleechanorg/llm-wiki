## Background
`getPendingComments` in `scm-github` calls `gh api graphql` to get review threads with `isResolved` status. When GraphQL quota is exhausted (5000/hr), the call fails with no REST fallback -- the lifecycle-worker stalls on `changes-requested` detection, leaving workers idle for up to 1hr.

## Goals
- Prevent lifecycle-worker stalls when GraphQL is exhausted
- Graceful degradation to REST API approximation

## Tenets
- Prefer real data (GraphQL `isResolved`) when available; fall back to REST approximation
- Never return an error that blocks the lifecycle loop -- return partial data instead

## High-level description of changes
Added a REST fallback inside `getPendingComments` catch block. When GraphQL throws a rate-limit error, the function now:
1. Fetches inline PR comments via `GET /repos/{owner}/{repo}/pulls/{number}/comments`
2. Fetches issue comments via `GET /repos/{owner}/{repo}/issues/{number}/comments`
3. Returns them as `ReviewComment[]` -- `isResolved` is set to `false` (REST has no resolution state)

This commit also fixes the corresponding unit test which was asserting the old (broken) throw-on-rate-limit behavior -- replaced with a test that verifies the REST fallback is called and returns the correct comment shape.

## Testing
- Unit test updated: `getPendingComments REST fallback when GraphQL fails with rate-limit error`
- Unit test added: `throws on non-rate-limit error` (unchanged behavior for other errors)
- Existing `getPendingComments` tests still pass

## Files changed
- `packages/plugins/scm-github/src/index.ts` -- REST fallback in `getPendingComments` catch block
- `packages/plugins/scm-github/test/index.test.ts` -- test fix: REST fallback behavior + non-rate-limit throw
