## Background

Lifecycle-worker processes were vulnerable to PID recycling false positives: `kill -0` alone cannot distinguish between a genuine lifecycle-worker and a recycled PID now running something else. Repeated runs of `setup-extended.sh` could also leave duplicate workers running.

## Goals

1. Prevent PID recycling false positives in worker startup guard
2. Provide a reliable `ao update` that kills/restarts workers atomically
3. Prevent duplicate workers in `setup-extended.sh`
4. Add `ao doctor` visibility into worker process counts

## Tenets

- Prefer additive changes over modifying upstream core files
- PID verification via `ps` is safe on both macOS and Linux
- Graceful teardown (SIGTERM → SIGKILL fallback) for worker restarts

## Changes

### packages/cli/src/lib/lifecycle-service.ts
- Add `isLifecycleWorkerProcess(pid, projectId)` that uses `ps -p <pid> -o args=` to verify the process command contains `lifecycle-worker <projectId>`
- Falls back to `kill -0` when `ps` fails (process gone or permission denied)
- Update `getLifecycleWorkerStatus()` and `stopLifecycleWorker()` to use the new function

### scripts/ao-update.sh
- Add `kill_existing_workers()`: reads projects from canonical config, unloads launchd agents and kills PID-file workers
- Add `restart_workers()`: restarts workers via `ao lifecycle-worker <projectId> &` for all projects
- Call `kill_existing_workers` before git pull; call `restart_workers` after clean rebuild

### scripts/setup-extended.sh
- For each project, kill existing launchd worker (`launchctl unload`) and PID-file workers before writing and loading the new plist

### scripts/ao-doctor.sh
- Add `check_lifecycle_workers()`: counts lifecycle-worker processes per project via `ps aux` and warns on count >= 2

## Testing

- All 224 tests pass (18 test files)
- `pnpm build` succeeds
- Manual verification: `ps aux | grep lifecycle-worker` confirms worker count per project
