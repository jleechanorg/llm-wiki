## Background

P0 bug (jleechan-v7oa): `packages/core/src/lifecycle-manager.ts` detects merged/killed session status transitions but never calls `sessionManager.kill()` for the `killed` case, causing zombie tmux sessions to accumulate indefinitely. The merged case was already handled (bd-s4t.1), but killed was missed.

Known zombie session examples: `bb5e6b7f8db3-ao-302`, `-314`, `-315`. Related prior work: PR #86 (lifecycle-manager cleanup).

## Goals

- [x] Call `sessionManager.kill()` when session transitions to `killed` status
- [x] Preserve existing merged-case kill behavior
- [x] Add test asserting kill is called on killed-state transition
- [x] Add test asserting kill is called when PR is closed (→ killed)

## Tenets

- Minimal scope: only touch the terminal-status kill path
- Preserve ordering: exit proof validation and outcome recording run before kill
- Test coverage: both killed-via-runtime-dead and killed-via-PR-closed paths

## High-level description of changes

Extended the existing merged-status auto-kill block in `checkSession()` to also fire for `newStatus === "killed"`. The merged case already existed (bd-s4t.1); the killed case was simply absent. Both terminal statuses now trigger `sessionManager.kill()` in the same location, after exit proof validation and outcome recording, so the worktree remains available for commit proof checks.

## Testing

- Extended existing "detects killed state when runtime is dead" test: added `expect(mockSessionManager.kill).toHaveBeenCalledWith("app-1")` assertion
- Added new test "detects closed PR → killed and calls sessionManager.kill (jleechan-v7oa)": models the PR-closed → killed transition path, asserts both status and kill call
