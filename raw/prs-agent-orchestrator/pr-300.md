## Goals

Fix the check-run ordering bug that causes skeptic-cron to always see FAIL even when the newest Skeptic Gate run succeeded.

## Tenets

- Minimal change (one jq expression fix)
- Preserve fail-closed behavior (unknown/error still blocks)
- Cross-layer verification per Skeptic Change Verification Protocol

## High-level description of changes

The jq `last` selector in skeptic-cron.yml picks the **oldest** check-run from GitHub's API (which returns newest-first). When multiple Skeptic Gate runs exist for the same SHA (common after re-triggers), cron always sees the oldest FAIL run even when a newer run succeeded.

**Fix**: Replace `| last |` with `| sort_by(.started_at) | reverse | .[0].` to explicitly get the newest run.

**Root cause analysis**: PR #293 had 5 Skeptic Gate runs on SHA `1d30614`: 4 failures (21:05-21:08) and 1 success (21:21). The cron at 21:29 used `last` which got the 21:05 failure, not the 21:21 success.

## Testing

- Verified API returns newest-first for check-runs on SHA `1d30614`: `[success 21:21, failure 21:08, failure 21:06, failure 21:05, failure 21:05]`
- `last` → oldest (failure), `sort_by(.started_at) | reverse | .[0]` → newest (success)
- Traced the full skeptic-cron 21:29 run log confirming it saw 'failure' for PR #293 despite check-run success
