## Raw Body
## Background

Workers post green signal (6/6 criteria met) but PRs stall permanently — the lifecycle-manager fires the `approved-and-green` reaction and calls `scm.mergePR()`, but the merge gate fails because CI is still completing when the PR transitions to mergeable. The `ao-pr-poller` removal in PR #352 left no replacement for the merge step.

Fix: workers should run `gh pr merge --auto` after green signal + 5min wait.

## Goals

- PRs that reach 6/6 green merge automatically without stalling
- GitHub's native `--auto` flag handles CI completion race condition
- Merge executes within ~5 minutes of green signal (matching issue spec)

## Tenets

- Prefer GitHub native features over polling/retry loops
- Non-breaking: all existing tests pass
- Configurable: `autoMergeWaitSeconds` is an optional reaction parameter

## High-level description

Add `autoMergeWaitSeconds` parameter to the `approved-and-green` reaction config. When > 0, `scm.mergePR()` passes `--auto` to GitHub which waits for required status checks to complete before merging. This eliminates the race where the PR is mergeable but CI hasn't fully settled.

Changes:
- **types.ts**: Add `autoMergeWaitSeconds?: number` to `ReactionConfig` and `mergePR()` signature
- **scm-github/index.ts**: `mergePR()` accepts `autoWaitSeconds`; passes `--auto` to gh CLI and `head_branch` to REST API
- **lifecycle-manager.ts**: Pass `autoMergeWaitSeconds` from reaction config into `scm.mergePR()`
- **agent-orchestrator.yaml**: Set `autoMergeWaitSeconds: 300` on `approved-and-green` reaction (5 min)
