## Raw Body
### Background
`~/.openclaw/agent-orchestrator.yaml` had effectively become both the staging config and the production source of truth, and several scripts plus `ao start` flows could still create or prefer repo-local `agent-orchestrator.yaml` files. That created namespace-shadowing risk because AO hashes `dirname(configPath)` and resolves symlinks before hashing. The user explicitly corrected the intended model: staging under `~/.openclaw`, production under `~/.openclaw_prod`, no staging symlink to prod, and production updated only after validation.

### Goals
1. Make staging and production AO/OpenClaw configs separate real files with canonical paths.
2. Provide a portable bootstrap/install/promote flow for new machines.
3. Prevent `ao start` and helper scripts from reintroducing split-brain config namespaces.
4. Preserve explicit env override behavior and intentional legacy alias symlinks.

### Tenets
- Prefer managed canonical config paths over accidental repo-local shadow configs.
- Keep staging editable and production promotion explicit.
- Centralize topology logic instead of duplicating path heuristics across scripts.
- Add tests for both discovery rules and the bootstrap/promote shell flow.

### High-level description of changes
This adds a shared config-topology helper in core, updates config discovery to prefer managed staging/prod paths before local shadow configs, and changes `ao start` onboarding flows to bootstrap or mutate the staging canonical config instead of auto-writing repo-local `agent-orchestrator.yaml` files. On the shell side, it adds `scripts/bootstrap-openclaw-config.sh` and `scripts/promote-openclaw-config.sh`, centralizes topology rules in `scripts/lib/ao-config-topology.sh`, and updates doctor/update/setup/start-all/validate flows to use the new managed staging/prod layout and reject staging-prod symlink collapse.
