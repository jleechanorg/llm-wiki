## Background

The lifecycle backfill loop (`backfillUncoveredPRs`) currently only spawns workers for PRs with no session coverage. When a live agent session dies with an unresolved CHANGES_REQUESTED review from CodeRabbit, the PR becomes orphaned â€” no worker will pick it up until a human notices.

## Goals

- Spawn a fresh worker for any open PR with a CHANGES_REQUESTED review that has no active session
- Respect a per-cycle rate limit (max 2 such respawns per backfill cycle) to prevent spam
- Build coverage from alive sessions first, then prune only those whose tmux sessions are confirmed dead

## Tenets

- Fail-open on tmux probe errors (don't accidentally prune live sessions when the probe fails)
- Sanitize contributor-controlled CodeRabbit review body before including in spawn prompt
- Prefer build-from-alive over delete-from-full to avoid race conditions when two sessions cover the same PR

## High-level description

`backfillUncoveredPRs` now calls `getReviewDecision` for each uncovered PR. If the decision is `"changes_requested"` and no session currently covers the PR, it spawns a new worker via the configured CLI profile and claims the PR. A per-cycle counter prevents more than 2 such respawns per backfill invocation. The tmux liveness probe is fail-open: probe errors preserve the session as live rather than pruning it.
