## Raw Body
### Background
`gh pr view --json reviews` returns `author.login` WITHOUT the `[bot]` suffix (e.g. `coderabbitai` not `coderabbitai[bot]`). The GitHub REST API returns `[bot]` but the gh CLI strips it. This caused `evaluateCoderabbitApproval()` in `merge-gate-coderabbit.ts` to always return `{ passed: false, detail: "No CodeRabbit approval found" }` because `getLatestDecisiveReview()` found zero reviews matching the hardcoded `"coderabbitai[bot]"`.

Root cause discovered during evolve-loop cycle 4 (2026-03-30) while investigating why Skeptic always returned `CR: FAIL` even on PRs where CodeRabbit had APPROVED.

### Goals
- Make `evaluateCoderabbitApproval()` correctly detect CR approval regardless of whether the author string has `[bot]` suffix or not
- Fix zero-touch merge failures caused by this bug across all PRs

### Tenets
- Minimal change — only fix the author detection logic
- Be defensive — check both forms so it works with both gh CLI and REST API consumers

### High-level description of changes
Auto-detect which author form is present in the reviews array. If any review has author `"coderabbitai[bot]"`, use that; otherwise default to `"coderabbitai"` (the gh CLI form).

### Testing
- All 30 merge-gate and merge-gate-skeptic tests pass
- Pre-existing `wholesome.test.ts` and `config.test.ts` failures are unrelated to this change

### Low-level details
- **`packages/core/src/merge-gate-coderabbit.ts`**: Added `crAuthor` detection in `evaluateCoderabbitApproval()`. Detects presence of `"coderabbitai[bot]"` in reviews array; defaults to `"coderabbitai"` if not found. Added explanatory comment per existing code comment style.
