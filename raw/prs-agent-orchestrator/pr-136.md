## Background

The `agent-stuck` reaction fires after a session idles beyond a threshold (e.g., 15 min). However, it was only firing **once** — on the `working → stuck` transition — then never again, no matter how long the session stayed stuck.

Root cause: reactions only fire on status **transitions** (`newStatus !== oldStatus`). Once a session enters `"stuck"`, every subsequent poll returns the same status, hitting the `else` (no-transition) branch where no retry logic existed for stuck sessions. The session received one nudge message and was permanently abandoned if it didn't recover.

## Goals

- Re-send the `agent-stuck` nudge periodically when a session stays stuck across polls
- Use the configured `threshold` as the retry interval (fallback: 15 min)
- Mirror the existing pattern used for `"mergeable"` periodic retry

## Tenets

- Minimal change: touches only the no-transition else branch
- No new config surface: reuses existing `agent-stuck.threshold` as the retry cooldown
- Consistent with existing `mergeable` retry pattern

## High-level description of changes

In `lifecycle-manager.ts`, added a cooldown-retry block for `"stuck"` status in the `else` (no-transition) branch, parallel to the existing `"mergeable"` retry:

```typescript
if (newStatus === "stuck") {
  const reactionConfig = getReactionConfigForSession(session, "agent-stuck");
  if (reactionConfig?.action && reactionConfig.auto !== false) {
    const STUCK_RETRY_COOLDOWN_MS = parseDuration(reactionConfig.threshold) || 15min;
    if (now - lastAttempt >= STUCK_RETRY_COOLDOWN_MS) {
      executeReaction(...)  // re-send the nudge
    }
  }
}
```

Also adds `stuckRetryTimestamps` map and GC cleanup.
