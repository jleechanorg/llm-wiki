## Background

Workers silently substitute easier implementation paths when design docs exist in docs/design/ or specs/, bypassing the specified approach. This violates the principle that design docs encode decisions made for good reasons.

## Goals

- Prevent workers from silently substituting implementation paths when a design doc exists
- Require explicit stopping and PR comment when a designed approach is infeasible
- Ensure PR descriptions document which design elements were implemented

## What this PR does

This PR version-controls the bd-dfl bead entry documenting the design-fidelity enforcement rule.

The rule itself lives in `~/.openclaw/agent-orchestrator.yaml` (runtime config, not committed to repo). The `.beads/issues.jsonl` entry documents the decision and links it to its design origin.

## Scope clarification

This PR does NOT add code/config to the repo. It only:
- Adds `bd-dfl` to `.beads/issues.jsonl` as a closed task
- Documents that the enforcement rule was added to the runtime config

Future work (separate PR) could add the rule text to a repo-tracked example config (e.g. `docs/config-example/` or `agent-orchestrator.yaml.example`) if version-controlled enforcement is desired.
