## Raw Body
### Background
The `agent-orchestrator` review-fix loop stopped self-healing even though `changes-requested` reactions and `backfillAllPRs` were configured. The live failure was that `ai.agento.lifecycle-all` sat in launchd as `state = not running`, `agent-orchestrator` lost its lifecycle worker, and uncovered PR sessions remained stuck at `pr_open` while GitHub had already moved them to `CHANGES_REQUESTED`.

This turned out to be a harness failure, not a worker-quality problem. First, the watchdog was parsing `launchctl print` incorrectly, reducing `state = not running` to `not`, then classifying it as `UNKNOWN` instead of repairing it. Second, even after the wrapper was revived, child lifecycle workers still died immediately because the Slack notifier treated the checked-in placeholder webhook config as a fatal invalid URL.

### Goals
1. Restore self-healing when `ai.agento.lifecycle-all` is registered but dormant and one or more child lifecycle workers are missing.
2. Add regression coverage for the exact `launchctl` state formats that caused this outage.
3. Fail closed when `start-all.sh` cannot actually verify that all lifecycle workers came up.
4. Prevent optional notifier placeholder config from crashing lifecycle worker startup.

### Tenets
- Fix the harness, not just the symptom.
- Keep the watchdog shell-only so launchd does not gain a new runtime dependency.
- Add executable regression proof for the parser behavior that failed in production.
- Treat unresolved placeholder config as absent optional config, not as a process-fatal misconfiguration.
- Minimize scope to the lifecycle wrapper/watchdog path and the startup blocker that kept repaired workers from staying alive.
