## Background

A 24-hour audit found 8/12 merged PRs were NOT 6-green: 4 had CR CHANGES_REQUESTED, 2 had NO CR review, 3 had no skeptic verdict. The lifecycle-manager was calling `checkMergeGate()` but there was no named, explicit entry point that made the gate contract clear and testable. CLAUDE.md instructions are insufficient — agents bypass them.

## Goals

- Add a named `verify6Green(pr, scm, projectMergeGate?)` function to lifecycle-manager that blocks merges when any core gate fails
- Wire it into the `approved-and-green` reaction before any merge attempt
- Add comprehensive unit tests covering all 5 enforced gates

## Tenets

- **Fail-closed**: if any gate fails, the merge MUST be skipped — never merge a non-green PR
- **Backward compat**: respect the project's `mergeGate.enabled` setting so existing disabled projects continue working
- **Skeptic optional**: Gate 6 (Skeptic) and Gate 7 (Evidence) remain per-project optional gates, not hard-coded here

## High-level description of changes

### New function: `verify6Green(pr, scm, projectMergeGate?)`
Located in `lifecycle-manager.ts`, this is a thin wrapper over the existing `checkMergeGate()` that:
- Hard-codes `skepticRequired: false` (Skeptic is Gate 6, not part of the 6-green contract)
- Respects the project's `mergeGate.enabled` setting for backward compat
- Enforces all 5 core gates via `checkMergeGate`:
  - **Gate 1**: CI green
  - **Gate 2**: No merge conflicts
  - **Gate 3**: CodeRabbit APPROVED (actionable reviews only — APPROVED/CHANGES_REQUESTED; COMMENTED/DISMISSED ignored)
  - **Gate 4**: Bugbot clean (no error-severity cursor[bot] comments)
  - **Gate 5**: All inline comments resolved (GraphQL reviewThreads.isResolved; REST fallback is fail-closed)

### Auto-merge reaction update
The `auto-merge` reaction case now calls `verify6Green(freshSession.pr, scm, project.mergeGate)` instead of the previous inline `checkMergeGate` call. If the gate fails, the merge is skipped and the blocker is logged.
