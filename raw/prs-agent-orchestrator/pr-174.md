## Background
Merged/closed PRs were leaving zombie tmux sessions that accumulated past the 15-session spawn gate, blocking new spawns. The session-reaper skipped terminal-status sessions assuming they were already dead, but the lifecycle-manager only killed sessions on `merged` status — `killed` (closed PRs) was missed, and the reaper had no explicit PR-state check.

## Goals
- Zombie tmux sessions are killed immediately when their PR transitions to merged or closed.
- The spawn gate is no longer blocked by zombie sessions.

## Tenets
- Prefer real cleanup over reaper safety nets; reaper is the fallback only.
- Never double-kill: reaper skips terminal-status sessions already handled by lifecycle-manager.
- PR state must be persisted in metadata so the reaper can check it without SCM access.

## High-level description of changes
1. **lifecycle-manager.ts**: Extended the direct kill to also fire when `newStatus === "killed"` (closed PRs). Added `prState` persistence to session metadata on every SCM poll.
2. **session-reaper.ts**: Raised `maxKillsPerRun` from 5 → 15. Added explicit zombie kill for sessions whose `session.pr?.state` is `"merged"` or `"closed"`.
3. **types.ts + metadata.ts + session-from-metadata.ts**: Added `SessionMetadata.prState` and `PRInfo.state` fields with full read/write plumbing from SCM → metadata → session object.
