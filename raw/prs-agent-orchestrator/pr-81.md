## Background

Diagnosed via /auton (bead bd-wse): 15 concurrent sessions caused GitHub API rate limit exhaustion, blocking all PR progress. Root cause in lifecycle-manager.ts and lifecycle-worker.ts.

## Goals

1. Stop concurrent session polling from firing N*4 GitHub API calls simultaneously
2. Stop all lifecycle-workers from polling at T=0 on restart

## Tenets

- Minimal change: fix the polling model, do not redesign it
- Preserve allSettled semantics: single failing session must not block others

## High-level description of changes

Fix 1 - Sequential session polling (lifecycle-manager.ts): Changed pollAll() from Promise.allSettled to a sequential for-loop with per-iteration .catch(). 12 sessions x 4 calls = 48/cycle at 30s = 5760/hr previously; now serialized.

Fix 2 - Startup jitter (lifecycle-worker.ts): Random 0-10s delay before lifecycle.start(). Without this, all workers poll at T=0 on restart, immediately exhausting the rate limit.

## Testing

- 54 pre-existing lifecycle-manager tests: all pass (55 total with new test)
- New test: verifies sequential loop does not short-circuit when one checkSession rejects

- packages/core/src/lifecycle-manager.ts: replace Promise.allSettled with sequential for-loop + .catch()
- packages/cli/src/commands/lifecycle-worker.ts: add startup jitter before lifecycle.start()
