## Raw Body
### Background
When a PR merged, co-worker sessions that had no PR assignment were left running indefinitely. The `session-reaper.ts` module existed with a `noPrThresholdMs` (default 4h) to handle this, but it was never wired into the lifecycle-worker — dead code.

This came up because after a wave of merges cleared all open PRs in agent-orchestrator, 3 workers (`ao-748`, `ao-749`, `ao-750`) were left idle with no PR, waiting for direction that would never come.

### Goals
- Reap idle no-PR sessions immediately when a PR merges, not nightly
- Wire the existing `session-reaper.ts` into an actual call site
- Keep reap errors non-fatal (must not break the merge transition)

### Tenets
- Minimal change — use existing `reapStaleSessions` unchanged
- Fail-safe — errors caught and logged, never propagated
- Scoped — only fires on `merged` transitions, not all terminal states

### High-level description of changes
In the `TERMINAL_STATUSES` block inside `checkSessionStatus`, after `sessionManager.kill()` for a merged session, call `reapStaleSessions` with a 5-minute `noPrThresholdMs`. This is much shorter than the 4h default — appropriate for a post-merge sweep where co-workers have clearly finished their work.

Added import for `reapStaleSessions` and `DEFAULT_REAPER_CONFIG` from `./session-reaper.js`.
