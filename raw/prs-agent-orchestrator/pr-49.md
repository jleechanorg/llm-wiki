## Background
GitHub GraphQL rate limit (5000/hr) was being exhausted by the lifecycle manager polling ~269 sessions every 30s with ~5 GraphQL calls each (161k calls/hr). REST API has a separate 5000/hr budget that was untouched.

## Goals
- When GraphQL rate limit is exhausted, fall back to REST API instead of failing
- Reduce API call volume by skipping exited sessions in lifecycle manager
- Maintain GraphQL as primary (more efficient) — REST is only a fallback

## Tenets
- GraphQL-first, REST-fallback — not a full conversion
- TDD: failing tests written before implementation
- Handle response format differences between GraphQL and REST APIs

## High-level description of changes
Added REST API fallback to ghWithRetry for gh pr view commands. When all 3 GraphQL retries are exhausted, the command is converted to gh api repos/{owner}/{repo}/pulls/{number} (REST). Each method that consumes the response handles the format differences between GraphQL and REST (e.g., merged PRs return {state: "MERGED"} in GraphQL but {state: "closed", merged: true} in REST).

Also skips sessions with activity === "exited" in the lifecycle manager's pollAll(), which eliminates ~90% of unnecessary API calls.
