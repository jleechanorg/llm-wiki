## Background

Merged PR sessions can still remain alive in tmux if lifecycle evaluation hits
runtime/activity killed paths before merged PR state is observed. This race creates
zombie tmux sessions tied to already-merged PRs.

## Goals

- Kill zombie tmux sessions when a merged PR transitions to killed before merged state is observed
- Avoid persisting killed state when SCM check fails, allowing retry
- Add regression tests for the killed-transition + merged PR race

## Changes

### packages/core/src/lifecycle-manager.ts

**bd-kki absorb check**: when determineStatus returns killed and session has a PR,
query SCM for PR state before persisting. Three outcomes:
- SCM confirms merged: persist killed, proceed with terminal processing
- SCM confirms not merged: skip persisting killed, next poll retries
- SCM unreachable: skip persisting killed, next poll retries

Restructures transition block to use effectiveStatus so absorbed transitions
return early without running reaction/notification/terminal logic.

**bd-kki belt-and-suspenders** (terminal status block): when entering killed
from a non-terminal oldStatus, query SCM and call sessionManager.kill() if PR
is merged. Guarded on TERMINAL_STATUSES.has(oldStatus) === false to avoid
redundant SCM calls when transitioning between terminal states.

### packages/core/src/__tests__/lifecycle-manager.test.ts

Four regression tests:
1. killed transition + merged SCM response -> sessionManager.kill() called
2. killed transition + SCM unreachable -> killed not persisted, retry next poll
3. killed transition + non-merged PR -> killed not persisted, retry next poll
4. killed transition from terminal status -> SCM check skipped (guard)
