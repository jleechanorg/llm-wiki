## Background

When GitHub GraphQL rate limits are exhausted (common when multiple AO agents are running simultaneously), `getCISummary` was catching the error and fail-closing to `"failing"`, and `getReviewDecision` was throwing uncaught. The lifecycle poller then fired `ci-failed` and `changes-requested` reactions on every poll cycle, spamming agents with \"CI is failing on your PR\" when the real issue was a transient rate limit.

Observed in session jcc-13: agent correctly assessed PR #319 as green via REST API, but lifecycle kept firing `ci-failed` â†’ agent ran `gh pr checks` â†’ rate limited â†’ loop at 56% context.

## Goals

- Stop spurious `ci-failed`/`changes-requested` reactions during GraphQL rate limit windows
- Let the poller retry silently on the next cycle instead

## Tenets

- Transient errors should not be treated as permanent failures
- Fail-closed is correct for unknown errors, not for known-transient ones

## High-level description of changes

In `getCISummary`: catch block now checks if the error message contains \"rate limit\" before fail-closing. If rate-limited, returns `\"none\"` (neutral) instead of `\"failing\"`.

In `getReviewDecision`: wraps the `gh pr view` call in try/catch; rate limit errors return `\"none\"` instead of throwing, which would propagate and potentially cause incorrect reaction firing.

- TypeScript build passes (`pnpm --filter \"*scm*\" build`)
- Manual verification: jcc-13 was in an infinite loop due to this exact bug
