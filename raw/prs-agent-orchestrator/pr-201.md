## Background

When `backfillUncoveredPRs()` fails to claim a PR (e.g. branch already checked out in another worktree), the session's `kill()` call fails silently because the session isn't fully registered yet. The worktree and its local branch are left behind, permanently blocking every subsequent backfill attempt for that PR.

Real-world impact: 55 dead worktrees accumulated in `~/.worktrees/jleechanclaw/`, blocking lifecycle-worker from claiming PRs #393 and #383 for over 13 hours.

## Goals

- Eliminate cascading poison: a single `claim_failed` must not permanently block a PR
- Clean up both worktree directory AND local branch on all cleanup paths
- Never accidentally delete user branches (main, master, develop, etc.)

## Tenets

- Prefer additive changes to existing files; minimize diff against upstream
- All cleanup is best-effort (wrapped in try/catch) — never crash the lifecycle-worker
- Only delete branches matching AO-managed patterns (`feat/*`, `session/*`, `fix/*`, etc.)

## High-level description of changes

Three cleanup paths were audited and fixed:

1. **`workspace-worktree destroy()`** (already implemented in prior commit): Captures checked-out branch before `git worktree remove`, then deletes it if AO-managed pattern.
2. **`backfill claim_failed` handler** (already implemented in prior commit): Direct worktree cleanup fallback when `sessionManager.kill()` fails, including branch deletion via `git branch -D`.
3. **`pruneStaleWorktrees()` in session-manager.ts** (this PR): Both Pass 1 and Pass 2 now capture the branch and delete it after worktree removal.

## Changes

### `packages/core/src/session-manager.ts`
