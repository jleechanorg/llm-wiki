## Background

The runtime-antigravity plugin's `create()` flow didn't match how Antigravity actually works — it clicked workspace labels instead of using the "Start new conversation" button, looked for separate conversation windows instead of operating within the Manager window, and tried to click Send buttons that are web-rendered and not in the A11y tree.

## Goals

Align the runtime-antigravity plugin with the proven `/antig` skill patterns from `~/.claude/skills/antigravity-computer-use/SKILL.md`.

## Tenets

- **TDD approach**: Tests written first encoding the correct flow, then implementation fixed to pass them
- **Skill-faithful**: Every interaction pattern directly translates from the canonical skill documentation
- **No-reinvention**: Uses the exact same screencapture + python3 PIL blue-pixel detection for Allow prompts

## High-level description

### runtime.ts — complete rewrite of create() and sendMessage()

- **create()**: Uses "Start new conversation" button (not workspace labels), scrolls sidebar UP first, pastes prompt with workspace path prefix, presses Return (not Send button), verifies via progress_activity, handles Allow prompt via screencapture + PIL
- **sendMessage()**: Always uses paste + Return (Send button is web-rendered in active convos)
- **destroy()**: Simplified — conversations persist in Manager, no window close needed
- **Session model**: windowId = managerWindowId (conversations live inside Manager, not separate windows)

### peekaboo.ts — two new functions

- `screencapture()`: Wraps macOS `screencapture -R` for region capture
- `clickCoordinates()`: Wraps `peekaboo click --coords` for screen-absolute coordinate clicks

### runtime.test.ts — comprehensive TDD test suite

New tests encoding the correct flow:
- Clicks "Start new conversation" button, not workspace label
- Uses Manager window for all operations (no separate conversation windows)
- Includes workspace path in prompt text
- Uses Return key to send (not Send butt
