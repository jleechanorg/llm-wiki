## Raw Body
## Summary

Implements upstream ComposioHQ PR #541's fix for issue #373: long tmux paste messages (>200 chars) frequently had their trailing Enter keystroke swallowed because a fixed 1000ms delay was insufficient for large messages.

### What changed

**packages/core/src/tmux.ts**:
- Extract `isLongMessage` check to a named const (DRY)
- Replace inline `new Promise(setTimeout)` with injectable `sleep()` helper wrapping `node:timers/promises`
- **Adaptive delay**: scales with message length — `1000ms + 200ms per KB`, capped at 2000ms, using `Buffer.byteLength` for correct UTF-8 byte counting (handles emoji/CJK correctly)
- **Enter retry loop** (messages >1KB): after sending Enter, capture pane before/after and retry Enter up to 3 times with 300ms backoff per attempt if output is unchanged (Enter was swallowed by slow agent processing)
- Export `tmuxInject()` so tests can inject `fakeExecFile` + `fakeSleep` without relying on `vi.mock` for `node:child_process` (which fails in Vitest ESM mode due to module caching)

**packages/core/src/__tests__/tmux.test.ts**:
- Replace `vi.mock("node:child_process")` factory approach with `tmuxInject()` — the `vi.mock` factory does not replace `execFile` in `tmux.ts` once the ESM module is already cached
- `beforeEach` calls `tmuxInject()` with fresh `vi.fn()` mocks and `vi.clearAllMocks()`
- 3 new retry-specific tests:
  - `retries Enter for >1KB messages when output unchanged` — all 3 retries fire, 6 captures, 4 enters
  - `breaks retry when output changes on third attempt` — output changes on attempt 2, breaks after 6 captures
  - `does not retry for short messages under 1KB` — 0 captures, 1 enter
