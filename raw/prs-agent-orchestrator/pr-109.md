## Raw Body
### Background

When send-to-agent delivers a message via tmux paste-buffer, a flat 300ms delay before Enter was insufficient for large messages. The Enter keystroke arrived before the paste completed, so the agent saw the pasted content but never responded -- requiring manual Enter in 8 sessions (ao-411 through ao-420).

The upstream fix (ComposioHQ #373) introduced an adaptive delay formula and an Enter retry loop with output verification.

Beads: bd-orch2v3 (P0), bd-qhf (P1)

### Goals

- Fix Enter timing for long messages so agents reliably start triage without manual intervention
- Add retry logic for Enter delivery reliability
- Keep the fix backwards compatible (short messages unchanged at 300ms)

### Tenets

- Minimal change to the paste-buffer delivery path
- Backwards compatible: short messages <=200 chars keep the existing 300ms flat delay
- Adaptive: delay scales with message size up to a 2s cap

### High-level description of changes

Adaptive delay formula (replacing flat 300ms/1000ms):
- For long messages (>200 chars or multiline): base 1000ms + 200ms per KB, capped at 2000ms
  - 1KB = 1200ms, 5KB = 2000ms (capped)
- Short messages: 300ms (unchanged)

Enter retry loop (new, long messages only):
- After sending Enter, capture pane output (20 lines)
- Check if agent started responding: any activity token (spinner chars, "Thinking...", "Running tool") OR pane no longer ends with the pasted message tail
- If not yet started, retry Enter up to 3 times with 1s between attempts
