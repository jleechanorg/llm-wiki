## Background

13/19 open PRs were stuck at `mergeable=UNKNOWN`. Workers went idle because the `merge-conflicts` reaction only fired when GitHub explicitly reported `CONFLICTING` â€” not when it reported `UNKNOWN` (merge status still being computed). The existing `merge-conflicts` reaction (which sends a rebase message to the worker) was never triggered.

## Goals

- Workers receive a rebase prompt when their PR's merge status is unknown (stale branch), not just when it has explicit conflicts.

## Tenets

- Prefer reusing existing reaction infrastructure over adding new event types.
- UNKNOWN is semantically "not confirmed mergeable" â€” safe to treat as `noConflicts=false` for lifecycle purposes.

## High-level description

In `scm-github.getMergeability()`, when GitHub reports `mergeable=UNKNOWN` (string) or empty/null, the code was leaving `noConflicts=true` (from the earlier assignment `noConflicts = mergeable === "MERGEABLE"`). This caused `determineStatus()` to return `"pr_open"` instead of `"merge_conflicts"`, so the `merge-conflicts` reaction never fired.

Fix: add `noConflicts = false` in the `UNKNOWN`/`""` branch so the existing `merge_conflicts â†’ merge.conflicts â†’ send-to-agent` reaction pipeline fires and the worker rebases its stale branch.
