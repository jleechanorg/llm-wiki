## Summary

Fixes the skeptic-gate CI wait loop to be robust against GitHub API eventual-consistency delays.

## Changes

**`.github/workflows/skeptic-gate.yml`**:
- `concurrency: cancel-in-progress: false` — prevents subsequent CI runs from cancelling in-progress skeptic-gate runs
- Restores wait loop: polls check-runs API every 30s for up to 5 min until CI jobs complete
- `--paginate` flag on all gh api check-runs calls (CHECK_RUNS_FAILED, CHECK_RUNS_PENDING, TOTAL_CHECKS) for reliability
- `MIN_CORE_JOBS=2`: require only Lint+Typecheck as minimum valid CI signal (not all 4 jobs)
- 10-second settle delay before first poll to avoid GitHub API eventual-consistency gap
- Updated Gate 1 comment to accurately describe MIN_CORE_JOBS logic

## Root cause

The gh api check-runs endpoint returns incomplete results for ~5-10s after CI completes (GitHub propagates check-run entries with a small delay). The previous Gate1 logic required all 4 CI jobs to be visible before evaluating, but the API would sometimes return only 2-3 jobs during this window, causing false FAIL with "timeout waiting for CI" even though CI had passed.
