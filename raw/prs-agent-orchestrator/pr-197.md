## Background
When `backfillUncoveredPRs()` fails to claim a PR (e.g. because the branch is already checked out elsewhere), the worktree created during `spawn()` was left behind with the PR branch locked. Every subsequent backfill retry created a NEW worktree that also failed because git refuses to fetch into a branch checked out in another worktree. This cascaded until dozens of ghost worktrees accumulated, permanently blocking the lifecycle-worker from claiming affected PRs.

Real-world impact: 55 dead worktrees accumulated in `~/.worktrees/jleechanclaw/`, blocking lifecycle-worker from claiming PRs #393 and #383 for over 13 hours until manual cleanup.

## Goals
1. Delete the local branch after destroying a worktree to prevent it from remaining checked-out and locking future worktrees
2. When `sessionManager.kill()` fails during `claim_failed` (session not yet registered), perform direct worktree cleanup and continue to the next PR instead of aborting entirely

## Tenets
- Keep cleanup best-effort (always wrapped in try/catch) — never let cleanup failure crash the lifecycle-worker
- Use the existing `git()` helper from the workspace plugin for workspace-level cleanup
- Accept `execSync` for backfill-level cleanup since the `git()` helper may not be in scope there

## Changes

### Fix 1: `packages/plugins/workspace-worktree/src/index.ts` — `destroy()`
Before removing the worktree, capture the checked-out branch. After successful worktree removal, delete the local branch if it's an AO-managed branch (not main/master/develop). The NOTE comment is updated to explain the rationale.

### Fix 2: `packages/core/src/backfill-extensions.ts` — `claim_failed` handler
When `sessionManager.kill()` fails (orphan_cleanup_failed), add a direct worktree cleanup fallback that computes the worktree path from `session.id` and `projectId`, then uses `execSync` to unlock + remove the worktree and delete the stale branch.
