## Summary

Add a `taskQueue` config section to `agent-orchestrator.yaml` that lets users list beads to process with a concurrency limit. The lifecycle-worker drains the queue automatically: when an active session completes/merges, it spawns the next queued bead.

## Goals

- Users can declare an ordered list of bead IDs and a `maxConcurrent` limit per project
- Lifecycle-worker drains the queue autonomously, up to the concurrency limit
- Dispatched beads are tracked via `queuedBeadId` in session metadata (no YAML mutation)
- Bead title/description resolved via `br show <beadId>` with safe fallbacks
- Custom `taskTemplate` supported per project

## Tenets

- No YAML mutation at runtime — track in-flight beads via session metadata
- One spawn per poll cycle (30s throttle) to avoid thundering-herd
- Separate from `backfillAllPRs` — both can run simultaneously
- On spawn failure: log, record, and skip (no retry)

## Config example

```yaml
projects:
  worldai-claw:
    taskQueue:
      enabled: true
      maxConcurrent: 6
      beads:
        - wc-5cj
        - wc-gjc
        - wc-od5
      taskTemplate: |
        Fix bead {beadId}: {beadTitle}
        After completing work and creating the PR, run /learn.
```
