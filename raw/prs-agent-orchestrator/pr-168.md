## Background

`ps aux` shows 2x `lifecycle-worker` processes for `jleechanclaw` and 2x for `agent-orchestrator`. Duplicate workers send contradictory nudges to the same session causing agents to confirm green and stop instead of re-engaging.

Root cause: Two concurrent `lifecycle-worker` processes (e.g. launchd `KeepAlive` restart racing with `ao start`) can both pass the PID-file status check before either records its PID -- a classic TOCTOU race. Additionally, `ensureLifecycleWorker` had no way to detect a launchd-spawned worker that had not written its PID file yet.

Beads issue: orch-886k

## Goals

1. Guarantee at most one `lifecycle-worker` process runs per project at any given time.
2. Close the TOCTOU window between the "not running" check and PID registration in `lifecycle-worker.ts`.
3. Detect launchd-spawned workers in `ensureLifecycleWorker` even when no PID file exists yet.
4. Zero regression to existing tests; new unit tests prove the guard logic.

## Tenets

- Prefer atomic OS primitives (`O_EXCL`) over advisory checks -- advisory checks can race.
- `ps aux` scan is a safety net, not the primary mechanism. PID files remain canonical.
- Guards must be fail-safe: if detection is indeterminate (ps failed, lock stale), do not spawn.
- No new dependencies -- stdlib only.

## High-level description of changes

Two complementary guards eliminate the duplicate-worker race:

**1. `tryAcquireLifecycleLock` (new, in `lifecycle-service.ts`)**

Uses `O_EXCL` to atomically create a per-project lock file (`lifecycle-worker.lock`). Only one process can succeed; the second gets `EEXIST` and returns `null`. In `lifecycle-worker.ts`, the lock is acquired before the PID-file status check and released immediately after writing the PID -- closing the window where two processes could both see "not running" and both proceed.
