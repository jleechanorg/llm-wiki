## Background

Overnight monitoring loops log idle forever because tmux has-session returns ALIVE even when the agent CLI has exited -- the bash shell stays alive in the tmux pane. Workers stuck waiting for user feedback or that have exited without creating PRs are never detected or cleaned up.

## Goals

1. Detect when AO worker agents have exited but their tmux sessions persist (false liveness)
2. Detect when agents are stuck waiting for user feedback/permissions
3. Kill exited sessions and nudge stuck ones automatically
4. Integrate the probe into the lifecycle-manager polling loop after 3+ consecutive idle cycles

## Tenets

- Conservative detection: only act after 3+ idle cycles to avoid false positives
- Pattern priority: exiting > waiting > shell-prompt > activity indicators > no action
- Fork isolation: new files for detection logic, minimal diff to lifecycle-manager.ts
- Injectable dependencies for full testability

## High-level description of changes

**Commit 1: feat(core): add stuck worker detector**
- New module stuck-worker-detector.ts with pattern-based pane analysis
- Shell prompt patterns (bash $, zsh %, starship, fish >, root #)
- Agent activity indicators (Claude spinners, Braille spinners)
- Waiting patterns (permission prompts, Y/N, approval required)
- Exiting patterns (Exiting, goodbye, session ended)
- Idle cycle tracking with per-session counters
- High-level checkStuckWorker() API with injectable tmux callbacks
- Full export from packages/core/src/index.ts

**Commit 2: feat(lifecycle): integrate stuck worker probe into polling loop**
- Import stuck-worker-detector functions and tmux wrappers
- In the no-transition else block (session in steady state), call checkStuckWorker for non-terminal, non-orchestrator sessions with runtime handles
- Reset idle cycle counter on state transitions
- Kill action sets killConfirmed: "stuck-probe" metadata
- All probe outcomes logged via observability framework
