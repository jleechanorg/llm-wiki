## Raw Body
## Background

The `generate-pr-design-docs.yml` workflow fires on `pull_request.synchronize`, generates design docs, commits them as `github-actions[bot]`, which triggers another `synchronize` event → infinite commit loop. This is P0 blocking PR #259.

The existing `github.actor != 'github-actions[bot]'` check in the job `if:` condition doesn't catch this because `github.actor` reflects the user who triggered the workflow (the PR author who pushed), not the commit author.

## Fix

Add a guard step (`Skip if last commit is from github-actions[bot]`) that calls `gh api repos/ComposioHQ/agent-orchestrator/pulls/{PR}/commits --jq '.[-1].author.login'` before any expensive operations. If the last commit author is `github-actions[bot]`, `exit 1` immediately — breaking the regenerate loop.

The `pull_request.synchronize` event fires after the push completes, so `github.event.pull_request.head.sha` at workflow start is the new bot commit SHA. Querying that commit's author correctly detects our own preceding run.

## Testing

- [ ] Triggered workflow on a PR where the last commit is from a human → proceeds normally

Closes: #259



## Evidence

**Claim class**: unit

This PR modifies a single GitHub Actions YAML workflow file. The fix adds a guard step that queries the last commit author via `gh api` and exits early when the author is `github-actions[bot]`. The logic is:

- If `LAST_AUTHOR = "github-actions[bot]"` → `should_skip=true` output → all subsequent steps are gated with `if: steps.skip-check.outputs.should_skip != 'true'`
- If `LAST_AUTHOR != "github-actions[bot]"` → `should_skip=false` output → proceeds to checkout and generate

The guard step uses `should_skip` step output (not `exit 1`) so the step exits 0 — avoiding red CI status on the PR when the loop is correctly broken.
