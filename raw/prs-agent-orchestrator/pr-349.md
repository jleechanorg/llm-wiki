## Background

Sessions ao-2303, ao-2312, and ao-2322 were stuck with `status=merged` in the AO session DB but their tmux processes were still alive and consuming session slots. Sessions ao-1957, ao-1966, ao-2092, ao-2223 were stuck with no active work and high `scmFailureCount` values (900-1500).

## Root Cause

**Two independent bugs:**

### Bug 1: Zombie detection relies on stale `session.pr?.state` (bd-ara.2)

`session-reaper-extensions.ts` `checkAndKillZombie()` used `session.pr?.state` as the zombie signal. But `lifecycle-manager.ts` `persistPrState()` only writes when the state *changes*:

```typescript
if (batch.state !== prevPrState) {
  persistPrState({ session, state: batch.state, ... });
}
```

If the last successful poll already cached `prState=open` and then the PR merged, `persistPrState` would never update — the merged state is never persisted. Sessions ao-2303/ao-2312/ao-2322 had `status=merged` (set correctly by lifecycle-manager) but `prState=open` (stale). The reaper's zombie check saw `prState=open` → returned `{ action: "none" }`.

Additionally, `checkAndKillZombie()` ran AFTER the terminal-status skip in `session-reaper.ts`. Sessions with `status=merged` were skipped as "terminal" before the zombie check could run.

### Bug 2: `scmFailureCount` overflow (bd-ara.2)

`lifecycle-manager.ts` tracks consecutive SCM failures to prevent premature worktree destruction. The counter persisted in session metadata accumulated without bound — ao-1957 had `scmFailureCount=1524`, ao-1966 had `1511`, ao-2092 had `1341`. While this didn't directly prevent reaping (the no-PR path doesn't depend on this counter), it indicated the counter was accumulating from a previous version that didn't reset properly.
