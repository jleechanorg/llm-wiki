## Summary
- **Bug**: When Codex CLI fails with ENOBUFS/spawnSync errors, skeptic returns VERDICT: FAIL which permanently blocks the PR gate. Observed in worldarchitect.ai PR #6161.
- **Fix**: Add a 3-model fallback chain (codex, claude, gemini). Only infrastructure failures trigger fallback; legitimate verdicts (even FAIL) are accepted immediately.
- **Key change**: When ALL models fail with infra errors, returns VERDICT: SKIPPED (not FAIL) so the gate does not permanently block PRs.

## Production Code Changes

**Before**: `runSkepticReview()` calls `ao skeptic verify --model codex` once. Any CLI crash returns `verdict: "FAIL"`.

**Now**: `runSkepticReview()` tries each model in sequence via `tryModel()` helper:
1. codex (primary) - if infra failure then
2. claude - if infra failure then
3. gemini - if infra failure then
4. Return `verdict: "SKIPPED"` with all error details

**Why**: ENOBUFS errors on macOS are transient spawn buffer overflows. Retrying with a different CLI binary (claude vs codex) often succeeds because they use different spawn strategies.

## Test Changes
- 6 new tests in `skeptic-reviewer.test.ts` covering fallback chain scenarios
- 1 existing test updated: ENOENT now returns SKIPPED (was FAIL) after exhausting fallback chain
- All 45 skeptic-related tests pass

## Known Limitations
- Fallback chain is hardcoded (codex, claude, gemini). Cross-model preference not yet implemented at this layer.
- `tryModel()` fetches triggerSha independently for each attempt (minor redundancy, non-blocking).
