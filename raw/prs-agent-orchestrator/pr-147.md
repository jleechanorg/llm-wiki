## Background

When a tmux session is killed manually, the AO session metadata persists in the filesystem. The lifecycle-worker keeps polling it, detecting PR state changes (e.g. CHANGES_REQUESTED), and firing reactions (`ao send`) to the dead session + notifying OpenClaw.

## Goals

- Prevent reactions from firing to dead tmux sessions
- Ensure lifecycle-worker correctly short-circuits to "killed" status without evaluating stale PR state

## Tenets

- bd-kki check in `checkSession` already handles the "killed → check if PR merged" case — no need to duplicate that logic in `determineStatus`
- Reactions must never fire for dead sessions regardless of their cached PR state

## High-level description of changes

In `determineStatus()`, when `runtime.isAlive()` returns `false`, return `"killed"` immediately instead of falling through to step 4 (PR state evaluation). Previously, the code deferred `"killed"` to after PR state checks so that auto-merge could fire for green PRs with exited agents — but this caused PR state reactions to fire for dead sessions before the kill transition was recorded.

The bd-kki check in `checkSession()` detects the `"killed"` return and separately verifies whether the PR was merged before finalizing the status, preserving the auto-merge capability.
