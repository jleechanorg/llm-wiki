## Background

## Goals

- Prevent git worktree add from failing with an ambiguous-ref error when origin/<branch> is shadowed by a local branch of the same name
- Auto-remediate when safe; throw actionable error when remediation fails

## Tenets

- Prefer auto-remediation over explicit user action
- Non-destructive: rename the conflicting local branch to backup/<name>, do not delete
- Fail gracefully with actionable error messages

## High-level description

Added a pre-flight check in workspace.create() (workspace-worktree plugin) that runs before git worktree add:

1. Run `git branch --list origin/<defaultBranch>` to detect a local branch with the same name as the remote-tracking ref
2. If no local branch found → proceed normally (unambiguous)
3. If local branch found → auto-rename it to `backup/<name>` before git worktree add
4. If rename fails → throw actionable error with manual fix instructions

**Why `git branch --list` instead of `git for-each-ref`?** `git for-each-ref "origin/main"` does NOT match `refs/remotes/origin/main` (short names unsupported as patterns). `git branch --list` reliably detects local branches matching the remote-tracking ref name.
