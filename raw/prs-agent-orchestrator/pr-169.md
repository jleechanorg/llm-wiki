## Background

OpenCode emits specific UI patterns when waiting for user input (press Enter, `[y/n]` prompts, menus, question marks). The `detectActivity()` method in `agent-opencode` was returning `"active"` for all non-empty output, losing this signal.

## Goals

- `detectActivity()` returns `"ready"` for OpenCode waiting-for-input patterns
- Gives the lifecycle manager accurate real-time activity signals

## Tenets

- Minimal, safe change — no breaking API changes
- All new patterns are additive (existing "active" behavior preserved for unrecognized output)

## High-level description

Added pattern matching in `detectActivity()` for OpenCode waiting indicators: `press ... to continue`, `[y/n]`, `[Y/n]`, `waiting`, `confirm`, `menu`, `?` prompts, etc. Returns `"ready"` when matched, falls back to `"active"` otherwise.

> [!NOTE]
> **Medium Risk**
> Changes `detectActivity()` behavior to return `ready`/`waiting_input` for specific terminal prompt patterns, which could affect session lifecycle decisions if patterns match unexpectedly. Scope is limited and covered by new unit tests, with unrecognized output still treated as `active`.
> 
> **Overview**
> Improves `agent-opencode` terminal activity classification by adding hoisted regex pattern sets for OpenCode “waiting” UI output. `detectActivity()` now returns **`waiting_input`** for interactive prompts (e.g., `[y/n]`, `confirm?`, trailing `?`, menu arrows) and **`ready`** for neutral wait states (e.g., “press … to continue”, “waiting/ready”), falling back to `active` otherwise.
> 
> Adds targeted tests to validate the new prompt
