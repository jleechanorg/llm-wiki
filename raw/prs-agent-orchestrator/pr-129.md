## Background

The lifecycle-worker reaction engine was firing but **completely invisible** — zero observability logging existed for reaction execution. More critically, `backfillAllPRs: true` was set in config but had **no implementation** in the codebase. When worker sessions died, nothing respawned them. This was the root cause of non-autonomy: the system could monitor sessions but not create them.

Tracked by: bd-y5v (green loop epic), bd-ara (autonomy blockers)

## Goals

1. Make `backfillAllPRs` actually work — lifecycle-worker spawns sessions for open PRs without active workers
2. Add reaction observability so we can diagnose reaction engine behavior from logs
3. Close the autonomy gap where dead sessions leave PRs orphaned

## Tenets

- Config-first: the feature is already configured, just needs implementation
- One PR per cycle: avoid thundering herd spawning
- REST-only: avoid GraphQL exhaustion for PR listing
- Observable: every reaction execution must be logged

## High-level description of changes

Three components:
1. **Type additions**: `backfillAllPRs` on ProjectConfig, `listOpenPRs()` on SCM interface
2. **GitHub SCM plugin**: implements `listOpenPRs()` using REST API
3. **Lifecycle-manager**: backfill loop (throttled to 5-min intervals) + reaction observability logging

The backfill loop runs inside the existing poll cycle but is independently throttled. It lists open PRs via SCM, cross-references against active sessions, and spawns one session per cycle for uncovered non-draft PRs.
