## Raw Body
## Background
Reaped co-worker sessions (post-merge sweep) were not emitting `session.exited` notifications. When `reapPostMergeCoWorkers()` killed idle co-workers after a PR merge, the Slack thread terminal updates were silently skipped — only the primary merged session received the notification.

This caused stale-worker lifecycle noise: Slack would show merged workers as "still running" even after cleanup.

## Goals
- Reaped co-worker sessions emit `session.exited` notifications deterministically.
- Terminal notification emission is idempotent — safe to call for already-dead sessions.

## Tenets
- Exit proof failures are non-fatal (warning-logged) to avoid corrupting the reap result.
- Backward compatible: `exitProofDeps` is optional; existing callers without deps are unaffected.

## High-level description of changes

**`packages/core/src/fork-lifecycle-postmerge.ts`**:
- Added `PostMergeExitProofDeps` interface (config, registry, notifyHuman, createEvent, observer).
- `reapPostMergeCoWorkers()` now accepts optional `exitProofDeps`.
- After reaping each co-worker session, calls `validateAndEmitExitProof()` to emit the exit proof + `notifyHuman` notification.
- Exit proof errors are caught and logged as `lifecycle.post_merge_exit_proof` failures — non-fatal.

**`packages/core/src/lifecycle-manager.ts`**:
- Passes `exitProofDeps` when calling `reapPostMergeCoWorkers()` on merge transition.

**`packages/core/src/__tests__/fork-lifecycle-postmerge.test.ts`**:
- 5 new tests covering: single-session exit proof emission, multi-session emission, backward-compat skip when deps not provided, skip for merged session itself, and `lifecycle.exit_proof` operation recording.
