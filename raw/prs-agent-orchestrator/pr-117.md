## Background
Phase 2 of bd-8y9 API call reduction initiative. The fork uses ~8 gh CLI invocations per session per lifecycle poll cycle. This PR reduces that by combining two separate API calls into one and throttling review backlog polling.

## Goals
- Reduce per-session gh CLI invocations by ~3 calls per cycle on average
- Maintain identical status determination behavior
- Zero breaking changes to existing plugin implementations (fallback path preserved)

## Tenets
- Backward compatible: SCM plugins without getPRStateAndReview still work via fallback
- Throttle is transparent: always runs on transitions, only skips steady-state polls
- Minimal core diff: new logic in types.ts and review-backlog.ts, small change in lifecycle-manager.ts

## High-level description of changes

**bd-sm7: Combined PR state + review decision query**
- Added optional `getPRStateAndReview()` method to SCM interface in types.ts
- Implemented it in scm-github plugin — single `gh pr view --json state,reviewDecision`
- Lifecycle manager uses combined call when available, falls back to separate calls
- Saves 1 gh CLI invocation (~2 GraphQL points) per session per cycle

**bd-yjo: Throttled review backlog polling**
- Review backlog API checks (getPendingComments + getAutomatedComments) now run every 3rd poll cycle
- Always runs on status transitions (oldStatus !== newStatus)
- Saves 2+ gh CLI calls per session on 2 out of every 3 steady-state cycles

- Updated plugin-integration test mock sequence for combined call
- Added review backlog counter reset to test beforeEach to prevent cross-test throttle interference
