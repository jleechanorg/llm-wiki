## Background
Phase 3 of bd-8y9 API call reduction initiative. Per-session API calls in `determineStatus()` currently require 4-6 separate `gh` CLI invocations (getPRState, getCISummary, getReviewDecision, getMergeability). Each `gh` call costs ~2 GraphQL points.

## Goals
- Replace all per-session API calls with a single batch GraphQL query
- Save ~5 gh CLI invocations per session per cycle (best case: approved+CI passing path)
- Maintain backward compatibility with SCM plugins that don't support batch

## Tenets
- Backward compatible: fallback to individual calls when getBatchPRStatus is not implemented
- Same status determination logic: identical branching as the individual-call path
- Plugin-first: implementation lives in scm-github plugin, core only dispatches

## High-level description of changes

**bd-att: Batch PR status query**

Added `getBatchPRStatus()` method that fetches all PR status fields in a single `gh pr view --json state,reviewDecision,statusCheckRollup,mergeable,mergeStateStatus,isDraft` call. This replaces the serial chain of:
1. `getPRState()` — 1 gh call
2. `getCISummary()` → `getCIChecks()` — 1 gh call
3. `getReviewDecision()` — 1 gh call
4. `getMergeability()` — 3 gh calls (getPRState + pr view + getCISummary)

**Net savings per session per cycle:**
- Best case (approved+CI passing): ~6 gh calls → 1 = save 5 calls (~10 GraphQL points)
- Worst case (PR merged/closed, early return): ~1 gh call → 1 = save 0 calls
- Typical (CI failing or changes_requested): ~3 gh calls → 1 = save 2 calls
