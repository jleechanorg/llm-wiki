## Background

`ao review-check` silently returns `{pendingComments: 0, reviewDecision: null}` on **any** GitHub GraphQL error, including transient rate-limit errors. When GitHub rate-limits the lifecycle manager's session-polling loop, it has no retry or deferral â€” mutations fail into an errors array and work is silently dropped.

## Goals

- Make every GraphQL operation **non-blocking**: rate-limit errors trigger exponential backoff retry (1s â†’ 2s â†’ 4s, cap 30s) before deferring
- Deferred operations are tracked with attempt count + timestamp and retried on the next poll cycle if sufficient backoff has elapsed
- Stale deferrals expire at 10 minutes to prevent indefinite retry loops
- Non-rate-limit errors (auth failures, malformed queries) fail **immediately** â€” no retry

## Tenets

- Prefer existing patterns over novel abstractions
- Follow the same `isGhRateLimitError` / `ghSleep` shared utilities already in `gh-rate-limit.ts`
- `review-check` should print a clear warning listing all deferred PRs so operators know what was skipped

## High-level description of changes

### `packages/core/src/gh-graphql-defer.ts` (new)
`DeferredGraphQLExecutor` wraps any `GraphQLExecutor` with exponential-backoff retry on rate-limit errors and deferred-state tracking. Implements the `GraphQLExecutor` interface via a passthrough `execute(query, vars)` method, and adds `executeWithLabel(label, query, vars)` which returns `ResilientResult<T>`:

```typescript
interface ResilientResult<T> {
  data: T | null;          // null when wasDeferred
  deferred: DeferredItem[]; // items that couldn't complete
  wasDeferred: boolean;     // true if data is null due to deferral
}
```

Deferred items carry `label`, `deferredAt`, `attempts`, `lastError`. A second call with the same label skips the retry if insufficient backoff has elapsed since the last attempt.
