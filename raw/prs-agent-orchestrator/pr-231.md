Closes #232

This change hardens the core lifecycle polling manager around the edges that were still weak after the earlier iterations. The lifecycle loop is the component responsible for normalizing session state, running reactions, and driving operator-visible status transitions, so ambiguous runtime failures here are costly: they make debugging configuration problems harder and can cause otherwise-fixable operator input mistakes to look like missing sessions or silent no-ops.

The root cause was that the public lifecycle-manager entry points trusted a few runtime assumptions that TypeScript alone cannot enforce. `createLifecycleManager()` only checked for truthy values and produced generic errors, `check(sessionId)` did not normalize externally supplied IDs before lookup, and reaction config validation treated any string as an action even though the supported action set is fixed. That left a gap between compile-time types and runtime behavior.

The fix adds explicit runtime guards for supported reaction actions, introduces more descriptive dependency and session-ID validation errors, trims session IDs before lookup, and validates `start()` arguments before ignoring duplicate starts. The exported `createLifecycleManager()` JSDoc was also expanded so the module's external contract is clearer. The test suite now covers the new constructor validation, duplicate-start validation, trimmed session IDs, non-string IDs, and the fallback path for malformed reaction actions.
