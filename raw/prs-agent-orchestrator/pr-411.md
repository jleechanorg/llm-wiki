## Raw Body
### Background
Codex worker sessions can currently inherit a generic per-project `agentConfig.model` before CLI-keyed defaults are considered. In practice this allowed `worldarchitect` Codex runs to launch with `--model claude-sonnet-4-6`, which Codex CLI rejects. This PR fixes the precedence bug in selection logic so CLI-specific model defaults are authoritative for the selected agent.

### Goals
1. Prevent invalid Codex model launches caused by generic shared `agentConfig.model` values.
2. Keep model selection deterministic and explicit: CLI-keyed model settings should win over generic shared defaults.
3. Add regression coverage that would have failed with the prior precedence order.

### Tenets
- Prefer explicit CLI-specific configuration (`modelByCli`) over generic cross-agent settings.
- Keep the core diff minimal and isolated.
- Add tests that exercise the exact failure class (Codex + generic Claude model).

### High-level description of changes
`resolveAgentSelection` now resolves models with `modelByCli` precedence ahead of generic shared/role `agentConfig` model fields for both worker and orchestrator roles. This makes per-CLI defaults the primary source, with generic config as fallback. Added regression tests that verify Codex selects `modelByCli.codex` models even when shared config specifies incompatible Claude model strings.
