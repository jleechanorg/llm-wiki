## Background

1. **Wrong hook event names**: AO wrote `PostToolUse`/`PreToolUse` to `.gemini/settings.json`, but Gemini CLI only recognizes `AfterTool`/`BeforeTool` (confirmed via `gemini hooks migrate` source). These keys are silently ignored, so `metadata-updater.sh` never runs.

## Background

Gemini sessions were spawning idle and never tracking their work in AO. Two root causes identified:

1. **Wrong hook event names**: AO wrote `PostToolUse`/`PreToolUse` to `.gemini/settings.json`, but Gemini CLI only recognizes `AfterTool`/`BeforeTool` (confirmed via `gemini hooks migrate` source). These keys are silently ignored, so `metadata-updater.sh` never runs.

2. **Wrong timing**: `postLaunchSetup` wrote the hooks AFTER Gemini launched. Gemini reads `settings.json` only at startup (confirmed via `gemini.js` source). Even if the event names were correct, the hooks would never be loaded.

Result: AO could never track when Gemini ran `gh pr create` or `git checkout -b`, so sessions stayed "idle" in AO tracking and were treated as stuck.

## Goals

- Fix hook event names so Gemini CLI's `AfterTool`/`BeforeTool` hooks fire correctly
- Fix timing so `settings.json` exists before Gemini reads it at startup
- Keep Claude Code's `PostToolUse`/`PreToolUse` as default (no behavior change for other agents)

## Tenets

- Minimal surface change: add config field, keep defaults unchanged
- TDD: failing tests first, then implement
- Plugin-first: change in plugin config, not core

## High-level description of changes

Added `hookEventNames` config to `AgentPluginConfig` with defaults `PostToolUse`/`PreToolUse`. `setupHookInWorkspace` accepts these as params and uses them when writing `settings.json`. Gemini plugin sets `hookEventNames: { postToolUse: "AfterTool", preToolUse: "BeforeTool" }`.

For timing: `session-manager.ts` `spawn()` now calls `setupWorkspaceHooks()` BEFORE `plugins.runtime.create()` — hooks are written before the agent starts.
