## Raw Body
### Background
AO session restore and PR claim were reporting healthy session state while leaving the runtime idle. In practice this showed up as restored Codex workers sitting at a generic prompt after recovery, and newly claimed PR sessions only starting work when callers explicitly opted into `sendInitialMessage`. That made restore/claim look successful in metadata while tmux showed no PR-specific kickoff.

### Goals
- Ensure PR claims send a kickoff task by default.
- Ensure restoring a PR-tracked session re-delivers the PR kickoff so the agent resumes useful work.
- Lock the behavior down with focused session-manager tests.

### Tenets
- Fix the harness, not just the symptom.
- Keep failure handling fail-soft: a restore should remain usable even if kickoff redelivery fails.
- Preserve explicit opt-out behavior for callers that intentionally suppress the initial message.

### High-level description of changes
This change makes `claimPR()` default to sending the initial PR task message unless a caller explicitly passes `sendInitialMessage: false`. It also teaches `restore()` to detect PR-tracked sessions, resolve the PR through the SCM plugin, and send the same kickoff task message back into the restored runtime after launch. The tests were updated to reflect the new default-send behavior, cover restore-time redelivery, and keep ownership-switch tests explicit about suppressing kickoff when message delivery is irrelevant to the scenario.
