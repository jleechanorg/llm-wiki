## Background

The `ai.agento.orchestrators` launchd daemon (KeepAlive=true) was silently failing to start `ao-orchestrator` and `wa-orchestrator` on every retry. The error in `/tmp/ao-orchestrators.err.log`:

```
✖ Orchestrator setup failed
Error: Failed to setup orchestrator: Session ao-orchestrator already exists but is not in a reusable state
```

Root cause: when lifecycle-manager marks an orchestrator session as `killed`, it writes a metadata file (`status=killed\nrole=orchestrator\n`) with **no `runtimeHandle`**. On the next launchd restart, `spawnOrchestrator` skips cleanup because its cleanup branch is guarded on `existingRaw["runtimeHandle"]` being truthy. Then `reserveSessionId` (O_EXCL) fails because the file already exists — throwing the error indefinitely, every 60 seconds.

## Goals

- `ao start` self-heals on launchd restart after any crash/kill without human intervention
- Terminal ghost files (`status=killed`, `status=merged`, no `runtimeHandle`) are cleared automatically
- Regression tests prevent this class of failure from recurring

## Tenets

- Minimal blast radius: only touch the ghost-file case (no `runtimeHandle` + terminal status)
- No behavior change for normal paths (alive session reuse, dead-runtime-with-handle cleanup)
- Test the exact real-world scenario (write a plain text file, not a full metadata object)

## High-level description of changes

Added a pre-cleanup step in `spawnOrchestrator` (before `reserveSessionId`): if the existing session file has a terminal status (`killed` or `merged`) and no `runtimeHandle`, delete it unconditionally. These files have no live runtime and are always safe to clear.

Added two regression tests that write the exact file format lifecycle-manager produces and verify `spawnOrchestrator` succeeds without throwing.
