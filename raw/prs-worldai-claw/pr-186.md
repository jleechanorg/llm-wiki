## Background
GitHub issue #80 reported that the faction simulation cooldown was being consumed even when `processInPlayTriggers` encountered an error, because the cooldown timestamp was being set before the simulation logic was awaited.

## Goals
Ensure the faction simulation internal rate-limiting cooldown is only consumed upon successful triggering of `processInPlayTriggers`.

## Tenets
- Fail-safely without penalizing users' cooldown times for internal faults.
- Test-Driven Development (TDD) first.

## High-level description
Shifted the timing of `lastFactionSimRealTime.set(...)` in `app.ts` so it is executed only after `processInPlayTriggers(...)` returns successfully.

## Testing
- Created `app_faction_cooldown.test.ts` to mock `processInPlayTriggers` intentionally throwing an error.
- Verified that a subsequent turn properly re-triggers the simulation, confirming that the cooldown was not incorrectly logged as consumed.

## Low-level details
- `packages/backend/src/app.ts`: Moved `lastFactionSimRealTime.set` after `await processInPlayTriggers` inside `processLivingWorldIfNeeded`.
- `packages/backend/tests/app_faction_cooldown.test.ts`: Added Jest test mimicking a session taking turns and throwing on simulation.
