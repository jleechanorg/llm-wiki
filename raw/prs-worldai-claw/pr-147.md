## Background

apiClient.ts had a `VITE_WORLDCLAW_TOKEN` env var check that ran before localStorage lookup. This allowed the build-time env var to bypass the settings UI — users who changed their token in Settings would find their API calls still using the stale env var token.

Additionally, `AppContext` stored `baseUrl`/`authToken` in React state but never hydrated from localStorage on initialization, creating a parallel source of truth disconnected from what the Settings screen saves.

## Goals

1. Make `worldclaw_auth_token` in localStorage the **sole canonical source** for the auth token used by apiFetch
2. Ensure AppContext and apiFetch both read from the same localStorage keys
3. Remove env var bypass that prevented settings UI from controlling runtime auth

## Tenets

- Settings screen saves → the only thing that matters at runtime
- No env var fallbacks that short-circuit user-configured settings
- TDD: tests written before implementation, verifying the correct behavior

## High-level description of changes

**apiClient.ts**: Removed `VITE_WORLDCLAW_TOKEN` env var lookup. `getStoredAuthToken()` now reads ONLY from `worldclaw_auth_token` in localStorage — the same key that Settings writes to.

**AppContext.tsx**: `getInitialState()` now reads `openclaw_url` → `baseUrl` and `worldclaw_auth_token` → `authToken` from localStorage on init. Previously `authToken` was always empty and `baseUrl` always defaulted to localhost.

**Tests added**:
- `apiClient.test.ts`: Verifies env vars are ignored, localStorage is canonical
- `appContext.test.tsx`: Verifies localStorage hydration of baseUrl and authToken
- `settings.test.tsx`: Verifies Settings writes both `openclaw_token` and `worldclaw_auth_token`
