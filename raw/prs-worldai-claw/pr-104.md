## Background
`fork()` in `packages/backend/src/storage/entity_chain.ts` previously used `append()` which calls `Date.now()` for each block, destroying original timestamps. The `appendWithTimestamp()` private method was added to preserve original block timestamps when forking entity chains. However, no regression tests existed to guard this behavior.

## Goals
- Add TDD regression tests that prove `fork()` preserves original block timestamps
- Ensure forked chains maintain hash-chain integrity with preserved timestamps
- Guard against future regressions where someone might revert to `append()`

## Tenets
- Test-driven: tests written first, verified to fail without the fix (RED), then pass with it (GREEN)
- Real SQLite: all tests use in-memory `better-sqlite3`, no mocks

## High-level description of changes
Added 7 regression tests covering:
1. **Timestamp equality** — forked block timestamps match originals
2. **No Date.now() batching** — timestamps with known gaps are preserved (not collapsed)
3. **Hash-chain integrity** — forked chain passes `verify()` with preserved timestamps
4. **SQLite persistence** — timestamps are correctly written to DB, not just in-memory
5. **Content fidelity** — forked block content matches original exactly
6. **Entity ID correctness** — forked blocks use new entity ID
7. **Post-fork behavior** — new appends after fork use `Date.now()`, not old timestamps

## Testing
- RED phase: temporarily reverted `fork()` to use `append()` → 4/7 tests failed (timestamp mismatches)
- GREEN phase: restored `appendWithTimestamp()` → 7/7 tests pass
- Existing `entity_chain.test.ts` tests (10/10) still pass
