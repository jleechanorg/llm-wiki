## Background
ChainAnchor exists as a standalone module (`network/chain_anchor.ts`) with MCP tool handlers already wired in `mcp/server.ts`, but `app.ts` had no explicit integration -- no import, no lazy instantiation, and no startup status logging.

## Goals
- Lazy-instantiate ChainAnchor in `buildApp()` when env vars (`BASE_RPC_URL`, `ANCHOR_CONTRACT_ADDRESS`, `ANCHOR_PRIVATE_KEY`) are present
- Log configuration status on startup for operational visibility
- Verify MCP tools `dm_anchor_entity` and `dm_verify_anchor` return graceful `{ status: 'not_configured' }` when env vars are absent
- TDD: write tests first, watch them fail, implement minimal code

## Tenets
- Prefer real tests over mocks -- MCP tool tests use `app.inject()` with auth headers against real Fastify instance
- Minimal change -- only add what is needed for the integration
- Follow existing patterns (P2P sync lazy init pattern at line 2737)

## High-level description of changes
Added `getChainAnchor` import to `app.ts` and a lazy instantiation block in `buildApp()` that:
1. Calls `getChainAnchor()` to trigger singleton creation
2. Logs `[ChainAnchor] configured` or `[ChainAnchor] not configured` based on `isHealthy()`

The MCP tools were already fully implemented in `mcp/server.ts` -- this PR adds the app-level integration and comprehensive TDD test coverage.

## Testing
- 10 TDD tests in `chain_anchor_app_integration.test.ts` -- all pass
- 34 total tests across all 3 chain_anchor test suites pass
- Tests cover: lazy init with/without env vars, startup logging, MCP tool routing for both configured and unconfigured states, entity-not-found errors
