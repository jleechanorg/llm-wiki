## Background
The SSE/streaming path in the web frontend (`handleSendMessage` in `App.tsx`) had several robustness gaps: silent failure on partial stream loss, no retry on transient network disconnects, no timeout indicator, and no visible connection state for the user.

## Goals
- Show a clear error message when streaming fails (including when partial text was already received)
- Retry on transient network disconnect: up to 3 retries with exponential backoff (1s, 2s, 4s)
- Timeout handling: show "taking longer than expected" after 30s with no events
- Connection state indicator: show Connecting / Connected / Timeout / Error states visually

## Tenets
- Minimal scope: only touch `packages/web/src/`
- No server changes
- Preserve existing stream parsing logic exactly

## High-level description of changes
`handleSendMessage` in `App.tsx` is refactored into an `attemptStream(attempt)` helper that:
1. Sets `streamingState` to `'connecting'` on each attempt
2. Sets a 30s timeout that triggers `'timeout'` state if no events arrive
3. On transient error (`TypeError` / network), waits `RETRY_DELAYS_MS[attempt-1]` and recurses up to 3 times
4. On final failure, appends `[Connection lost]` to any partial text (was previously silent)
5. Uses `AbortController` per attempt so stale requests are cancelled if a newer message is sent

`GameScreen.tsx` gains a `streamingState` prop and renders a badge with `aria-live="polite"` between the chat log and input area. Input and Send button are disabled while streaming to prevent duplicate submissions.

`styles.css` gains a `.streaming-status` utility class.
