## Background

Players sometimes attempt impossible actions (fly without wings), send ambiguous input ("I deal with the merchant"), or reference things outside the game's scope. Previously, the system instruction had a basic UNSUPPORTED / AMBIGUOUS INPUT FLOW section but lacked explicit rules for how the narrator should handle these cases in-world.

## Goals

- Never break immersion with meta-game refusals ("you can't do that")
- Always narrate impossible attempts through the world itself, with consequences
- Present ambiguous actions as in-world decision points with choices
- Ensure every player action is acknowledged and resolved or redirected

## Tenets

- In-world responses over system messages
- TDD contract phrase assertions before implementation
- Minimal scope: prompt rules + tests only, no runtime code changes

## High-level description of changes

Two new sections added to the LLM system instruction (system_instruction.md):
1. **IMPOSSIBLE ACTION HANDLING — IN-WORLD RESPONSES ONLY** — bans meta-game refusals, requires narrating the attempt and its world-consequences via state_delta/mechanic_requests, and mandates 2-4 redirect choices.
2. **AMBIGUOUS ACTION HANDLING — CLARIFY IN CHARACTER** — forbids breaking narration to ask "what do you mean?", presents interpretations as in-world choices, allows best-fit intent when context suffices.

The master_directive.md mirrors both policies in its Unsupported & Ambiguous Action Policy section.
