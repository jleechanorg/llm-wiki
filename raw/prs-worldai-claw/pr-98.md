## Raw Body
## Background
Companion RNG seeds were constructed as `companion.id:tickSeed` where `tickSeed` came from `state.simulationTickCount`. When `runCompanionTicks` ran as a standalone WorldScheduler phase without `runSimulation` having advanced the tick counter, companions produced identical action outcomes across ticks. The faction simulator already addressed this by seeding its RNG with `campaignId::tick::day::hour`.

## Goals
- Ensure each scheduler tick produces different but deterministic companion actions
- Match the tick-varying seed pattern already used by faction simulation RNG
- Preserve backward compatibility for callers that dont pass world time

## Tenets
- Determinism preserved: same inputs always produce same outputs
- Minimal scope: only touch RNG seed composition, no action logic changes
- No breaking API changes: worldDay/worldHour are optional fields

## High-level description of changes
Added `worldDay` and `worldHour` optional fields to `CompanionTickContext`. The RNG seed in `resolveCompanionAction` now includes day and hour: `companion.id:tick:day:hour`. Both callers (`runSimulation` and `runCompanionTicks`) inject `state.worldTime.day` and `state.worldTime.hour` into the context. `processCompanionTicks` propagates these through its internal `actContext`.

## Testing
- 6 new unit tests in `determinism.test.ts` covering:

  - Same tick + different day = different output
  - Same tick + different hour = different output
  - Different tick + same day/hour = different output
  - Consecutive ticks produce varied BALANCED action types
  - Backward compat when worldDay/worldHour omitted

  - processCompanionTicks with different worldDay
  - processCompanionTicks with different worldHour
  - processCompanionTicks determinism with same world time
