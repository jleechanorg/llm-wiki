## Background
Concurrent append operations in `packages/backend/src/storage/entity_chain.ts` have a race condition: `block_count` and `prev_hash` are computed from the in-memory `this.blocks` array **outside** the SQLite transaction. If an external writer (another chain instance or process) inserts blocks between the read and the transaction commit, the append produces a duplicate `block_id` (UNIQUE constraint violation) or an incorrect `prev_hash` (hash-chain fork).

## Goals
- Eliminate the race window by reading `block_count` from the DB **inside** the SQLite transaction
- Detect stale in-memory state and recover by reading the last block from the DB
- Reload in-memory blocks when external writes are detected after commit

## Tenets
- Prefer real tests over mocks
- Minimal, auditable changes — only the append path is modified
- TDD: tests written and verified RED before implementation

## High-level description of changes
Moved block_id and prev_hash computation inside the `better-sqlite3` transaction function. The transaction now reads `block_count` from `entity_chains` (the authoritative source) instead of `this.blocks.length`. When the in-memory array length doesn't match the DB count, the last block is read from the DB to compute `prev_hash`. After the transaction commits, if the new block's ID indicates external writes occurred, `loadBlocks()` is called to resync.
