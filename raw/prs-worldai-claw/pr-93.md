## Background
Companion actions (movements, interactions, decisions) need to survive server restarts. The core persistence layer (schema, saveCompanionAction, getActionsSince, tick-save in runSimulation) was already in place. This PR fills the remaining gaps.

## Goals
1. Add getActionsByCompanion(companionId, limit?) for querying actions by companion ID
2. Add startup companion state validation that loads and counts companions on boot
3. Add comprehensive round-trip persistence tests proving data survives DB close/reopen

## Tenets
- Minimal changes - build on existing patterns, do not restructure
- Prefer real SQLite tests over mocks
- Verify persistence survives restarts (close/reopen cycle)

## High-level description of changes
- Added getActionsByCompanion() to SQLiteCompanionRepository - the existing getActionsSince() only queries by session ID, not by companion ID. The new method supports an optional limit parameter.
- Added startup companion state loader in server.ts that eagerly loads all companions across campaigns after SQLite initialization and logs the count. This validates persistence is working on every boot.
- Added 10 new tests in companion_action_persistence.test.ts covering: CRUD round-trip, ordering, limit, DB close/reopen survival, tick integration with processCompanionTicks, campaign-wide loading, and rewards serialization (null + complex nested).

## Testing
- 10 new tests: all pass
- 71 existing companion tests: all pass (no regressions)
- Tests use real SQLite (temp DB files), not mocks

## Low-level details

- **packages/backend/src/storage/sqlite_repository.ts**: Added getActionsByCompanion(companionId, limit?) method. Queries companion_actions table by companion_id with optional LIMIT, returns CompanionAction[] ordered by timestamp ASC.
- **packages/backend/src/server.ts**: Added startup companion state loader. After SQLite initialization succeeds, loads all campaigns and their companions via getCompanionRepository().getCompanionsByCampai
