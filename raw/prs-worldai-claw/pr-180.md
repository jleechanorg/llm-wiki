## Background
Replaces the static 2D Javascript renderer with a proxy endpoint that calls an Image Generation API (e.g. OpenAI DALL-E, Grok, etc.) to dynamically construct and deliver character sprites during RPG gameplay.

## Goals
- Allow AI-generated high-fidelity image building blocks for characters.
- Abstract the proxy logic to the `worldai_claw` backend to avoid exposing API keys on the browser.
- Automatically scale down massive generative outputs to fit the active 48px tile grid of the game.

## Details
- Endpoint `POST /api/sprites/generate` created in `packages/backend/src/app.ts`.
- Asynchronous pre-fetch mapped directly into `worldData.visual_config` inside `llm-client.js`.
- `scene.load.image` utilized to synchronously append Base64 URIs into Phaser's preload bootloader.

> [!NOTE]
> **Medium Risk**
> Adds a new authenticated backend proxy that calls an external image-generation API and returns base64 image payloads, which increases security/configuration risk (API key handling, timeouts, error paths) and can affect runtime performance/latency. Client-side rendering changes are localized but introduce new async loading paths that could fail or regress asset loading if not handled correctly.
> 
> **Overview**
> Enables *ML-generated player/NPC sprites* by adding `POST /api/sprites/generate` in `packages/backend/src/app.ts`, which validates input, requires agent auth, and proxies to a configurable image-generation API (model/url/key via env) returning base64 (or url) image data.
> 
> Updates the `rpg-engine` to prefetch these sprites during world generation (`llm-client.js`), store them into `worldData.visual_config` (`player_base64`, `npc_base64s`), and load them via Phaser (`scene.load.image`) with a procedural-canvas fallback (`sprites.js`). Player and NPC sprites are also normalized to the tile grid by scaling oversized textures down to `TILE_SIZE` and adjusting physics bodies (`game.js`, `npc.js`).
