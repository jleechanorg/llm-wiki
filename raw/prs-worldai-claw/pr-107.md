## Background
`getCid()` on `SQLiteEntityChain` performs a real DB lookup (`SELECT ipfs_cid FROM entity_blocks WHERE entity_id = ? AND block_id = ?`) but had zero test coverage. The implementation was added as part of the entity chain interface but tests were never written.

## Goals
- Add comprehensive test coverage for `getCid()` to catch regressions
- Verify all return-value paths: null (missing block, null CID) and valid CID string

## Tenets
- TDD: tests written and verified against the existing implementation
- Real SQLite `:memory:` databases, no mocks
- Minimal scope: one test file, no production code changes

## High-level description of changes
Added 6 test cases covering the full behavior surface of `getCid()`:
1. Returns null when no matching block exists
2. Returns null when block exists but `ipfs_cid` is NULL
3. Returns the CID string when block has one set
4. Selects correct CID among multiple blocks
5. Cross-entity isolation (wrong entity_id returns null)
6. Non-existent entity_id returns null

## Testing
All 6 new tests pass. All existing entity_chain test suites continue to pass (6 suites, 0 failures).

## Low-level details
- **packages/backend/src/storage/entity_chain_getCid.test.ts**: New test file with 6 tests exercising `SQLiteEntityChain.getCid()` against in-memory SQLite databases. Uses direct DB updates to simulate IPFS CID assignment (matching how `pinBlockAsync` works in production).
