## Background

In commit 0405f55, the `append()` method in `entity_chain.ts` was fixed to move IPFS pinning after the SQLite row insert, preventing a race condition where IPFS failure could leave the chain in an inconsistent state. However, no regression tests were added to guard this ordering contract.

## Goals

- Add comprehensive regression tests that verify IPFS pinning happens after SQLite insert
- Prevent the race condition from being reintroduced by future changes
- Cover edge cases: IPFS failure isolation, async CID writeback, concurrent appends

## Tenets

- Prefer real tests over mocks — only mock the IPFS bridge (external network dependency)
- Test the ordering contract, not implementation details
- Minimal scope — only add the test file, no production code changes needed

## High-level description of changes

Adds 12 TDD regression tests in a new test file organized into 5 test groups:

1. **SQLite-before-IPFS ordering** — verifies block exists in SQLite immediately after `append()` returns, before IPFS resolves; confirms `ipfs_cid` is not set synchronously
2. **IPFS failure isolation** — verifies `append()` succeeds when IPFS bridge throws, hash chain remains valid, no data loss across multiple appends
3. **Async CID writeback** — verifies `ipfs_cid` is written to DB and in-memory block after async pin resolves; stays NULL on rejection
4. **Concurrent append integrity** — verifies rapid sequential appends maintain hash chain with pending IPFS; `appendAsync` serializes writes correctly
5. **No-bridge fallback** — verifies `append()` works without IPFS bridge configured

Uses a `MockIpfsBridge` with controllable resolve/reject for deterministic testing.
