## Background

`processCompanionTicks()` existed in `faction_simulator.ts` and was called during simulation, but only as an implicit side-effect buried inside `runSimulation()`. The WorldScheduler had no direct visibility into companion processing.

## Goals

- Make companion tick processing an explicit, visible phase in the WorldScheduler tick loop
- Allow companion ticks to run even if the faction simulation phase throws
- Enable independent testing of the scheduler to companion tick integration
- Maintain backward compatibility for all other callers of `runSimulation()`

## Tenets

- Minimal scope: extract and wire, do not refactor unrelated code
- Backward compatible: other callers of `runSimulation()` still get companion processing by default
- Resilient: companion ticks run even if faction simulation fails

## High-level description of changes

Extracted a standalone `runCompanionTicks(campaignId, db)` function that encapsulates the full companion tick lifecycle (load companions, build LLM context, plan actions, execute ticks, narrate, persist). The WorldScheduler now calls `runSimulation(skipCompanionTicks: true)` followed by `runCompanionTicks()` as separate phases. A new `RunSimulationOptions.skipCompanionTicks` flag prevents double-processing.

## Testing

- 5 new unit tests in `world_scheduler_companion_ticks.test.ts`
- Verifies scheduler calls runSimulation(skipCompanionTicks=true) then runCompanionTicks
- Verifies companion ticks run even when runSimulation throws
- Verifies multi-campaign sequential processing and concurrent tick locking
- Existing scheduler tests confirmed same pass/fail counts (pre-existing native binding issues in worktree)

## Low-level details

- **packages/backend/src/world/faction_simulator.ts**: Added `RunSimulationOptions` interface, `skipCompanionTicks` option, and exported `runCompanionTicks()` function (~140 lines)
- **packages/backend/src/world/world_scheduler.ts**: Import runCompanionTicks, split tick into faction sim + c
