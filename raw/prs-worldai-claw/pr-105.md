## Raw Body
### Background

`verifyCids()` (later renamed `hasCids()`) was flagged as misleading — it claimed to verify IPFS CIDs but only checked if CID strings were present on blocks, never fetching from IPFS. Tracked as WC-hnt in the roadmap.

### Goals

1. Keep `hasCids()` as an honest lightweight CID-presence check (already renamed)
2. Add a new `verifyCids()` that actually fetches content from IPFS by CID and verifies it matches block data
3. Add `fetchByCid()` to `IpfsBridge` to support real IPFS content retrieval
4. Full TDD test coverage for both methods

### Tenets

- Honest naming: methods do what their names say
- Real verification over presence checks
- TDD: tests written and failing before implementation
- Minimal scope: no unrelated changes

### High-level description of changes

- **`IpfsBridge.fetchByCid(cid)`**: New method that retrieves content from IPFS by CID. Checks local cache first, then fetches from Pinata gateway or local IPFS node (`/api/v0/cat`). Returns `null` on failure.
- **`SQLiteEntityChain.verifyCids()`**: New method that iterates blocks with CIDs, fetches each from IPFS via `fetchByCid()`, and compares the fetched JSON against the canonical block representation. Returns `{ verified, failed, errors }`.
- **`CidVerificationResult` interface**: Exported type for `verifyCids()` return value.
- **`EntityChainInterface`**: Updated to include `verifyCids()` in the contract.

### Testing

16 TDD tests in `entity_chain_cid_verification.test.ts`:

**hasCids() (5 tests):**
- Returns false without bridge, without CIDs, when IPFS unhealthy
- Returns true with CIDs present, even partial CID coverage

**verifyCids() (8 tests):**
- No CIDs → verified=0, failed=0
- No bridge → error message
- Unhealthy IPFS → error message
- Content match → verified=1
- Content mismatch → failed=1 with "content mismatch" error
- Content not found → failed=1 with "not found" error
- Mixed results across multiple blocks
- fetchByCid throwing → graceful error handling
