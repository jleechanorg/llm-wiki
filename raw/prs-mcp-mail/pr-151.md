## Summary

Fixes 5 critical violations where MCP tools claim `project_key` is "informational only" but actually require a valid project by calling `_get_project_by_identifier()`, which raises `NoResultFound` when the project doesn't exist.

This violates the documented global agent namespace design where agents are globally unique and should be accessible without project context.

## Changes

Fixed 5 MCP tools to follow the correct pattern:

- **fetch_inbox** (MCP-iu3): Look up agent first, use `agent.project_id`
- **send_message** (MCP-uxk): Look up sender first, use `sender.project_id`
- **whois** (MCP-c25): Global agent lookup without `project_key` requirement
- **delete_agent** (MCP-915): Look up agent globally, use `agent.project_id` for deletion
- **reply_message** (MCP-2pj): Look up sender first, use `sender.project_id`

## Root Cause

Tools were calling `_get_project_by_identifier(project_key)` BEFORE looking up agents, causing failures when `project_key` didn't exist—even though agents are globally unique and don't require project context.

## Fix Pattern

**Before (violating):**
```python
project = await _get_project_by_identifier(project_key)  # FAILS if project doesn't exist
agent = await _get_agent_by_name(agent_name)
```

**After (correct):**
```python
agent = await _get_agent_by_name(agent_name)  # Look up agent first (global)
project = await _get_project_for_agent(agent)  # Get project from agent
if project is None:
    project = await _get_default_project()  # Fallback to default
```

## Reference Implementation

`mark_message_read` and `acknowledge_message` correctly implement "informational only" by skipping `_get_project_by_identifier()` calls entirely.
