## Raw Body
### Motivation
- FastMCP/Pydantic tool introspection was broken by the instrumentation wrapper, causing server startup failures and schema inference issues for tools.
- Concurrent project creation, SQLite contention, and case-sensitive recipient deduplication caused crashes or data integrity errors under normal usage.
- A few safety/stability and hygiene issues (unbounded `limit` params, tiny agent-name namespace, global monkey-patch, and runtime dev deps) needed quick fixes and regression tests.

### Description
- Preserve FastMCP/pydantic introspection by exposing the original function signature on instrumented wrappers via `wrapper.__signature__ = signature` in `_instrument_tool` (keeps instrumentation behavior while restoring introspection). (`src/mcp_agent_mail/app.py`)
- Fix race on concurrent `ensure_project` inserts by catching `IntegrityError`, rolling back, and re-selecting the existing `Project` row. (`src/mcp_agent_mail/app.py`)
- Make recipient deduplication case-insensitive in message delivery by normalizing keys with `item.strip().lower()` in the `_unique()` helper to avoid duplicate `(message_id, agent_id)` inserts. (`src/mcp_agent_mail/app.py`)
- Reduce SQLite concurrency-induced locking by using a single connection pool for SQLite (`pool_size=1, max_overflow=0`) and move other DB engines to the original pooling; also avoid dropping the case-insensitive unique index at startup and run the duplicate-name migration only when the index is missing. (`src/mcp_agent_mail/db.py`)
- Replace the global `asyncio.iscoroutinefunction` monkey-patch with a scoped `warnings.filterwarnings(...)` to suppress the specific deprecation warning instead of modifying stdlib behavior. (`src/mcp_agent_mail/__init__.py`)
- Clamp `limit` parameters for inbox/search tools to a safe range (`min(max(limit, 1), 1000)`) to prevent unbounded memory/IO usage across `fetch_inbox`, `search_mailbox`, `search_messages`, and product variants. (`src/mcp_agent_mail/app.py`)
