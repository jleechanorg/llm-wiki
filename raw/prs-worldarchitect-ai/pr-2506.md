## Summary

This PR implements comprehensive dice roll integrity enforcement across 8+ defense layers, fixing the DICE-ayy fabrication gap and establishing production-grade dice auditing infrastructure.

### Problem Statement
Campaign audits revealed two critical issues:
1. **Statistical anomaly**: Campaign `tAE30bFvyfO0rUd9cgyv` showed 28/90 d20 rolls as "2" (31% vs expected 5%)
2. **DICE-ayy gap**: LLM could fabricate dice in narrative text during "smart skill checks" when no combat was detected, bypassing code_execution requirements

### Root Cause Analysis
- Gemini 3 Flash fabricates dice values instead of using code_execution
- Original detection only checked USER INPUT for combat keywords
- LLM could initiate skill checks in NARRATIVE without triggering dice requirements
- tool_results validation accepted ANY tool, not specifically dice tools

---

## Solution: 8-Layer Defense-in-Depth

### Layer 1: Unified Logging for Auditing (dice.py, llm_service.py via logging_util)
- DICE_AUDIT: Every dice roll with notation, rolls, modifier, total, nat20/nat1 flags
- DICE_TOOL_EXEC: Tool invocation entry with name and arguments
- DICE_TOOL_RESULT: Tool execution result with all relevant data
- **Unified logging**: Same `logging_util` calls work locally (stdout) and in production (GCP Cloud Logging)
- Enables production monitoring and anomaly detection

### Layer 2: "DICE VALUES ARE UNKNOWABLE" Prompt Framing (prompts/game_state_instruction.md)
Frames dice values as external data that must be queried (like weather or stock prices), making fabrication logically inconsistent rather than just forbidden.

### Layer 3: Rejection Loop for Fabrication Detection (llm_service.py)
- _is_code_execution_fabrication() detects when model claims dice_rolls but didn't use code_execution
- Code_execution fabrication triggers dice_integrity_violation, activating reprompt loop
- Enhanced reprompt messaging explains both code_execution and tool_requests paths
