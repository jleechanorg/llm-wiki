## Why this fix is needed alongside #5977

PR #5977 (now merged to main) fixes **where** world_events appear in story entries for the streaming path — extracting them from `state_updates` nesting so the UI can see them.

This PR fixes **what happens to existing events** when the LLM sends a partial status update. These are two distinct bugs at different layers:

### The bug this PR fixes

When a background event like this exists in game state:
```json
{"actor": "Trade Ship 'The Salty Gull'", "action": "Smuggling contraband",
 "event_type": "long_term", "location": "Harbor", "status": "pending", ...}
```

And the LLM emits a status-only update (Turn 7):
```json
[{"actor": "Trade Ship 'The Salty Gull'", "status": "discovered"}]
```

`update_state_with_changes()` in `firestore_service.py` treated the list as an opaque overwrite, replacing the full event with just the skeleton. **All fields stripped** — `action`, `event_type`, `location`, `discovery_condition`, etc.

### The fix

`_merge_background_events()` performs actor-keyed merge: find the existing event by `actor`, update only the provided keys (`status`), preserve everything else.

### Why #5977 doesn't cover this

#5977 fixes the streaming *extraction path* (`streaming_orchestrator.py`). It correctly surfaces world_events in story entries. But it does not change how `update_state_with_changes()` handles list overwrites in `firestore_service.py`. Without this fix, the first time the LLM updates an event's status, the full event object is destroyed — in both streaming and non-streaming paths.

### Evidence from live testing (iteration_003)

Turn 7 confirmed: `discovered:2` — two events correctly updated to `status: discovered` while preserving all other fields (`action`, `event_type`, `location`, etc.).

Without this fix, those same Turn 7 events would be saved as `{actor, status, turn_generated, scene_generated}` — 4 fields instead of 10+.
