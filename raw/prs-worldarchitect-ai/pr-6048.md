## Background
The explicit caching implementation in `_call_llm_api_with_explicit_cache` sends old story entries TWICE per request:
1. In the cache prefix (at \$0.05/M cached rate)
2. In the live `story_history` JSON (at \$0.50/M full rate)

This doubles the cost of old entries — ~\$0.069/req wasted for a campaign like Visenya v5 (529 entries, 138K double-billed tokens).

## Goals
- Eliminate double-billing for old story entries when explicit cache is active
- Maintain full context for the model via a `story_history_note` field
- Zero regression to existing caching behavior

## CR Review Fixes (POSTED: P1 + P2 addressed)

**P1 - N-1 Deferral Guard** (`llm_service.py:1831-1853`): Added guard that detects when `uncacheable_json.story_history` is empty (occurs when `entries_to_cache=story_count` during cache rebuild) and falls back to `full_content_for_retry` to ensure the current turn is always sent.

**P2 - Static Fields Preserved** (`llm_service.py:1977-1982`): In cache-reuse path, added `live_json.update({game_mode, user_id, selected_prompts, use_default_world})` from `cacheable_json` to preserve required context fields in the live request.
