## Root Cause Analysis

### The Bug
The LLM sometimes outputs `world_events` to the wrong nested location:
```
state_updates.custom_campaign_state.world_events  (wrong)
```
Instead of:
```
state_updates.world_events  (correct)
```

### Why This Happens
**The system instructions ARE correct.** Line 16 of `mvp_site/prompts/living_world_instruction.md`:
```
DO silently generate events in state_updates.world_events
```

Lines 308-336 show the correct structure with `world_events` as a sibling of `custom_campaign_state`, not nested inside it.

**However, LLMs are probabilistic.** They do not always follow instructions perfectly. In production campaign STpjRuwjeUt97tpCl5nK (250 turns), the LLM wrote `world_events` to the wrong nested location at least once (turn 246).

### Why It Was Not Caught
1. `extract_structured_fields()` only checked top-level `state_updates.world_events`
2. `update_state_with_changes()` recursively merges ALL state_updates into game_state
3. Result: New events stored at `custom_campaign_state.world_events` but never extracted for UI
4. Old stale events at `game_state.world_events` (turn 27) displayed instead of recent ones (turn 246)

### The Fix
**Defensive coding:** Since we cannot guarantee the LLM follows instructions 100% of the time, we extract `world_events` from BOTH locations with top-level taking precedence.
