## Raw Body
## Problem with Previous Fix

PR #2021 used `--suppress-logs` flag, but this **didn't actually work**:
- Latest auto-deploy run (19362228213) still failed with log streaming error
- Build 91d4514f-df2a-4b83-8383-51fa5a4d554f succeeded, but workflow failed
- `--suppress-logs` only suppresses stdout; gcloud still tries to stream logs internally

## Root Cause

The permission error occurs during gcloud's internal log streaming connection, not during stdout printing. The `--suppress-logs` flag can't prevent this.

## Correct Solution: --async with Polling

Use `--async` flag to submit build without waiting, then poll for completion:

```bash
# Submit build asynchronously (returns immediately with build ID)
build_id=$(gcloud builds submit . --tag "$image_tag" --async --format="value(id)")

# Poll for completion every 5 seconds
while [[ "$status" != "SUCCESS" && "$status" != "FAILURE" ]]; do
  sleep 5
  status=$(gcloud builds describe "$build_id" --format="value(status)")
done
```

## Benefits

✅ Completely avoids log streaming permission check  
✅ Still waits for build completion  
✅ Provides status updates during polling  
✅ Links to Cloud Console for full logs on failure
