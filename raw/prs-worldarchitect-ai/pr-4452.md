## Raw Body
### Motivation
- Prevent consolidated summary posts from silently failing when the combined body exceeds GitHub’s comment size limit.  The previous change concatenated all replies into one body without a total-size guard, which can cause the POST to fail and result in zero replies posted. 
- Ensure idempotency and detection correctness for the new consolidated `Re: [Comment #<id>]` format so reruns do not post duplicate summaries.
- Keep automation guidance and tests aligned with the consolidated-summary workflow and ensure coverage checks correctly treat consolidated summaries as addressing comments.

### Description
- Added size-bounded chunking and fail-fast posting to consolidated summaries in the Claude implementation (`.claude/commands/commentreply.py`), including logic to split replies into multiple comment parts when needed and to abort with non-zero status if posting fails.
- Mirrored the same chunking + failure handling and idempotency fixes in the Gemini implementation (`.gemini/scripts/commentreply.py`).
- Updated idempotency detection to accept both legacy and consolidated reference patterns (checks for `In response to [comment #N]` and `Re: [Comment #N]`) so repeated runs do not duplicate replies.
- Adjusted the consolidated-summary collection flow to return a boolean success indicator and treat failures as fatal (exit non-zero) after posting attempts.
- Updated PR automation guidance and detection logic in `automation/jleechanorg_pr_automation/jleechanorg_pr_monitor.py` to: document the consolidated mode more explicitly (added "do not post individual GitHub API replies alongside /commentreply"), extract `Re: [Comment #<id>]` references from existing consolidated summaries, and apply those addressed IDs across comment types so issue/review/inline comments are recognized as replied.
