## Summary

- **Bug**: `level_up_pending=True` in `custom_campaign_state` caused \"LEVEL UP AVAILABLE!\" to persist in UI, but planning block showed only narrative choices (no level-up option)
- **Root cause**: Three-layer failure — stale `needed_for_next_level` field (64000 instead of 85000), conflicting LLM prompt trigger, and stale flag never written back to Firestore
- **Investigated via**: Direct Firestore query on campaign `b9LPKcLHEwvG4FGsQDpu` (Ramsay V1, seq=343)

## Production Code Changes

**`game_state_instruction.md`**: Removed 5-step LLM-facing level-up trigger that compared `current >= needed_for_next_level`. The `needed_for_next_level` field was stale (64000 = level 10 threshold) so LLM triggered false level-up at 70550 XP. Now defers entirely to rewards system.

**`game_state.py`**: Added `_recalculate_needed_for_next_level_in_place()` called from `_canonicalize_player_character_data_in_place`. When `level` is known, always sets `needed_for_next_level = xp_needed_for_level(level+1)`. Eliminates the stale value that caused the false trigger.

**`agents.py`**: Added `clear_stale_level_up_pending()` and wired into both stale-detection sites. When `_is_stale_level_up_pending` returns True, now also writes `level_up_pending=False` back in-place so the flag doesn't persist across turns.

## Test Changes

5 new TDD tests in `test_level_up_stale_guards.py`:
- `TestNeededForNextLevelRecalc` (3 tests): Verify `needed_for_next_level` is recalculated correctly after level-up
- `TestStaleLevelUpPendingClearedInState` (2 tests): Verify stale flag is cleared via `clear_stale_level_up_pending()`
