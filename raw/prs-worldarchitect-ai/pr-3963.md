## Summary
This PR updates only `mvp_site/` non-test code to improve Gemini caching behavior, robustness, and diagnostics.

## Changes (mvp_site non-test only)
### `mvp_site/gemini_cache_manager.py`
- GOAL: Centralize explicit cache lifecycle (create/reuse/delete/TTL) per campaign.
- CHANGE: New `CampaignCacheManager` with rebuild threshold, TTL, and cache creation that embeds system_instruction in `CachedContentConfig`.
- JUSTIFICATION: Explicit caching requires coordinated cache management; isolating it avoids leaking Gemini-specific cache logic into request flow.

### `mvp_site/llm_providers/gemini_provider.py`
- GOAL: Allow explicit cache references while honoring Gemini API constraints.
- CHANGE: Add `cache_name` to generate paths; when cache is set, skip tools/system_instruction and attach `cached_content` to config. Propagate cache name through native tools and code_execution flows (Phase 1 only).
- JUSTIFICATION: Gemini rejects cached requests if tools/system_instruction are present; the provider layer is the correct enforcement point.

### `mvp_site/llm_request.py`
- GOAL: Improve implicit cache hit rate and support explicit cache split.
- CHANGE: Reorder fields in `to_json()` (static → append-only → dynamic) and add `to_explicit_cache_parts()` to separate cacheable vs uncacheable payload.
- JUSTIFICATION: Prefix-stable ordering increases implicit cache hits; explicit caching requires a deterministic partition of stable vs dynamic content.
