## Summary
Enhanced ralph scripts with symlink-safe path resolution, support for custom repository paths via `--dir` flag, improved process matching, and tool availability validation. These changes make the scripts more robust and flexible for various deployment scenarios.

## Changes
- **Symlink-safe path resolution**: Updated `dashboard_server.sh` and `status.sh` to properly resolve `SCRIPT_DIR` when invoked via symlinks, enabling placement in `PATH`
- **Custom repository path support**: Added `--dir` argument to `dashboard_server.sh` and `status.sh` to allow specifying an explicit repository path, useful when ralph/ lives outside the target repo
- **Improved argument parsing**: Implemented proper argument parsing with support for both short (`-o`, `-w`) and long (`--open`, `--watch`, `--dir`) flags
- **Process matching refinement**: Changed `pgrep` patterns from `ralph/ralph.sh` to `ralph\.sh` to match both symlinked and direct invocations
- **Tool validation**: Added verification in `ralph.sh` that the selected tool (amp or claude) is available in PATH before execution
- **Default tool change**: Changed default tool from `amp` to `claude` in `ralph.sh`
- **Socket reuse fix**: Added `TCPServer.allow_reuse_address = True` to prevent "Address already in use" errors when restarting the dashboard server
- **Documentation updates**: Updated README with new usage examples and clarified symlink-safe behavior

## Benefits
- Scripts can now be safely symlinked into `PATH` for convenient access from any directory
- Users can monitor/manage multiple projects by specifying different repository paths
- Better error messages when required tools are missing
- Prevents socket binding errors when restarting the dashboard server
- More flexible deployment options for teams with non-standard project structures
- Clearer documentation of available options and use cases

## Testing
