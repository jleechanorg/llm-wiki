## Raw Body
### Motivation
- Ensure logs are written to both Cloud Logging (stdout/stderr) and the local per-run file even when convenience logging functions are invoked before explicit setup.
- Make the default service name configurable via environment to avoid surprising file names for non-entrypoint callers.
- Reduce friction for callers that use module-level `getLogger()` or convenience functions without remembering to call the global setup.
- Document the auto-initialization behavior so callers understand when unified logging is enabled.

### Description
- Add `_ensure_logging_initialized()` which lazily calls `setup_unified_logging()` using `LOGGING_SERVICE_NAME` or a `_default_service_name` (`"app"`).
- Call `_ensure_logging_initialized()` from convenience methods and `LoggingUtil.getLogger()` so `info()/error()/warning()/debug()/critical()/exception()` auto-initialize logging when needed.
- Expose `_default_service_name` and update the top-level module docstring to explain the lazy-init behavior and env override.
- Preserve existing idempotent initialization semantics: the first `setup_unified_logging()` call still determines the configured log file, and subsequent requests reuse the original configuration.

### Testing
- Unit tests for the logging util exist in `tests/test_logging_util.py` and were added earlier to validate repo/branch parsing, idempotency, and thread-safety.
- This specific lazy-init change did not have new tests executed as part of the edit (pre-commit linter reported issues), so full test runs were not performed for this patch.
- Recommended verification: run `pytest tests/test_logging_util.py` and `./run_tests.sh` locally or in CI to confirm no regressions.
