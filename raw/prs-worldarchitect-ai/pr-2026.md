CRITICAL IMPROVEMENT: Fix copilot workflow to prioritize code correctness over comment coverage metrics

## Problem Solved (PR #740 Lessons)
- Previous workflow missed critical admin rate limiting bug
- Binary DONE/NOT DONE protocol hid lack of actual implementation
- 100% comment coverage metric masked production-blocking bugs
- Agent delegation caused critical bugs to be classified as "routine"

## Key Changes

### Phase 0: Critical Bug Scan (NEW - MANDATORY FIRST STEP)
- Scan ALL comments for critical keywords before any processing
- Keywords: CRITICAL, BUG, SECURITY, BLOCKER, rate limit, authentication, admin UID
- Direct implementation of critical bugs (NO agent delegation)
- Write tests and verify fixes
- Commit each critical bug fix before proceeding

### Priority Hierarchy (NEW)
1. CRITICAL: Security vulnerabilities, production blockers, data corruption
2. BLOCKING: CI failures, build failures, breaking API changes
3. IMPORTANT: Performance issues, logic errors, missing validation
4. ROUTINE: Code style, documentation, optional refactoring

### Action-Based Response Protocol (REPLACES binary protocol) **Before:**
- Binary DONE/NOT DONE responses
- No accountability for actual implementation
- Hid missing fixes behind "DONE" responses

**After:**
- Response types: FIXED, DEFERRED, ACKNOWLEDGED, NOT_DONE
- Required fields for FIXED: category, action_taken, files_modified, commit, verification
- Clear accountability with commit SHA and test verification
- Example: "FIXED - Implemented UID-based admin check in RateLimitTool.ts:145 - Commit: 53702d91 - Tests pass"

### Quality-Focused Metrics (REPLACES coverage metrics) **Before:**
- Comments responded to: 34/34 (100%)
- Hid that critical bugs weren't fixed

**After:**
- Critical bugs fixed: X/X (100% required)
- Blocking issues fixed: X/X (100% required)
- Important issues: X fixed, Y deferred
- Routine responses: Z
