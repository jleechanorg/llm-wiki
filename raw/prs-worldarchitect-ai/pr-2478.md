## Overview
This PR hardens the orchestration runner against **rate-limit / quota exhaustion** failures and generalizes the “fallback” concept into an **ordered CLI chain**.

Primary goals:
- Stop **silent no-op runs** where a CLI prints quota/rate-limit errors but still exits `0`.
- Allow users to specify **fallback order** explicitly (example: `gemini,codex`).
- Produce **valid JSON** results with enough metadata to debug/monitor fallback behavior.
- Make the same chain behavior work in `--fixpr` automation via `--fixpr-agent`.

---

## Problem
In `--fixpr` mode (and other agent runs), Gemini CLI can emit quota/rate-limit errors such as:
- `You have exhausted your daily quota on this model`
- `Resource exhausted` / `RESOURCE_EXHAUSTED`

…but still exit with code `0`.

The older runner logic treated `exit_code == 0` as success unconditionally, so:
- a task could be marked **completed**,
- the result JSON would be written as **completed**,
- and **no fallback** would be attempted,

…even though the agent effectively did no work.

---

## Solution (high-level behavior)
### 1) Ordered CLI chains
Support comma-separated model/CLI chains that are tried **in order**, stopping on first clean success.

Examples:
- `--agent-cli=gemini,codex` → try Gemini, then Codex if Gemini fails or rate-limits
- `--fixpr-agent gemini,codex` → same, but specifically for fixpr mode

### 2) Rate-limit detection is log-based (not just exit-code)
For each attempt, the runner:
- captures the attempt’s **log segment** boundaries (line offsets),
- scans *only the new lines* appended by that attempt,
- treats any match to the quota/rate-limit pattern as **failure**, even if the CLI exited `0`.

Current pattern (case-insensitive):
- `exhausted your daily quota|rate limit|quota exceeded|resource_exhausted`
