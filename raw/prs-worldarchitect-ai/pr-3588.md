## Raw Body
## Problem

Dice rolls are sometimes missing from the `dice_rolls` field even when they exist in `action_resolution.mechanics.rolls`. For example, the Dispater execution entry had dice rolls in `action_resolution.mechanics.rolls` but the `dice_rolls` field was empty, causing dice rolls to not display in the UI.

## Root Cause

The current system asks the LLM to populate BOTH `action_resolution.mechanics.rolls` AND `dice_rolls` separately. This creates a dual-maintenance burden and the LLM sometimes forgets to populate `dice_rolls` even when `action_resolution.mechanics.rolls` is correctly populated.

## Solution

**Single Source of Truth Approach:**
1. **LLM Instructions Updated:** Tell LLM to ONLY put dice rolls in `action_resolution.mechanics.rolls` (DO NOT populate `dice_rolls` directly)
2. **Backend Extraction:** Automatically extract dice rolls from `action_resolution.mechanics.rolls` and format them for the `dice_rolls` field
3. **Backward Compatibility:** Still accepts direct `dice_rolls` population for legacy responses, but prefers extracted rolls

## Changes

### Prompt Updates
- `narrative_system_instruction.md`: Updated to tell LLM to centralize dice rolls in `action_resolution.mechanics.rolls` only
- `game_state_instruction.md`: Clarified that `dice_rolls` is auto-extracted (DO NOT populate directly)

### Backend Changes
- `action_resolution_utils.py`: 
  - Added `extract_dice_rolls_from_action_resolution()` function
  - Added `extract_dice_audit_events_from_action_resolution()` function
  - Updated `add_action_resolution_to_response()` to automatically extract and populate `dice_rolls` and `dice_audit_events`
