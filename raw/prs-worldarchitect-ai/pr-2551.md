## Summary
- Fix Gemini code_execution dice uniformity by **seeding RNG in the system prompt** (`random.seed(time.time_ns())`) so each execution has fresh entropy.
- Add a real-server chi-square test script in `testing_mcp/` to verify 20-roll and 100-roll distributions without seeding in the script.
- Keep RNG seeding strictly in **mvp_site game code** (not tests), so production behavior is corrected.

## Root Cause (from real Gemini 3 Flash runs)
- Gemini code_execution appears to **re-seed deterministically per execution**, causing repeated first-draw values (extreme chi-square) when requesting one roll per call.
- With explicit time-based seeding **inside the executed code**, distributions normalize.

## Fix (code)
- `mvp_site/llm_providers/gemini_provider.py`
  - Adds `import time` and `random.seed(time.time_ns())` in the code_execution example.

## Tests / Evidence
- `testing_mcp/chi_square_gemini_seeded.py` (real local MCP server, Gemini-3 Flash)
  - Uses **game code seeding** (no seeding in the script).
  - Computes chi-square for 20 rolls and writes evidence under `testing_mcp/evidence/chi_square_seeded/`.

**Additional real-world validation (local MCP server, Gemini-3 Flash):**
- 20 actions: chi-square **22.0**
  - Evidence: `/tmp/worldarchitect.ai/dev1766449707/real_campaign_20_actions/`
- 100 actions: chi-square **18.8**
  - Evidence: `/tmp/worldarchitect.ai/dev1766449707/real_campaign_100_actions_retry/`

## Notes
- RNG verification remains in `gemini_code_execution.py` to guard against fabrication.
- Seeding is performed in **production prompt**, not in tests, to fix real behavior.
