## Root Cause: Why The Cache Was Failing (Often Immediately)

`should_rebuild()` only checked entry count and code_execution flag — **it never checked if the model changed**. The `CampaignCacheManager` had no memory of which model a cache was built for.

**Exact failure sequence (pre-fix):**
1. User plays 5+ turns → cache built with model A (e.g. `gemini-2.0-flash`)
2. User changes model to B (e.g. `gemini-3-flash-preview`) in settings
3. Next request: `should_rebuild()` → False (entry count fine) → returns old cache (built for A)
4. Gemini API: request uses model B + cache for model A → `FAILED_PRECONDITION: model mismatch`
5. Old code: `else: raise` → hard failure surfaced to user

**Why "fast":** Failing on the very first request after a model settings change. With `REBUILD_THRESHOLD=5`, caches are built every ~5 turns. The N-1 deferred switch (old cache used for 1 extra request after rebuild) compounded this — even if model tracking triggered a rebuild, the N-1 logic would return the wrong-model old cache.

## Production Code Changes

### `mvp_site/gemini_cache_manager.py`

**Model tracking:**
- Added `_cached_model` / `_pending_model` fields — track which model each cache was built for
- `should_rebuild(model_name=...)` — force rebuild when model changes, log `CACHE_MODEL_MISMATCH`
- `create_cache` / `promote_pending_cache` / `reset_cache` maintain model fields consistently

**Immediate switch on model mismatch:**
- Before: N-1 deferred logic always returned the OLD cache for 1 extra request after rebuild (even if that old cache was for the wrong model → guaranteed FAILED_PRECONDITION)
- Now: Model mismatch triggers immediate cache switch (same path as `code_execution` tool upgrade) — logs `CACHE_CREATED (model_mismatch immediate switch)`
- Result: After a model change, the first request uses the brand-new cache (slight propagation-delay risk of 0 cached_tokens) instead of guaranteed FAILED_PRECONDITION
