## Root Cause Analysis

Investigation revealed **two distinct issues**:

### 1. FAILED_PRECONDITION (400 error)
- Intermittent API rejection from `gemini-3-flash-preview`
- Works on retry - transient Google-side issue
- **Fix**: Auto-retry up to 3 times with exponential backoff

### 2. Truncated JSON Responses
- Gemini returns partial JSON (e.g., 27 chars when expecting thousands)
- Current logging showed truncation but not WHY
- **Fix**: Log `finish_reason` to diagnose (STOP, SAFETY, LENGTH, etc.)

## Changes

### Retry Logic
- `GEMINI_RETRY_MAX_ATTEMPTS = 3`
- `GEMINI_RETRY_BASE_DELAY_SECONDS = 2.0` (exponential: 2s, 4s)
- `_is_retriable_gemini_error()` - Identifies FAILED_PRECONDITION
- `_log_retry_attempt()` - Consistent retry logging
- `_add_api_retry_warning_to_response()` - Adds user warning

### Response Metadata Logging
- `_log_gemini_response_metadata()` - Logs finish_reason, text_length, safety_ratings
- Warns when finish_reason != STOP (indicates truncation)

### Enhanced Error Logging
- Extracts nested RPC debug details from error.details[]
- Logs raw response text (first 500 chars)
- Captures request IDs from headers

### User Warning
When retry succeeds, user sees:
```
API retry required (attempt 2/3) - transient error recovered
```
(Similar to dice integrity warnings in `_server_system_warnings`)
