## Problem: Two Critical Bugs in Campaign kuXKa6vrYY6P99MfhWBn

### Bug 1: RewardsAgent Infinite Loop (52+ consecutive calls)
After combat ended, `rewards_processed` stayed `False` in game state. This caused:
- Agent selector to pick RewardsAgent every turn (combat ended but rewards not marked processed)
- RewardsAgent was called 52+ times consecutively
- User had to manually fix state via God Mode
- Living world stopped updating because RewardsAgent doesn't trigger living world

**Root cause:** LLM sometimes failed to set `rewards_processed=True` in the ESSENTIALS protocol.

### Bug 2: Living World Not Showing in Debug UI
Even when living world WAS working, `world_events` didn't appear in the debug panel:
- LLM correctly generated `world_events` in structured response
- Events were saved to Firestore storage path
- But API response didn't include them for frontend display

**Root cause:** Dec 28 commit fixed storage but broke the API response path - `world_events` wasn't copied from `structured_fields` to `unified_response`.

### Bug 3: RewardsAgent Generating Narrative
RewardsAgent was generating story continuation text and choices, blurring the line between rewards processing and narrative generation. This caused confusion when stuck in rewards mode.

## Fixes

### Fix 1: Server-Side `rewards_processed` Enforcement
Added `_enforce_rewards_processed_flag()` in `world_logic.py` as a safety net:

```python
# Sets rewards_processed=True when:
# 1. Combat ended (combat_phase in COMBAT_FINISHED_PHASES) + combat_summary exists
# 2. Encounter completed + encounter_summary exists  
# 3. XP increased during active combat/encounter
```

This prevents infinite RewardsAgent loops even if LLM forgets the ESSENTIALS protocol.
