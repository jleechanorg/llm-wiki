## Summary

Add end-to-end test for safety limits false rejection bug where PRs are incorrectly rejected with "Internal safety limits exceeded" despite having 0/50 attempts.

**Bead**: WA-3gx  
**Affected PRs**: #3185, #3664, #3096

## Problem

Production logs show:
```
🚫 Internal safety limits exceeded for PR jleechanorg/worldarchitect.ai #3185; skipping
```

But safety manager shows:
```
PR #3185: ALLOWED (0/50 attempts)
pr_attempts.json: {}
pr_inflight.json: {}
```

## Root Cause Analysis

**Bug is environmental, not a logic error**:
- try_process_pr() works correctly in isolation ✅
- Multiple cron jobs (fixpr, fix-comment, comment-validation) run every 30 minutes
- All compete for same pr_inflight.json file
- atomic_update() returns False when file lock held by another process
- try_process_pr() incorrectly returns False
- Error message misleading: says "safety limits" when it's "file lock contention"
