## Fix Gemini Safety Settings Application

### Problem
The application was receiving "Response did not contain valid text" errors, which broke JSON parsing.
- Logs revealed `BlockedReason.OTHER`, indicating a safety/policy refusal.
- The `google-genai` SDK requires `safety_settings` to be passed **INSIDE** `GenerateContentConfig`, not as a top-level argument to `generate_content()`.
- Attempting to pass them as a top-level argument causes `TypeError: Models.generate_content() got an unexpected keyword argument 'safety_settings'`.
- The SDK signature only accepts: `model`, `contents`, `config`.

### Solution
- Modified `mvp_site/llm_providers/gemini_provider.py` to pass `safety_settings` **inside** the `GenerateContentConfig` object when creating it.
- Per SDK documentation: "Use safety_settings inside a GenerateContentConfig object" (from `python-genai/codegen_instructions.md`).

### Verification
- **Reproduction:** Created a script (`debug_safety_real.py`) using real campaign data that reproduced the failure.
- **Fix Verification:** Applied the fix and verified the script successfully generated narrative without safety refusals.
- **Regression Test:** Added `mvp_site/tests/test_end2end/test_gemini_safety_settings_end2end.py` to ensure `safety_settings` are always passed inside `config`, not as top-level kwargs. This test was verified to fail when incorrectly passed as top-level and pass when correctly placed in config.

### Changes
- `mvp_site/llm_providers/gemini_provider.py`: Moved `safety_settings` from top-level `generate_content()` call into `GenerateContentConfig` constructor.
- `mvp_site/tests/test_end2end/test_gemini_safety_settings_end2end.py`: New regression test verifying correct placement.
