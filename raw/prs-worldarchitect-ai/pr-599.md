## Overview

Fixes critical production bug where Firestore write-then-read verification failed despite all tests passing. Root cause was incorrect tuple parsing in `collection.add()` calls and inadequate test coverage due to testing mode confusion.

## Key Changes

### 🔧 **Fixed Firestore Write-Then-Read Bug**
- **Fixed tuple parsing**: `collection.add()` returns `(Timestamp, DocumentReference)` - was using wrong index
- **Switched to ID-based verification**: Direct document lookup instead of unreliable text matching  
- **Added progressive retries**: 3 attempts with configurable delays for Firestore eventual consistency
- **Enhanced error logging**: Better debugging for verification failures

### 🧪 **Separated Testing Modes**
- **`AUTH_SKIP_MODE`**: Bypass authentication only (for testing real services)
- **`MOCK_SERVICES_MODE`**: Skip verification + use test models (for unit tests)
- **Removed `TESTING` backward compatibility**: Clean separation, no legacy mode mixing

### 📊 **Improved Test Coverage**
- **Fixed `FakeFirestoreClient`**: Now returns proper tuple format like real Firestore
- **Updated failing tests**: All tests now use appropriate mode (`MOCK_SERVICES_MODE` for unit tests)
- **Red-green testing**: Demonstrates production bug was caught by improved fakes

## Why This Matters

**Before:** All tests (including "end-to-end") used `TESTING=true` which bypassed production verification code entirely. Production bugs weren't caught.

**After:** Tests can use real service logic with `AUTH_SKIP_MODE=true` while unit tests use `MOCK_SERVICES_MODE=true` for speed.

## Technical Details

**Original Error:**
```
Write-then-read verification failed: Could not find document 'mock-doc-...' for actor='user' after 3 attempts
```

**Root Cause:**
```python
# WRONG: Using timestamp instead of document reference
doc_ref = add_result[0]  # This was a Timestamp\!

# FIXED: Correct tuple unpacking  
doc_ref = add_result[1]  # DocumentReference with .id attribute
```
