## Summary
Add timing metadata to the done event payload in story continuation streaming to enable clients to track when streams begin and measure end-to-end latency.

## Changes
- Capture `stream_start_time` as a UTC ISO 8601 string at the start of `continue_story_streaming()`
- Calculate `e2e_latency_seconds` as elapsed time from preparation start to done event
- Include both fields in the done event payload for client-side timing and monitoring
- Add comprehensive test to validate payload structure and data types

## Benefits
- Clients can display when story streams began to users
- Clients can measure total round-trip duration for performance monitoring
- Enables better observability and debugging of streaming latency issues
- ISO 8601 format ensures consistent timestamp handling across systems

## Testing
- Added `test_continue_story_streaming_done_payload_includes_stream_start_time_and_e2e_latency` to validate:
  - `stream_start_time` is present and a valid UTC ISO 8601 string
  - `e2e_latency_seconds` is present and a non-negative float
  - Both fields are correctly included in the done event payload
