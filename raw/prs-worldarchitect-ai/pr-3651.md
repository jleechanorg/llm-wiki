## Summary

This PR fixes critical JSON parsing failures that occur when Gemini uses code execution mode, causing "Invalid JSON response received" errors in production. The solution removes code execution artifacts before JSON parsing and includes comprehensive test coverage.

## Problem Statement

**Production Issues Observed:**
- "Invalid JSON response received. Please try again." errors shown to users
- JSON parsing failures with error: `"Expecting value: line 1 column 2 (char 1)"`
- Errors occur specifically when Gemini uses code execution mode (e.g., for dice calculations)
- Campaign `yxU6r6UuGFthtDvVsxSl` experiencing repeated failures

**Root Cause:**
When Gemini uses code execution mode, the response text includes code execution artifacts (whitespace, stdout output, or JSON arrays) before the actual JSON object. The parser attempts to parse from the start, hitting invalid characters at position 1, causing `JSONDecodeError: Expecting value: line 1 column 2 (char 1)`.

**GCP Log Evidence:**
```
2026-01-16 02:11:51,833 - ERROR - Failed to parse JSON response: 
Expecting value: line 1 column 2 (char 1) Error at position 1.

GEMINI_CODE_EXECUTION_PARTS: code_execution_used: True, 
stdout: '[{"notation": "1d20+799", ...}]\n' (valid JSON)
```

## Changes

### 1. Code Execution Artifact Removal (`mvp_site/narrative_response_schema.py`)
Added intelligent prefix removal in `parse_structured_response()`:
- **Detects non-JSON prefixes**: Checks if response doesn't start with `{`
- **Finds actual JSON start**: 
  - If array `[` comes first (code execution output), finds the `{` that comes **after** the array closes
  - Otherwise finds the first `{` or `[`
- **Removes prefix**: Strips everything before the JSON start character
- **Logs removal**: Records how many characters were removed for debugging
