## Summary

- `shutil.rmtree(ignore_errors=True)` in `_fresh_clone` can silently no-op on macOS (locked files, permission issues), leaving the stale clone directory in place
- The subsequent `git clone` into the non-empty dir exits 128 with an opaque error, causing `fix-comment` to fail with no actionable message
- Audit showed **56 PR dispatch failures** in 24h, all from this root cause

## Production Code Changes

**Before**: `rmtree(ignore_errors=True)` → `git clone` (clone fails with exit 128 if dir persists)

**After**: `rmtree(ignore_errors=True)` → check `base_dir.exists()` → raise `RuntimeError` with clear message + manual cleanup path if dir still present → `git clone`

**Why**: Surfacing the real failure (rmtree failed) with a clear error and remediation path is strictly better than an opaque `git clone` exit 128 that looks like a network or auth problem.

## Test Changes

Added `test_fresh_clone_raises_when_rmtree_silently_fails` (TDD: RED → GREEN):
- Monkeypatches `shutil.rmtree` to a no-op to simulate silent failure
- Asserts `RuntimeError` is raised with `"Failed to remove existing clone"` message
- All 64 existing tests pass

## Known Limitations

- Does not auto-retry with forced removal; operators must manually `rm -rf` the stale clone. A follow-up could add a `stat`-based fallback or alert.
