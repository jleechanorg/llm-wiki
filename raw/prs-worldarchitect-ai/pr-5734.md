## Design Tenet — LLM-Recoverable Workflow

The pairv2 workflow automates what a human does manually:
1. Give the coder a spec → wait for it to say it's done
2. Run verifier → ask "is it really done?"  
3. If not → tell coder to keep going
4. Repeat N cycles or until verifier says 100% PASS

**The workflow must NEVER stop or FAIL for any reason other than:**
- Verifier LLM says task is 100% complete (PASS)
- All N retry cycles exhausted (max_cycles reached)

**Specifically banned:**
- Gating on exact file paths, signal filenames, or directory naming conventions
- Downgrading verifier verdicts based on non-LLM checks (missing signal files, timing checks, schema validation)
- Hard-failing on missing artifacts without letting the verifier judge
- Fuzzy matching / rglob fallbacks (still brittle, just differently)

**Instead, use:**
- Process lifecycle as completion signal (exit = done)
- LLM inference to evaluate workspace against left/right contracts
- Notes/warnings for anomalies, never hard gates
- Monitor that restarts stuck agents instead of killing the whole flow

## Production Code Changes

### `pair_execute_v2.py`
- **Before**: `_wait_for_implementation_ready_node` used threading.Event watcher thread polling for signal files; PASS downgraded to NEEDS_HUMAN when signal missing
- **Now**: Simple poll loop: (1) check signal file (fast path), (2) check tmux session alive (primary), (3) timeout (safety net); PASS accepted without signal file
- **Why**: Weaker models (MiniMax-M2.5) write signals to wrong paths; process exit is model-agnostic
