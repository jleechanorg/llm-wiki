## Summary

`pairv2` is the contract-first /pair orchestrator: it validates contract inputs, launches live coder/verifier attempts, verifies evidence and artifacts, and retries bounded cycles using a deterministic control plane while delegating only semantic judgment to an LLM evaluator.

The key design goal is to separate **what the script must prove by rule** from **what it lets the LLM judge semantically**.

## Runtime Design (Current)

Execution graph per attempt:
`generate_left_contract` ŌåÆ `generate_right_contract` ŌåÆ `left_contract_node` ŌåÆ `implement` ŌåÆ `verify` ŌåÆ `finalize`

Retry path: `verify` routes to `generate_left_contract` when cycle verdict is `FAIL` and cycles remain.

Fan-out (`--fan-out N`) still runs N attempts in parallel at the top level (per-attempt invocation with checkpoint thread scoping), then winner selection is applied in Python.

## Non-LLM Checks: What they do and why

### Input, schema, and integrity checks
- Contract file path/type/schema validation (`_validate_contract_file`, `_validate_contract_payload`) ŌĆö blocks malformed or missing contracts before execution.
- JSON load checks (`_load_contract_payload`, `_load_design_contract_text`, `_load_verification_report`) ŌĆö prevent runtime crashes from unreadable artifacts.
- Schema fallback handling (`_validate_contract_with_fallback`) ŌĆö keeps behavior stable when `jsonschema` is unavailable.
- LLM command/env checks for verdict evaluation (`_call_llm_json`) ŌĆö hard fail fast if verdict engine is unavailable.

### Contract readiness gates
- Left/Right readiness validation (`_contract_readiness_check`) ŌĆö enforces contract shape and readiness constraints before launch.
- Existing contract reuse (`_left_contract_node`, `generate_left_contract`, `generate_right_contract`) ŌĆö if contracts are supplied and valid, they are reused instead of regenerated.
- Open-question / `ready_to_build` checks (`_evaluate_left_contract_matrix`) ŌĆö prevents launching against incomplete planning contract.
