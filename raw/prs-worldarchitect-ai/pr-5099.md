## Summary
Updated installation documentation across automation and orchestration packages to clearly distinguish between production-safe installations and development-only editable installs, with explicit warnings about multi-worktree compatibility issues.

## Changes
- **automation/README.md**: 
  - Expanded troubleshooting section to explain why editable installs break in multi-worktree setups
  - Reorganized installation section with clear "Production/Cron Jobs (RECOMMENDED)" vs "Development Only" subsections
  - Added detailed warnings (⚠️ and 🚫) about when NOT to use editable installs
  - Provided three options for handling outdated PyPI packages
  - Added guidance for installing from git branches and building from source

- **docs/pr-guidelines/pr1723/guidelines.md**:
  - Updated code example comments to clarify production vs development installation methods
  - Specified that editable installs should only be used in automation directory for development

- **orchestration/README.md**:
  - Applied same installation best practices structure as automation package
  - Added "Production / Cron Jobs / Normal Usage (RECOMMENDED)" section
  - Added "Development Only" section with editable install guidance
  - Included warnings about editable install limitations
  - Added "When PyPI Package is Outdated" section with alternatives

## Benefits
- **Prevents production failures**: Clear warnings prevent users from using editable installs in cron jobs and multi-worktree setups where they break
- **Improves user experience**: Structured documentation makes it obvious which installation method to use for different scenarios
- **Reduces support burden**: Comprehensive troubleshooting section addresses common issues upfront
- **Enables safe development**: Developers understand when editable installs are appropriate vs dangerous
- **Supports multi-worktree workflows**: Explicitly documents that editable installs don't work reliably in git worktree setups
