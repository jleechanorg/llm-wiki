## Problem

When the LLM generates partial `world_time` data (e.g., only `hour`/`minute` without `year`/`month`/`day`), the missing fields defaulted to `0`, causing:

```
(0, 0, 0, 14, 15, ...) < (431, 5, 12, 14, 5, ...)
→ "TEMPORAL ANOMALY DETECTED: Time went backward!"
```

This was a **FALSE POSITIVE** - incomplete data is not time travel, it is malformed data that should be ignored.

## Solution

Add symmetric validation in `check_temporal_violation()`:
- Check **BOTH** `old_time` AND `new_time` for required date fields (`year`, `month`, `day`)
- If either is incomplete, return `False` (no violation) - we cannot reliably compare
- Only flag actual backward time travel when both timestamps are complete
