# LLM Planning Backend Refactor

## Problem Statement

Current architecture has the LLM managing game state through free-form `state_updates` JSON, leading to:
- **State drift**: LLM "forgets" resource values between sessions
- **No audit trail**: Dice rolls happen in narrative, not logged mechanically
- **Inconsistent mechanics**: Custom abilities lack structured execution

## Solution: Single-Inference with Backend-Owned State

**Before:** LLM generates narrative + guesses state changes → Backend accepts whatever LLM says
**After:** LLM generates `pending_operations` + outcome tables → Backend executes, rolls dice, selects outcome

```
Player: "I cast Fireball at the goblins"
    ↓
LLM generates:
  - pending_operations: [roll_dice(8d6), save_batch(goblins, DC 15)]
  - outcome_table: {
      high_damage: "31 fire. Fourteen goblins incinerated...",
      medium_damage: "22 fire. Eight goblins fall...",
      low_damage: "14 fire. Four goblins collapse..."
    }
    ↓
Backend executes:
  - Rolls 8d6 = 31
  - Runs 20 saves, 16 fail
  - Selects "high_damage" narrative
  - Returns full results to player immediately
```

**Key insight:** LLM pre-writes ALL possible outcomes with full narratives. Backend selects which one based on actual dice. No "continue" needed - immediate feedback.

## Features

### 1. Backend-Owned Resource Registry

```python
@dataclass
class ResourceDefinition:
    resource_id: str          # "spell_slot_3", "blood_points", "ki"
    display_name: str
    max_value: int
    current_value: int
    recharge_trigger: str     # "long_rest", "short_rest", "dawn", None
    computed_bonuses: dict    # Auto-applied: {"ac": "{divine_rank}", "attack": "{level}//4"}
    triggers: list[Trigger]   # Conditional effects
```
