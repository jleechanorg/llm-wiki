## Raw Body
## Why
Self-hosted runner was not reliably coming back after machine restart. `launchd` alone and existing monitor scripts did not guarantee recovery when startup timing was delayed.

## What changed
- Added `self-hosted/scripts/restart-on-reboot.sh` with boot-delay + retry loop + monitor fallback, writing `reboot-restart.log`.
- Updated `self-hosted/scripts/setup-github-runner-redundant.sh` to:
  - Copy both `monitor.sh` and `restart-on-reboot.sh` into the runner directory.
  - Install an `@reboot` crontab entry for boot recovery with configurable `REBOOT_DELAY_SECONDS`.
  - Include new script in cleanup and summary output.
- Updated `self-hosted/scripts/monitor.sh` to support lock-based single-instance execution, retry loop, and cleaner restart flow.
- Updated `self-hosted/scripts/README-crontab.md` to document reboot recovery job and current cron cadence (6h heartbeat).
- Added/updated bead `.beads/REV-self-hosted-runner-restart.md`.

## Runbook for target machine
- Set `REBOOT_DELAY_SECONDS` (default 60), `BOOTSTRAP_RESTART_DELAY_SECONDS`, `BOOTSTRAP_RESTART_ATTEMPTS`, and `BOOTSTRAP_RESTART_ATTEMPT_DELAY_SECONDS` if needed.
- Re-run `./self-hosted/scripts/setup-github-runner-redundant.sh` on the macOS runner host.
- Validate `crontab -l | grep "GitHub Actions Runner"` contains reboot, heartbeat, and monitor entries.
- After reboot, validate `crontab` boot log: `tail -f ~/actions-runner/reboot-restart.log`.
