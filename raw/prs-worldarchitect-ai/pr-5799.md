## Summary

- Added deploy-time warmup request to confirm service is live before returning

## Production Code Changes

**Before:** fastembed 0.7.4 defaults cache to `/tmp/fastembed_cache` → Cloud Run wipes `/tmp` on every cold start → workers find empty cache → HF API calls → shared Cloud Run IP gets 429 → exponential backoff (3+9+27s = 24-28s cold starts)

**Now:** `FASTEMBED_CACHE_PATH=/opt/fastembed_cache` forces model into image filesystem (survives restarts); gunicorn `on_starting` patches `model_info`, `list_repo_tree`, and `snapshot_download` to return cached results without any network calls; all 5 workers boot in <1s

**Why:** Cloud Run containers share an outbound IP (`2600:1900:...`) that HuggingFace rate-limits aggressively; the only reliable fix is to eliminate all HF network calls at runtime entirely

### Files changed
- `mvp_site/Dockerfile`: Added `ENV FASTEMBED_CACHE_PATH=/opt/fastembed_cache` before model download steps; updated tree-cache RUN to write JSON to same persistent path
- `mvp_site/gunicorn.conf.py`: Added `on_starting` hook with three patches (model_info, list_repo_tree, snapshot_download); added module-level imports for fastembed/httpx
- `deploy.sh`: Added warmup request after deploy to confirm service responds before script exits

## Test Changes

No test changes — verified in production via Cloud Run logs:
- `fastembed: model_info + list_repo_tree + snapshot_download patched — workers skip all HF API calls`
- All 5 workers boot within 1 second (timestamps 12:14:54 → 12:14:55)
- Zero HF 429 errors in logs

## Known Limitations

- Remaining ~10s cold start comes from Firebase initialization on first request (separate issue)
- `bge_small_tree_cache.json` SHA is baked at image build time; if the HF model changes SHA, a rebuild is n
