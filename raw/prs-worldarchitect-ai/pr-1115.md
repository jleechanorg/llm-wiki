## Problem

integrate.sh was reporting false positives, claiming branches had "unmerged commits" even after PRs were successfully merged. This happened because the script compared local branches against `origin/main` instead of the branch's own remote tracking branch.

**Scenario that failed:**
1. Create feature branch and push PR
2. PR gets merged on GitHub  
3. Run `integrate.sh` to clean up
4. Script incorrectly reports "unmerged commits" and blocks deletion
5. User forced to use `--force` flag unnecessarily

## Root Cause Analysis

The problematic logic was:
```bash
# Check against origin/main - WRONG after PR merge
commit_count=$(git rev-list --count origin/main..HEAD)
```

After PR merge, local branch still has commits not in `origin/main` until synced with remote tracking branch, causing false positives.

## Solution

**Primary Logic**: Compare local branch to its remote tracking branch
- If `local == remote`: Branch is synced after PR merge → Allow deletion
- If `local \!= remote`: Has unsynced changes → Block deletion with detailed status

**Fallback Logic**: For branches without remote tracking, preserve existing `origin/main` comparison

## Implementation Details

- ✅ **Enhanced error messages**: Show local/remote commit counts and summaries  
- ✅ **Backward compatible**: Fallback preserves existing behavior for non-tracking branches
- ✅ **Better UX**: Clear guidance on sync options vs generic "force" suggestions
- ✅ **Tested**: Verified both tracking and non-tracking branch scenarios
