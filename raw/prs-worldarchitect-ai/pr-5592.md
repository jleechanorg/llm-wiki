## Raw Body
### Motivation
- Prevent silent cross-provider fallbacks when user/provider settings are unavailable or invalid by enforcing a fail-closed selection policy. 
- Calls that would previously fall back to Gemini (or any other provider) must now surface an explicit error so downstream provider dispatch cannot run under a different provider than the user requested.

### Description
- Updated `_select_provider_and_model` in `mvp_site/llm_service.py` to raise `LLMRequestError` on missing/errored user settings (code paths: `PROVIDER_SELECTION_SETTINGS_UNAVAILABLE`, `PROVIDER_SELECTION_SETTINGS_EXCEPTION`) instead of returning a default provider/model. 
- Added explicit invalid-provider rejection (`PROVIDER_SELECTION_INVALID_PROVIDER`) which raises `LLMRequestError` when `llm_provider` is present but not allowed. 
- Adjusted logging to emit clear error messages (no “falling back to default model” wording in fail-closed paths). 
- Expanded TDD tests in `mvp_site/tests/test_llm_provider_selection.py` to assert fail-closed behavior and that `continue_story` surfaces selection errors without calling downstream LLM APIs.
