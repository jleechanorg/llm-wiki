## Summary

Post-PR-6003 bead followups — hardening cache management, tightening fallback logic, proactive TTL expiry, N-1 for first-ever caches, and provably-fair seed injection in streaming.

- **rev-ukw**: Parenthesize `and/or` in `create_cache()` for explicit precedence
- **rev-834**: Restrict PT011/B017 lint ignores to specific legacy test files (was global wildcard)
- **rev-oy0**: Narrow `reset_cache()` to actual cache errors; unknown errors re-raise (don't nuke valid caches for quota/safety/rate-limit failures)
- **rev-ixu**: Harden `test_failed_precondition_fallback`: cache extraction/deletion failure is a hard setup failure; fix cache name regex to match `cache_name=` and `new_cache=` log fields
- **rev-9td2**: Track `_cache_created_at`; proactively rebuild when `cache_age > TTL - 5min` to prevent FAILED_PRECONDITION on resume after long pause
- **rev-n1fc**: Apply N-1 deferral to first-ever cache creation via `CacheCreateResult(cache_name, deferred)` NamedTuple; callers preserve `system_instruction` when `deferred=True`
- **rev-seg3**: Inject provably-fair `SERVER_SEED` as prepended content part in streaming path for `MODELS_WITH_CODE_EXECUTION` (mirrors non-streaming path logic)

## Key Research Finding

`generate_content_stream` natively supports `code_execution` tools — Gemini runs code server-side, streams `text`/`executable_code`/`code_execution_result` parts in one call. This unblocks **rev-l0c4** (path unification) as a realistic separate PR. See `roadmap/2026-03-17-cache-streaming-unification-roadmap.md`.

## Production Code Changes

| Before | After | Why |
|--------|-------|-----|
| `reset_cache()` on any exception | Only on cache-specific errors | Quota/safety errors should not nuke valid caches |
| First-ever cache used immediately | Staged as pending (N-1 deferred) | Avoids FAILED_PRECONDITION from Gemini propagation delay |
| No TTL tracking | Proactive rebuild 5min before expiry | Prevents stale cache on campaign resume after pause |
