## 🚀 Three-Tier Memory-Safe Test Runner System

**Problem Solved**: 47GB+ memory consumption causing WSL2 crashes during test execution

### 🏗️ Architecture Decision: Shell Over Python

After architectural review with adversarial analysis, we chose a **simple shell-based solution** over the originally proposed Python multiprocessing migration:
- ✅ **Solves actual problem**: Memory/concurrency control  
- ✅ **MVP principle**: Minimal viable solution vs overengineering
- ✅ **Zero breaking changes**: All existing workflows preserved
- ✅ **50 lines vs 500+**: Dramatically simpler implementation

---

## 🎯 Three Environment Tiers

### Tier 1: Real GitHub Actions (`GITHUB_ACTIONS=true`)
- ❌ **No memory monitoring** (original behavior preserved)
- 🚀 **Full CPU parallelism** (`nproc` workers)
- 🎯 **Zero changes** to existing CI workflows

### Tier 2: CI Replica (`run_ci_replica` - `CI=true` only) 
- 🛡️ **10GB memory limit** with auto-kill protection
- 🚀 **Full CPU parallelism** for realistic CI testing locally
- ✅ **Safety net** for local CI debugging

### Tier 3: Local Development (default)
- 🛡️ **5GB memory limit** with auto-kill protection  
- 🔧 **4 workers** for conservative memory usage
- ✅ **Maximum safety** for WSL2/local development

---

## 🔧 Implementation Details

### Memory Monitor
- **Background process** monitors Python test processes every 10 seconds
- **Auto-kill protection** prevents system crashes
- **Environment-aware limits** (5GB/10GB/unlimited)
- **Non-blocking** - doesn't interfere with test execution

### Concurrency Control
- **xargs -P** limits parallel execution based on environment
- **Environment detection** via `GITHUB_ACTIONS` and `CI` variables
- **Backward compatible** - existing scripts work unchanged
