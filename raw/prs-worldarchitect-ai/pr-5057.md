> [!NOTE]
> **Medium Risk**
> Refactors the Gemini code-execution tool loop and enablement gating, which can affect when/which faction tools run and how two-phase retries behave; changes are well-covered by updated unit tests but touch core LLM execution paths.
> 
> **Overview**
> Moves Gemini `code_execution` tool-request orchestration out of `gemini_provider` and into `llm_service`, so the provider becomes an execution-only adapter and the service owns the two-phase loop (retry-on-enable, tool execution, prompt state update, and Phase 2 regeneration).
> 
> Tightens faction enablement detection in `faction_state_util.is_faction_enable_action` to rely on explicit structured flags or the exact `enable_faction_minigame` command (no keyword/phrase matching), and hardens tool-request synthesis/validation (safer `turn_number` coercion).
> 
> Improves `provider_utils` parsing and safety: supports top-level list responses for `tool_requests`, adds allowlisting for faction tools emitted via code-execution stdout, and makes tool-request injection failures non-f
