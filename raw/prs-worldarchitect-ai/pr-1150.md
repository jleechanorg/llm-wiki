# Updated PR Description for Standard Debug Mode Implementation

## Summary

Implements **standard debug mode behavior** where `debug_mode=True` shows **MORE** information for debugging purposes, following conventional debug mode semantics.

## Problem Analysis

The original issue was production users seeing confusing debug information when `debug_mode=True` was accidentally enabled. However, the proper solution is to:

1. **Maintain standard debug behavior**: `debug_mode=True` shows MORE information
2. **Rely on existing access controls**: Settings page already has debug mode toggle  
3. **Prevent accidental production exposure**: Use proper deployment practices

## Solution Implementation

### Standard Debug Mode Behavior
- **`debug_mode=True`**: Shows additional information including `state_updates` (for debugging)
- **`debug_mode=False`**: Shows only standard information (production-ready)
- **`state_changes`**: Always included for API compatibility

### Code Changes
Modified `mvp_site/world_logic.py` to implement standard debug behavior:

```python
# Always include state_changes for compatibility
unified_response["state_changes"] = response.get("state_changes", {})

# Add debug-only fields when debug mode is enabled (standard debug behavior)
if debug_mode:
    unified_response["state_updates"] = response.get("state_changes", {})
```
