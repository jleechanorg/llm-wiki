## Raw Body
## Summary

Based on controlled experiments, the anti-hallucination rules in CLAUDE.md proved ineffective and potentially harmful. This PR removes them while keeping useful debugging and evidence-based approaches.

## Experiment Results

We conducted blind A/B testing with:
- **Control**: Current CLAUDE.md with anti-hallucination rules
- **Treatment**: Optimized CLAUDE.md with more aggressive anti-hallucination rules

### Key Findings:
- **Both groups equally hallucinated** about test execution (claimed tests ran when they didn't)
- **Both fabricated** screenshot captures and specific test outputs  
- **Treatment group showed overcorrection** - falsely claimed inability to generate images when it could via Python scripts
- **Real improvement**: Only ~12%, mostly from better documentation, not reduced hallucinations

### What We Learned:
1. Abstract rules like "NEVER SIMULATE" don't prevent specific hallucinations
2. More rules can cause **learned helplessness** (claiming inability when capable)
3. Test execution hallucinations appear deeply ingrained and resist rule-based prevention

## Changes Made

### Removed (ineffective):
- 🚨 NEVER SIMULATE section
- 🚨 ANTI-HALLUCINATION MEASURES section
- 🚨 UNCERTAINTY ACKNOWLEDGMENT section  
- 🚨 ENHANCED ANTI-HALLUCINATION PROTOCOL section

### Kept (useful):
- ✅ EVIDENCE-BASED APPROACH - Good debugging methodology
- ✅ Screenshot rules - Clear technical requirements
- ✅ Browser testing guidelines - Specific implementation details

## Impact

This change acknowledges that:
- Some hallucinations may be architectural/training-based, not fixable via prompting
- Overly strict rules can harm performance without preventing core issues
- We should focus on specific, verifiable requirements rather than broad prohibitions
