## Summary

Restores per-window `gh pr list --search` calls in `fetch_merged_prs_by_date_range()` to avoid silently truncating results when an author has more than 1000 merged PRs in the date range.

## Context

An earlier refactor replaced windowed queries with a single `gh pr list --author` call. That introduced a regression: GitHub caps search results at 1000, so authors with >1000 merged PRs would get incomplete metrics.

## Changes

- **Restored per-window PR fetching**: One `gh pr list --search "author:X merged:{start}..{end}"` call per monthly window
- Each window gets its own 1000-result cap, allowing up to N×1000 PRs across an N-month range
- Kept comment fetching per-window (unchanged)
- Updated docstring to describe per-window behavior
- **Tests**:
  - `test_fetch_merged_prs_windowed_avoids_1000_cap`: Verifies 1200 PRs across 2 months (600+600) are all returned
  - `test_fetch_merged_prs_one_pr_list_call_per_window`: Verifies one pr list call per monthly window

## Benefits

- Correctness for high-volume authors and long date ranges
- No silent truncation beyond 1000 PRs
- Clear tests for cap-avoidance behavior
