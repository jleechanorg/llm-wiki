## Summary

Implements GraphQL-based detection of unresolved PR review threads with comprehensive error handling, pagination, and defensive programming fixes. Replaces unreliable REST API heuristics with GitHub's native `reviewThreads` field.

## Problem

The fixcomment workflow was running but making zero commits despite PRs having unresolved review comments. Root cause analysis revealed multiple issues:

1. **No ReviewThreads Access**: `gh pr view --json` uses REST API which does NOT expose the reviewThreads field
2. **No Resolution Status**: REST API provides no `isResolved` field for review comments
3. **Unreliable Heuristics**: Existing logic tried to infer thread resolution by checking for replies (incorrect)
4. **Timeout Violation**: Used 30s timeout instead of required 600s per CLAUDE.md
5. **Short-Circuit Bug**: Early return when GraphQL finds no unresolved threads skipped REST API issue comment check
6. **AttributeError Risk**: Missing defensive programming for null GraphQL responses

## Solution

Implemented comprehensive GraphQL-based detection with multiple defensive layers and bug fixes.

## Changes

### 1. GraphQL ReviewThreads Detection (Primary Fix)
- **Added `_has_unresolved_review_threads()` method**:
  - Uses GraphQL via `gh api graphql`
  - Fetches reviewThreads with `isResolved` field
  - **Pagination support**: Handles >100 review threads using `pageInfo.hasNextPage/endCursor`
  - Returns `True`/`False`/`None` (None on failure for fallback)
  - Includes debug logging with thread counts

### 2. Timeout Compliance Fix (f815668f7)
- **Fixed timeout violation**: Changed from 30s to 600s per CLAUDE.md timeout integrity rule
- Ensures GraphQL queries have sufficient time for large PRs

### 3. Short-Circuit Bug Fix (bc05038fe)
- **Fixed logic error**: Prevent early return `False` when GraphQL finds no unresolved review threads
- Now continues to check REST API for unaddressed issue comments
- Ensures both review threads AND issue comments are checked
