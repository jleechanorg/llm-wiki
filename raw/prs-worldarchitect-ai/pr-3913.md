## Summary
Unifies CLI timeout to 30 minutes for ALL CLIs (claude, codex, gemini, cursor) to allow agents sufficient time to complete complex PR fixes.

## Problem Identified
The previous implementation had a bug where only `claude` and `cursor` received 30-minute timeouts, while `codex` and `gemini` only got 10 minutes. However, **all CLIs use OAuth authentication** and can timeout on complex tasks:

- `claude` - unsets `ANTHROPIC_API_KEY` to force OAuth
- `codex` - unsets `OPENAI_API_KEY` to force OAuth  
- `gemini` - unsets `GEMINI_API_KEY` to force OAuth
- `cursor` - uses its own auth system

The hardcoded check `if attempt_cli in ["claude", "cursor"]` was incorrect and arbitrary.

## Changes
- **Unified timeout**: Set `RUNTIME_CLI_TIMEOUT_SECONDS` from 600s (10 min) → 1800s (30 min)
- **Removed redundant constant**: Deleted `RUNTIME_OAUTH_CLI_TIMEOUT_SECONDS` (no longer needed)
- **Simplified logic**: Removed CLI-specific if/else timeout logic in `task_dispatcher.py`
- **Updated comments**: Clarified that all CLIs use OAuth and may need time for complex tasks

## Benefits
- **Fixes timeout bug**: codex/gemini agents now get adequate time for complex tasks
- **Prevents premature timeouts**: All agents have consistent 30-min window
- **Simpler code**: Removed unnecessary branching logic
- **Better resource utilization**: Avoids wasted work from partial fixes that timeout

## Context
This change was needed after observing PR #3664 timeout after 15 minutes while trying to:
1. Fix import validation errors (try/except around imports)
2. Resolve 19 failing end2end tests
3. Push commits and verify CI

The agent was making progress but ran out of time. Analysis revealed the timeout distinction was fundamentally flawed.
