## Production Code Changes

**Before:** BYOK users got `daily_limit=1000` but `RATE_LIMIT_5HOUR_TURNS=25` still blocked them. Two separate functions fetched settings independently. Rate limit evaluation logic was copy-pasted between transactional and non-transactional code paths.

**Now:** Single `_get_user_turn_limits(user_id)` returns `(daily_limit, window_limit)` tuple with BYOK early exit. Shared `_evaluate_rate_limit()` handles all blocking logic for both code paths. BYOK limits: 5000 daily / 1000 per 5h (configurable via env vars).

**Why:** BYOK users pay for their own API calls and shouldn't be gated by free-tier throttles. Unlike full exemption, elevated limits maintain Firestore usage tracking for visibility into BYOK consumption patterns.

## Design Decision: Elevated Limits vs Full Exemption

This PR uses **elevated but bounded limits** (5000/1000) rather than full exemption (-1 unlimited). This preserves:
- Usage tracking in Firestore for analytics and abuse detection
- Configurable limits via environment variables for operational flexibility
- A safety net against runaway automation even with BYOK keys

See PR #5970 for the alternative "fully exempt" approach if unlimited is preferred.
