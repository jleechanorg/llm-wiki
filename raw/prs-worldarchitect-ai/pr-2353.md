## Problem / Motivation

Historically, the game relied on LLM prompt instructions for:
- performing D&D 5E computations (modifiers, prof, DC/AC comparisons, XP/level, etc.)
- rolling dice “in narrative”

That led to predictable failure modes:
- hallucinated or inconsistent dice outcomes
- incorrect D&D rules on edge cases (nat 1/20 behavior, advantage/disadvantage, etc.)
- poor debuggability (“why did the model decide that?”)

This PR centralizes the mechanics in code and standardizes how dice outcomes are produced and injected into story.

## Architecture Overview

### Core concept: tool-backed dice flow
The LLM never invents dice totals. Instead it requests rolls using standardized tools. The server executes them and the model must **copy results verbatim** into `dice_rolls`, `session_header`, and narrative.
