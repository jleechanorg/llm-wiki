## Raw Body
### Motivation
- Users need the ability to change the local OpenClaw gateway port from the Settings UI instead of only using the hardcoded default or server env var. 
- Persist and validate the port through the existing settings pipeline so runtime dispatch can use a per-user localhost port when `llm_provider=openclaw`.
- Add automated coverage for the UI+backend contract (requested `testing_mcp/` and `testing_ui/` locations) and unit tests to prevent regressions. 

### Description
- UI: added a numeric `openclawGatewayPort` input under the OpenClaw provider section in `mvp_site/templates/settings.html` and updated the help copy to clarify default vs server-level fallback.
- Frontend: extended `mvp_site/frontend_v1/js/settings.js` `SETTINGS_SCHEMA` with `openclaw_gateway_port` and added number-type handling in `readEntryValue`/`applyEntryValue` so the new field loads/saves like existing settings.
- Backend validation & defaults: added `validate_openclaw_gateway_port()` and registered it in `_STANDARD_VALIDATORS` in `mvp_site/settings_validation.py`, added `DEFAULT_OPENCLAW_GATEWAY_PORT` in `mvp_site/constants.py`, and wired defaults/setdefault for `openclaw_gateway_port` in `mvp_site/world_logic.py` so the unified settings payload always includes the key.
- API acceptance: allowed `openclaw_gateway_port` through the `/api/settings` accept-list in `mvp_site/main.py` and infer provider when `openclaw_model` is updated.
- Runtime plumbing: updated OpenClaw provider to resolve gateway URL from a `gateway_port` parameter and extended `mvp_site/llm_service.py` to read per-user `openclaw_gateway_port` (`_get_openclaw_gateway_port`) and pass `gateway_port` through both non-stream and streaming paths so calls use `ws://127.0.0.1:<port>` when configured.
- Tests: added/updated unit tests in `mvp_site/tests` covering provider handshake, streaming, actionable gateway-down error, and gateway-port override; added contract tests in `testing_mcp/test_openclaw_settings_contract.py`
