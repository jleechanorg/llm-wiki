## Summary
Consolidate runner-hosting policy in PR form: add a formal design decision and one tracking bead for self-hosted resilience.

## Goals
- Reduce recurring infra fragility by applying a consistent host-agnostic default to CI self-hosted jobs.
- Keep explicit host-specific checks intentionally narrow (runner infrastructure verification only).
- Decide where docker/container execution is valuable without forcing it across all self-hosted jobs.

## Scope added in this PR
- Added bead: `.beads/BD-runner-hosting-strategy.md`
- Added design doc: `docs/ci/runner-hosting-strategy.md`
- Policy decision: do not containerize all self-hosted jobs by default.

## Context
We already hit a fragility class where OS-specific assumptions leaked into shared self-hosted logic (`apt`-style assumptions + non-interactive sudo in hook-tests), which caused false infra failures even when host/workflows were otherwise healthy. This PR captures the policy decision to prevent that class of regression.

## Decision
1. Default to host-agnostic workflow logic for self-hosted CI jobs.
2. Keep host/supervisor checks (e.g., `test-self-hosted-runner.yml` runner binary check) explicit and isolated.
3. Do not adopt universal dockerization of CI workflows yet; containerize selectively only after per-job compatibility/reliability validation.

## Why this PR is valuable
- Prevents one-off host-specific fixes by creating a shared decision standard.
- Gives reviewers and future contributors a clear rule for when to write host-specific steps.
- Makes the next containerization work safer by defining explicit acceptance criteria and non-goals up front.
