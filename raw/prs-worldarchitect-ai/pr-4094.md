## Summary
Implements automatic retry mechanism for handling GitHub API 422 errors when posting replies to stale or deleted comments. This improves resilience by re-fetching comments and retrying failed operations instead of failing immediately.

## Changes
- **Enhanced return types**: Updated `create_threaded_reply()`, `create_issue_comment_reply()`, and `create_review_comment_reply()` to return a 3-tuple including error type classification (`None | "422_not_found" | "422_other" | "network" | "permission"`)
- **Error type tracking**: Added granular error classification to distinguish between stale comments (422_not_found) and other failures
- **Retry loop implementation**: Wrapped main comment processing in a retry loop (max 1 retry) that:
  - Tracks comment IDs that fail with 422_not_found errors
  - Re-fetches comments from GitHub after failures
  - Retries posting to previously failed comments
  - Exits early if no 422 errors occur
- **Improved logging**: Added detailed retry status messages and tracking of failed comment IDs
- **Idempotency preservation**: Maintained existing idempotency checks while supporting retry scenarios
- **Conditional summary posting**: Initial summary comment only posted on first attempt, not on retries

## Benefits
- **Increased reliability**: Handles race conditions where comments are deleted between fetch and reply operations
- **Better user experience**: Automatically recovers from transient stale comment errors without manual intervention
- **Clearer error reporting**: Distinguishes between recoverable (422_not_found) and non-recoverable failures
- **Reduced false negatives**: Prevents premature failure when comments become stale during processing
- **Maintains efficiency**: Single retry limit prevents infinite loops while allowing recovery from timing issues

## Testing
- Retry logic properly tracks 422_not_found errors separately from other failures
- Failed comment IDs are collected and re-fetched on retry attempt
- Idempotency
