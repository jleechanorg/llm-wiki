## Raw Body
### Motivation
- Provide a higher daily allowance for users who supply their own BYOK API key and are using the matching `llm_provider`, to reduce platform cost exposure while enabling heavier usage for BYOK users.
- Keep existing short-window (5-hour) fairness intact while only elevating the 24-hour ceiling for qualifying users.

### Description
- Added `RATE_LIMIT_BYOK_PROVIDER_DAILY_TURNS` (env var, default `1000`) and provider→key mapping `_PROVIDER_BYOK_KEY_NAME` to `mvp_site/rate_limiting.py` to model the elevated ceiling for BYOK users.
- Implemented `_get_user_daily_turn_limit(user_id)` which reads `get_user_settings(user_id)` and returns the elevated daily limit when the active `llm_provider` matches a non-empty BYOK key (e.g. `gemini_api_key`).
- Updated `check_rate_limit()` to compute and use a per-user `daily_turn_limit` (applied in both transactional and non-transactional paths), and adjusted daily-remaining calculations and error messaging to reflect the dynamic limit while leaving 5-hour window logic unchanged.
- Expanded `mvp_site/tests/test_rate_limiting.py` to mock both `rate_limits` and `users` collections and added tests covering: BYOK + matching provider (elevated limit applies) and BYOK key present but provider mismatch (default limit applies).
