## Background

Avatar downloads in `firestore_service.py` were returning `(None, None)` on not-found, masking errors from callers. The GCS `list_blobs` probe silently dropped valid blobs with `blob.updated is None`. Non-404 errors in the fallback probe were swallowed, giving callers a misleading 404.

Closes #6058

## Goals

1. Avatar downloads raise `ValueError` on not-found (callers get accurate 404)
2. GCS probe includes all valid avatar blobs (not just those with non-null `updated` metadata)
3. Non-404 storage errors propagate to callers (auth failures, 500s not silently swallowed)
4. Reduce unnecessary GCS probes via Firestore URL-hint fast path
5. Remove dead Grok/xAI avatar generation code

## Tenets

- Fix root causes, not symptoms — prefer behavior change over defensive null guards
- No silent swallowing of errors at system boundaries
- Tests must exercise the success path, not just error paths

## High-level description of changes

`download_user_avatar` / `download_campaign_avatar` now parse the extension from the stored `avatar_url` (fast path) and only fall back to GCS `list_blobs` when the URL is absent or stale. The probe no longer filters blobs by `updated is not None`. Non-404 errors are re-raised from the fallback probe. Routes map `ValueError` → HTTP 404. Grok endpoint removed.
