## Problem

Three critical bugs were blocking automation workflows from running proactively:

### 1. Dirty Repository State
The automation's base clone directory (`/tmp/pr-orch-bases/worldarchitect.ai`) would accumulate uncommitted changes from previous runs, entering a detached HEAD state. When `ensure_base_clone()` tried to reset with `git checkout main`, it would fail with:

```
error: Your local changes to the following files would be overwritten by checkout:
  .beads/beads.base.jsonl
  .claude/settings.json
  ...
Please commit your changes or stash them before you switch branches.
Aborting
```

### 2. Stale Inflight Cache
The `AutomationSafetyManager` loaded the inflight cache once at initialization and never reloaded it. When external processes cleaned up stale inflight slots (e.g., manual cleanup, other workflow instances), the in-memory cache never saw the updates. This caused PRs to be incorrectly blocked with "Internal safety limits exceeded" even when slots were available.

### 3. Daily Cooldown Blocking
The automation used daily cooldown (resets at midnight UTC) for attempt limits. PRs that hit the limit early in the day would be blocked for up to 24 hours, even if only 1 attempt remained. This prevented responsive automation and created unpredictable wait times based on time of day.

## Root Cause

### Dirty Repository
In `orchestrated_pr_runner.py:430`, the code ran:
```python
git checkout main          # FAILS on dirty repo
git reset --hard origin/main  # Never runs
git clean -fdx             # Never runs
```

### Stale Cache
In `automation_safety_manager.py:182`, the inflight cache was loaded once:
```python
def __init__(self, history_dir):
    # ...
    self._pr_inflight_cache = self._load_inflight_cache()  # Only loaded once!
```

The `try_process_pr()` method (line 388-408) checked this cached value without reloading from disk, causing stale data to persist across workflow runs.
