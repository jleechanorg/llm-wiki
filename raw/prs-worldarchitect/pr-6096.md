## Background
Self-hosted runners were silently capping at 2 despite expecting 6, and the launchd agent pointed to a worktree path that could be deleted. Runners were also repo-scoped instead of org-scoped.

## Goals
- Runners always start at the configured count on boot, from a stable path outside git
- Org-level runners serve all repos in jleechanorg
- Stale offline registrations can be cleaned with one command

## Tenets
- Stable over convenient: install to a path that survives repo moves and worktree deletes
- Single source of truth: RUNNER_COUNT in .env only, never hardcoded in scripts

## High-level description of changes

Root causes fixed:
1. launchd-start.sh had RUNNER_COUNT=2 hardcoded, overriding .env silently
2. launchd plist pointed to worktree path — breaks when worktree is deleted
3. Runners were repo-scoped; org-scope serves all repos

- install.sh: copies scripts to ~/.local/share/worldarchitect-runners/, points launchd there, wires RUNNER_ENV_FILE into plist
- launchd-start.sh: removed hardcoded RUNNER_COUNT=2 override
- status.sh: added running/desired count health check + --clean flag to delete stale offline registrations
- .env.example: switched default to RUNNER_SCOPE=org + ORG_NAME=jleechanorg, added RUNNER_COUNT=6
- README.md: documented ephemeral runner lifecycle, RUNNER_COUNT single source of truth, debug runbook

## Testing
- Ran ./install.sh — scripts deployed to ~/.local/share/worldarchitect-runners/
- launchd reloaded pointing to stable path (not worktree)
- 6 wa-oss-runner-* containers came up as org-level runners
- gh api orgs/jleechanorg/actions/runners confirmed all 6 online with RUNNER_SCOPE=org
