## Background
`update_campaign_game_state` (firestore_service.py:2983-3087) had a write-only Firestore transaction with no READ inside. Concurrent writes both succeeded (last-write-wins with no conflict detection) — silent data loss on inventory, HP, quest progress, etc.

## Problem
A write-only transaction (no get() inside) gives Firestore nothing to detect conflicts against. Two concurrent requests both read the doc version, then both write — the second write wins but the first's changes are only overwritten if they touched the same fields. When callers pass complete state snapshots (not partial updates), last-write-wins applied silently.

## Fix: READ + MERGE + WRITE inside the transaction

```python
@firestore.transactional
def _run_transaction(transaction):
    snap = game_state_ref.get(transaction=transaction)   # READ
    merged = dict(snap.to_dict()) if snap.exists else {}
    _deep_merge(merged, game_state_update)               # MERGE
    merged["last_state_update_timestamp"] = firestore.SERVER_TIMESTAMP
    transaction.set(game_state_ref, merged)              # WRITE
```

**Deep-merge** means:
- Fields only in current state → preserved
- Fields only in update → added
- Nested fields in both → recursively merged
- Top-level keys in both → update wins
- **None value in update → delete the key** (allows explicit field clearing)

This prevents two classes of lost updates:
1. Concurrent partial updates (combat + inventory + rewards) don't clobber each other
2. Concurrent complete-state writes: last write wins (correct semantic when two subsystems independently compute authoritative state)

Mirrors the pattern used by `rotate_personal_api_key` (line ~3264).
