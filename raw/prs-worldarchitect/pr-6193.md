## Background

On main (commit d868fec0c / PR #6161), the game UI stopped rendering the rewards_box when a player levels up. Investigation traced the regression to the new `mvp_site/rewards/` module pipeline introduced by PR #6161.

Evidence bundle: `/tmp/worldarchitect.ai/fix-rewards-box-dice-debug-render/investigation/latest/` — `/er` verdict **PASS** for investigation scope.

**Fix PR Evidence Gist:** https://gist.github.com/jleechan2015/4f694d18542f16d137f450c829fc441f (157 reward tests passing, 0 failures)

Key proof:

- Pre-#6161 main (`d868fec0c~1 = 8f95edde2`) has **zero** occurrences of `_normalize_rewards_box_for_ui` in `world_logic.py` and the same `if (rewards_box && xp_gained > 0)` frontend condition as live stable.
- Post-#6161 main has the normalizer wired in 12 call sites.
- PR #6161 is the exclusive regression source on `main`'s first-parent history (proven by `git merge-base --is-ancestor` checks on all suspect SHAs).

## Goals

- Stop silently erasing `rewards_box` payloads that have only level-up or progress fields (and no gold/loot/xp_gained).
- Keep the normalizer's existing responsibility: type-coerce and canonicalize fields, drop nothing.
- Match the comment at `app.js:923` ("server decides whether the payload is displayable") — by letting the server actually let the frontend render, rather than suppressing payloads server-side.

## Tenets

- Fix the specific suppression, not the whole pipeline.
- Do not revert #6161. The atomicity work has value; only the over-aggressive filter was wrong.
- Trust the frontend render block's existing inner guard to no-op on truly empty payloads.

## High-level description of changes

`mvp_site/rewards/builder.py` — remove the `has_visible_content → None` gate in `normalize_rewards_box_for_ui`. The function now always returns a normalized dict for any dict input (non-dict inputs still return None via the existing `isinstance` guard at line 17).
