## Background
PR #6161 fixes a stale polling issue where `rewards_box` and `planning_block` atomicity was broken during `get_campaign_state` polling. `rewards_box` was originally missing from the structured extraction utils, resulting in a badge without buttons state.

## Goals
1. Persist `rewards_box` alongside `planning_block` to Firestore.
2. Fix TypeErrors in the `world_logic.py` polling paths preventing dictionary access.
3. Ensure atomicity rules apply during polling equivalent to immediate action responses.

## Tenets
- Fail-closed: Ensure mismatched level-up badges are suppressed rather than showing broken UI.
- Strong shape guarantees: Polling functions must return primitive dicts instead of dataclasses.

## High-level description
- Integrated `FIELD_REWARDS_BOX` into the extraction layer.
- Refactored `get_campaign_state_unified` to accurately process LLM responses directly from history to evaluate rewards.
- Restored `game_state_dict` mapping to eliminate JSON serialization errors.

## New Files — Explicit Approvals

### `mvp_site/rewards/` module (user-approved)
This PR introduces `mvp_site/rewards/` as part of a user-approved rewards module extraction (bead bd-1xckb). The extraction moves rewards-specific logic out of `world_logic.py` and into a canonical module. Files: `__init__.py`, `domain.py` (data classes), `builder.py` (UI payload builders), `resolver.py` (level progression logic), `triggers.py` (pending rewards triggers).

### `testing_ui/test_level_up_rewards_planning_atomicity_browser.py` (new browser test)
This is a new browser-level integration test for the atomicity fix. No existing `testing_ui/` file covers the level-up rewards + planning_block co-visibility flow. The "no new test_*.py files" policy applies to `mvp_site/tests/` unit tests; `testing_ui/` browser tests are a separate tier with their own file structure.
