## Background

The dev deployment workflow fails with a version conflict race condition (ABORTED error) in `update_service_timeout()`. Also fixes stale `rewards_pending` flag on level-up modal exit (Goal 2).

## Goals

1. Eliminate flaky deployment failures from version-conflict races in `update_service_timeout()`
2. Fix stale `rewards_pending` level-up state not being fully cleared on modal exit

## Tenets

- Deployment success should not be gated on a best-effort post-deploy timeout update
- A successful `gcloud run deploy` is the authoritative result; the timeout update is advisory
- Non-conflict errors (auth, quota, config) should propagate to the caller — not be silently swallowed

## High-level description of changes

### `scripts/deploy_common.sh`
Replaced bare `gcloud run services update` call with a retry loop (up to 3 attempts):
- Retries only on `ABORTED|Conflict` errors (version race with concurrent deploy)
- Non-conflict errors (auth, quota, config) are propagated via `return $exit_code` — not silently swallowed
- Fixed sleep-after-last-attempt bug: only sleeps between attempts, not after the final one
- After exhausting retries on conflicts, returns 0 (deployment already succeeded via main deploy)

### `mvp_site/world_logic.py`
Clears `rewards_pending` using `DELETE_TOKEN` in the level-up exit handler to ensure Firestore removes the field (instead of setting it to None/null which leaves a stale key).

### `mvp_site/tests/test_level_up_stale_flags.py`
Added `test_rewards_pending_cleared_on_level_up_exit` to verify rewards_pending is fully cleared via DELETE_TOKEN on exit.
