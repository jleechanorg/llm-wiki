"## Background\n\nThe /sessions/:id/companions/catchup endpoint was calling companionRepository.updateLastSeen() BEFORE reply.send(), despite an existing comment claiming 'advance AFTER sending response'. This created an at-most-once delivery risk: if the server crashed after cursor advance but before TCP data left the socket buffer, the client would receive nothing yet have advanced its cursor \u2014 silently losing events.\n\n## Goals\n- Advance the companions_last_seen cursor ONLY after the response is fully flushed to the client TCP buffer\n- Ensure at-least-once delivery semantics: crash after flush-but-before-cursor means client retries with same cursor\n\n## Changes\napp.ts: Reordered catchup handler to call reply.send(), await reply.raw 'finish' event (TCP buffer flushed), then call updateLastSeen().\ncompanion_catchup_cursor.test.ts: New regression test covering cursor ordering.\n\n## Testing\n- Unit test: companion_catchup_cursor.test.ts\n- CI has correct build environment (local pnpm hoisting issue not representative of CI)\n\nCloses #499\n"
