## Background

Track H Web MVP gaps were blocking the worldai_claw web frontend from working correctly for real users. Four issues identified through code review of packages/web/src/.

## Goals

1. Fix API route naming drift so frontend consistently calls canonical backend routes
2. Ensure auth token has one consistent source of truth
3. Prevent users from bypassing setup when backend connection is unconfirmed
4. Make gameplay action buttons communicate their state clearly

## Tenets

- Minimal targeted changes; no broad refactors
- No breaking changes to backend routes
- TypeScript must remain error-free

## High-level description of changes

**API naming drift**: Dashboard and GameScreenWrapper were calling /api/worlds and /api/worlds/:id. The backend canonical route for listing/fetching campaigns is /campaigns (both /campaigns and /worlds exist but campaigns is the primary). Updated to /api/campaigns and /api/campaigns/:id.

**Auth token consistency**: Settings.tsx loads token from backend /api/settings response (field: openclaw_token) but apiClient reads worldclaw_auth_token from localStorage. When the backend token loaded on Settings mount, it updated component state but not localStorage, so apiClient would still use a stale token. Fixed by writing worldclaw_auth_token to localStorage whenever the backend response is loaded.

**Setup reliability**: The SetupWizard "Skip setup" button called skipToDashboard() directly, bypassing the OpenClaw connection test. skipToDashboard() already guards that the backend PATCH succeeds, but does not require the OpenClaw connection to be tested. Added a two-step confirmation: if connection has not been verified (state !== 'online'), the first click shows a warning with "Confirm: skip anyway" and "Cancel". If the connection is verified, skip proceeds immediately.

**Button UX**: GameScreen action buttons were wired to real actions (Roll Dice sends /roll d20, View World/View Characters toggle panels) but showed no state feedback
