## Background
The WorldScheduler uses in-process flags (`schedulerLock`/`isRunning`) to prevent concurrent ticks. However, in multi-process or crash-restart scenarios, the same tick could be double-processed, causing duplicated faction simulations and companion actions.

## Goals
Prevent double-processing of game ticks using persistent SQLite-backed dedup.

## High-level description
- Added `world_scheduler_dedup.ts` with `generateTickId()`, `isTickProcessed()`, `markTickProcessed()`, and `initTickLogTable()` utilities
- Created `scheduler_tick_log` SQLite table with `(tick_id, campaign_id)` composite primary key
- Wired dedup guard into `WorldScheduler.tick()` loop: each campaign tick is checked and recorded before processing
- Mark-after-success pattern: tick is only marked processed after successful lease acquisition

## Testing
7 unit tests in `world_scheduler_dedup.test.ts`:
- `generateTickId()` uniqueness
- `isTickProcessed()` / `markTickProcessed()` dedup semantics
- Cross-campaign independence
- Idempotent marking
