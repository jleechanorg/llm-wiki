## Background
Re-running `simulateWorldTick` on an already processed deterministic tick was triggering `updateCompanion` and `UPDATE sessions SET state` unconditionally, even though the `saveCompanionAction` or `player_offline_actions` inserts were silently ignored due to unique ID constraints (idempotent DB layer). This violated idempotency rules and risked overwriting concurrent player UI changes with stale state from a replay tick.

## Goals
Ensure that companion state and session JSON state are only overwritten if the corresponding action record was genuinely inserted (not ignored as a replay).

## Tenets
Maintain deterministic simulation and prevent state clobbering from overlapping or replayed ticks.

## High-level description
- Companion tick loops now await the result of `companionRepo.saveCompanionAction(…)` first. Only if it returns `true` (was inserted) is `updateCompanion(…)` executed.
- The player offline action block checks `insertRes.changes > 0` on the `player_offline_actions` table before running the `UPDATE sessions SET state` query and patching the in-memory `globalStore`.

## Testing
- Existing `catchup_level_gains.test.ts` and `faction_simulator.test.ts` pass.

## Low-level details
- `src/world/faction_simulator.ts`: Lines ~821-847, ~907-933, and ~1111-1135 were updated to conditionalize state updates based on successful action insertion.

Fixes wc-ca0
