## Background
Companion action history in `companion_actions` SQLite table was growing indefinitely with each tick cycle. Without pruning, long-running campaigns would accumulate thousands of action rows, degrading query performance and narration quality.

## Goals
- Cap companion action history per companion at a configurable limit (default 50 via `COMPANION_ACTION_LIMIT`)
- Prune oldest entries when limit is exceeded using age-based cutoff + overflow trimming
- Preserve the most recent entries and generate a compact JSON metadata summary of pruned actions
- Prevent `history_summary` audit records from accumulating unboundedly across compaction cycles (cap at 5)
- Ensure history_summary metadata (JSON) never leaks into narrator prompts or level-gain computation

## Tenets
- Real deletes over soft-deletes: pruned records are removed from DB, not just marked
- Audit trail over deletion: one `history_summary` record per compaction cycle for forensic recovery
- Structured metadata over prose: `generateActionSummary` returns valid JSON, not narrative text

## High-level description of changes
Added `compactCompanionHistory()` in `story_compactor.ts` to partition companion actions by age and overflow. Called from `saveCompanionAction()` in `sqlite_repository.ts` after each insert. `history_summary` records (audit trail) are excluded from pruning but capped at 5 most recent, with excess records added to the `pruned` list for DB deletion. `app.ts` now explicitly skips `history_summary` records in both the narrator event summaries loop and the level-gain computation loop.
